# Plan Registry

計画運用を直す時の入口。ここは規約と責務地図を持つが、個別の計画本文・状態・履歴・session-boardの記録は所有しない。計画を全repo共通のフォルダへ集める場所でもない。

## 1. 最初に読む順番

1. 全runtimeの最小ゲートは親の `../GLOBAL_AGENTS.md` §6–7 を読む。
2. この文書で規模・段階・レビューの**基準**と、責務、手動/機械の境界を確認する。
3. `../skills/plan-triage/` がその基準を依頼へ適用し、対象repo・既存plan・宣言済みの計画箱を**書き込まず**routeとして解決する。
4. 解決先の最寄り `AGENTS.md` を読み、物理バケット・テンプレ・area固有の置き方に従う。
5. 雛形生成、子計画マップ更新、静的lintだけを `../skills/plan-ops/` のscriptで行う。

置き場が未宣言、複数候補、正本不明なら新しい共通 `plans/` を作らず、人間に確認する。

## 2. 規模・段階・レビュー

### サクッと／ライト／フル

- **サクッと**: `GLOBAL_AGENTS.md` §7の3条件がすべてYESの時だけ、計画書なしで実装し自己確認後に事後報告する。1つでもNOまたは不明ならライト以上にする。
- **ライト**: 計画 → 実装 → 実装レビュー → 事後報告。規模は計画冒頭に宣言し、条件を外れたらフルへ引き上げる。
- **フル**: 企画 → 計画 → 計画レビュー → 実装 → 実装レビュー → （修正 ⇄ 再レビュー） → 人間確認 → 完了。単発planではこの順をそのまま使う。programでは下の「programの人間確認」を使う。

計画のフォルダ状態は `planning`（未着手）→ `active`（実装・修正・AIレビュー中）→ `done`（最終評価全PASS・人間クローズ待ち）→ `archive`（人間確認済みの閉じた参照専用）を正本とする。`archive` は終了区分と終了記録を必須とし、規約の機械適用は plan-ops に委譲する。

programの子計画マップは `企画 / 計画 / 計画レビュー / 実装 / 実装レビュー / 修正 / 人間確認 / 完了` を使う。実運用では `実装中`、`計画レビュー待ち`、`保留` の注記を許す。program化は、独立に卒業する子計画が2本以上必要になった時に選び、基本は6〜7子まで、OSレベルの大改修だけ10前後を目安にする。

### programの人間確認

- programは、親に `人間確認方針: 最終一括（危険操作は実行前に個別承認）` を書く。子は独立レビューと必要な修正が全PASSで、個別の人間ゲートが無ければ `完了（AIレビュー済み・全体人間確認待ち）` として閉じる。子ごとに人間確認を待たない。
- 親programは、全子の完了条件と統合評価が全PASSになった時だけ `人間確認` へ進む。人間はその時にprogram全体を一度確認し、承認後に親を完了にする。
- 削除・移動・改名・履歴改変・hook/launchd登録・push・main反映・外部公開・本番データ変更・DB migration適用は、子の `人間ゲート:` に操作を特定して書き、実行前に個別承認を得る。親の最終確認で遡って承認済みにしない。

### レビューと人間ゲート

- サクッとは実装者の自己確認、ライトは1パス・差し戻し上限1、フルは探索を含むレビュー・差し戻し上限2とする。一括は直接依存のない子を3子程度で束ねる既定とし、共通契約を変える修正が必要なら即差し戻し・都度化する。
- レビュー項目は実行前に定義し、対象（ファイルまたはセクション）と「正しい状態」を書く。評価・修正文書のファイル名と置き場は `../../my-brain/areas/AGENTS.md` を正とする。
- 人間の明示承認が必要な操作の正確な一覧は、全runtimeが読む `../GLOBAL_AGENTS.md` §7 を正とする。計画側は、その承認を先に得たことを前提に進める。
- 人間確認はレビューのたびに要求しない。programの通常子は独立レビューの全PASSで閉じ、親の統合評価後に一度だけ最終確認へ上げる。個別承認が必要な操作だけは、この例外にしない。
- 人間の実機目視確認は原則として親の最終一括確認に束ねる。子の段階で目視が必要な場合は実行契約の検証にその項目を明記し、目視結果（日時・確認点・結果）を評価mdへ記録した時だけPASSにできる。記録のない目視は保留として扱う。
- フル計画の計画レビューは、`explain/` の図解HTMLを最新化して人間へ提示し、双方の理解に相違がない明示を得てからactive昇格・実行開始する。並列子はレーン別のファイル担当とworktree方針を実行契約へ記載済みでなければ起動しない。
- 指揮官は原則3レーンまで、同時レビューは全体で2本までにする。超える編成は人間へ明示する。

## 3. 責務地図

| 所有者 | 持つもの | 持たないもの |
|---|---|---|
| `plan-registry/` | この規約、規模・段階・レビューの基準、責務地図 | 個別plan本文、状態、履歴、実行ログ |
| `GLOBAL_AGENTS.md` | 全runtime向けの最小ゲート、危険操作の承認条件 | 詳細な運用手順、テンプレ全文 |
| 対象repoの最寄り `AGENTS.md` | そのrepoの計画箱とローカル規約 | 他repoの置き場判断 |
| `my-brain/areas/AGENTS.md` | areaの物理バケット、テンプレの配置、評価文書、卒業手順 | 規模別レビューやSkillの責務判断 |
| `ai運用/AGENTS.md` | Personal OS基盤・横断計画のarea固有配置 | Global共通規約の複製 |
| `plan-triage` | registryのサクッと基準を適用した書込みなしのroute解決（対象repo・既存plan・宣言済み箱） | 規模基準の定義、ファイル書込み、計画内容、レビュー合否の判断 |
| `plan-ops` | 雛形、子計画マップ更新、静的lintなど決定的な手続き | 置き場、優先度、レビュー合否の判断 |
| `plan-create-review` | 利用者の目的を作成・合流・program管理・評価・終了（archive）のworkflowへ振り分け、triage/opsへ委譲する一本入口 | 規模基準・route・script・計画本文・レビュー合否・runtime露出の所有 |
| hook / session-board | 実行記録、通知、sessionの開始・終了 | 計画本文・計画状態、repo/計画箱/レビュー合否の決定 |

## 4. 手動・script・hookの境界

```text
依頼
  └─ plan-registry: 規模・段階・レビューの基準を定義
       └─ plan-triage: 基準を依頼へ適用し、書込みなしのrouteを解決
            └─ 人間/指揮官: 人間ゲートと、既存planへ合流するか新規起案するかの最終判断
                 └─ plan-ops: 雛形生成・地図更新・lintを機械実行
                      └─ 実装 / レビュー
                           └─ hook / session-board: 実行記録と通知だけ
```

- scriptは確定した入力を機械処理する。規模、置き場、レビュー合否を推測して決めない。
- hookは短い導線を示してもよいが、計画を作成・移動したり、Skillの実行やレビュー合否を強制したりしない。
- 人間ゲートを含む操作は、scriptやhookの成功だけで実行済みにしない。明示承認を先に記録する。

## 5. 物理的な置き場

- repo固有の仕事は、そのrepoの最寄り `AGENTS.md` が宣言する計画箱を使う。
- Personal OS、横断repo、Global Skill、repo、loopの計画は `../../my-brain/areas/ai運用/` を起点にし、`../../my-brain/areas/AGENTS.md` のバケット規約に従う。
- `plan-registry/` へ計画本文、評価結果、移行ログ、session-boardの行をコピーしない。必要な時は正本への相対パスを参照する。
