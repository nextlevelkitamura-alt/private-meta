# 計画システム再設計 — テーマ層とゴール実行

分類: 横断
種別: 既存改善
規模: フル
優先: ◎

テーマ: [../../../themes/計画システム再設計/theme.md](../../../themes/計画システム再設計/theme.md)
関連: [調査結果.md](./調査結果.md)（4リポ調査の根拠）・[explain/plan.html](./explain/plan.html)（図解）

---

## 目的

「テーマ（目的と完了条件）を熟成させる場所」と「1本の直列ラインで完遂まで走るゴール実行」を計画システムの中心に据え、
早期分割による停滞と死に計画の堆積をなくす。

## 非対象

- board DB（Turso）のスキーマ変更、focusmap の今日ボード実装の変更
- orca-cockpit / cockpit-supervisor の作り直し（既存の別課題のまま）
- 各リポの死に計画の**実際の移動・削除**（棚卸しリスト提示まで。移動は人間承認ゲート）
- `plans/` バケットの名前・位置の変更（hooks 互換のため恒久的に不変とする）

## 現状

[調査結果.md](./調査結果.md) 参照。要点3つ:

1. **早期分割が停滞を量産**。テンプレと Program 規約が「作成時に完了条件チェックリスト＋子計画へ分割」を強制し、分割単位ごとに評価・状態管理が発生。仕事リポは9計画中7停滞（Phase 0 据え置き3件）、personal-os は planning 9件滞留、career は未着手9件全放置。
2. **テーマの熟成層が不在**。`thinking/` 廃止後、未成熟な考えの置き場は「planning の plan.md 方針欄」しかなく、アイデアが計画の形を強制される。テーマは board DB の1行ラベルで本文を持てない。壁打ち（grill-me 等）の出力を溜める場所も未規定。
3. **クローズが回らない**。archive 60 vs done 2。目的を見失った計画が planning/active に堆積し、死に計画が正常な計画の視界を塞ぐ。

## 方針

### 1. テーマ層の新設（壁打ち・熟成の常設場所）

各 area に `themes/` を新設する。**テーマは計画ではない**。目的を育てる場所であり、上限なし・評価なし・停滞という概念なし。

```
my-brain/areas/<area>/
  themes/
    <テーマslug>/
      theme.md        # 下記テンプレ
      壁打ち/
        2026-07-24.md # 壁打ちセッションごとの追記ログ（日付ファイル）
    _closed/<slug>/   # 閉じたテーマ（達成/取り下げの区分と一行理由を theme.md 末尾に残す）
  plans/              # 既存5バケットは名前・位置とも不変（hooks 互換）
```

theme.md テンプレ（6節固定・すべて追記型で育てる）:

```markdown
# <テーマ名>

親ゴール: 年間#<slug>（my-brain/ゴール の的。無ければ「未接続」と明記し、月間計画に的のドラフトを提案）
board-theme-id: <ID>（board DB との照合キー。名前は DB 側が正・改名に耐える）

## 目的（なぜやるか。1〜3行）
## 完成形スケッチ（使用体験。誰が・どう使い・何がどう変わるかのシナリオ。ぼんやりのままなら「未確定」と書いて壁打ちで育てる）
## ゴール（達成されたと言える状態。3〜7個、検証可能な条件で。1行で「完成」できる浅い的にしない）
## 現在の考え（未確定のまま書く。壁打ちのたびに更新）
## 派生計画（plans/ へ切り出したものへのリンクと現状）
```

- **親ゴールで縦ladderと内容を接続する**: 現状は slug の形式だけで、年間計画には slug 未適用・月間の的は0%ドラフト2件のみ・計画側からゴールへの参照ゼロ。テーマが「的の内容（なぜ・どこまで）」を運ぶ器になり、ゴール階層 → テーマ → 計画 → 実行が内容で繋がる。
- **完成形スケッチが目標の浅さ対策**: 的が1行で「簡単に完成」してしまい、使用体験や完成形がぼんやりのまま AI が実装に走る現状への直接の解。/goal はこの節が「未確定」のままでは実行に入らない（→方針3）。
- **プロジェクト単位のマクロ目標**: 各実行repo（仕事/focusmap/career）にも `themes/` を薄く置き、repo テーマの親ゴールは my-brain の的（年間#slug）を指す。俯瞰（my-brain）→ プロジェクトのマクロ目標（repo テーマ）→ 計画、の3段で内容が落ちる。

- 壁打ちの運用: grill-me / naiyou-suriawase / 通常チャットの壁打ち結果は、必ず `壁打ち/日付.md` に追記し、確定した分だけ theme.md の「現在の考え」へ昇格させる。**チャットに揮発させない**。

**Focusmap Theme とのハイブリッド共存（2026-07-24 決定）**:
- **名前・当日の実行状態 = board DB（Turso）が正本**。Web UI 上でテーマ名が変更されうるため、AI はセッション開始時に毎回 DB から取得して考える。
- **目的・完成形スケッチ・完了条件（ゴール）・壁打ちログ・計画資料 = ローカル md が正本**。コンテキストの本体はこちらで、**完了判定は theme.md のゴールで行う**。本文は DB へ送らない。
- 照合キーは名前でなく **board-theme-id**（theme.md frontmatter に持つ）。md 新規テーマは `board.py theme-add` で DB 登録し ID を書き戻す。DB 側で改名されたら theme.md の表題だけ追従してよい（内容は影響なし）。

**ライトテーマ（割り込み・逸れ物の受け皿）**:
- テーマ作業中に出る「今すぐやること」「ふと思いついた修正」は、手を止めずにまず**既存ライトテーマと照合**し、合えばその配下に todo として入れ子登録する。合致が無ければ**親目的1行だけのライトテーマを自動起案**する。
- ライトテーマは恒常的な軽い受け皿（例: LINE返信、ちょい修正、経理まわり）。定義は `themes/_light/<名前>.md` の1ファイル（親目的1〜3行＋照合キーワード）に留め、完成形スケッチや評価は持たない。
- **セッションも入れ子で束ねる**: 現状「Theme/Plan未紐付けセッション」としてセッション単位でしか見えないものを、ライトテーマ配下にネスト表示する（例: LINE返信セッション群を1つのライトテーマで束ねる）。未照合のセッションだけが未紐付け欄に残る。
- 同類の todo が3件溜まったライトテーマは、通常テーマへの昇格（theme.md 化）を提案する。
- planning バケットの意味を「実行を決めた計画の順番待ち」に絞り、**上限5を新設**する。アイデア・構想の受け皿は themes/ が引き受けるため、planning の無制限は廃止できる。

### 2. 計画テンプレの直列化（実行ライン方式）

計画は「テーマから切り出した実行1本」。**基本は直列、並列は区間として埋め込む**。子計画への分割は既定で行わない。

新テンプレ:

```markdown
# <計画名>

テーマ: ../../themes/<slug>/theme.md
分類: / 種別: / 規模:（フル/ライト。省略時ライト）

## 目的（テーマから1行引用）
## 完了条件（3〜7個。テーマのゴールのうち今回満たすもの）
## 実行ライン
- [ ] 01 <ステップ>
- [ ] 02 <ステップ>
- [ ] 03 ⇉ 並列区間 { 03a … / 03b … / 03c … } → 04で合流
- [ ] 04 <合流・統合>
- [ ] 05 [SAVE] まとめ評価（1回。FAIL時は修正1回→再評価。再FAILのみ人間へ）
- [ ] 06 <完了条件チェック → done 申請>
## 記録（実行中の判断・つまずきの追記ログ）
```

- 既存テンプレの `非対象/現状/実行契約/方針` は、必要な内容だけを実行ラインの各ステップと「記録」に吸収する。**計画の重さの既定値を下げる**。
- **Program（親子分割）は例外に格下げ**。並列 worktree での分離実装が本当に必要な時だけ使い、その場合も評価は子ごとに行わず [SAVE] に集約する（→方針4）。
- `## 工程` 節を読む daily-start / fetch-context.sh との互換: `## 実行ライン` を新設したうえで、移行期間は fetch-context.sh に節名の別名（工程|実行ライン）を1行追加して対応する。

### 3. ゴールコマンド（/goal）

Global Skill `goal` を新設する（`AIエージェント基盤/skills/goal/SKILL.md`）。「目的と完了条件を定めたら、打てば完遂まで走る」の実体。

フロー:

0. **入場条件（実装ファースト制御）** — theme.md に「完成形スケッチ」と検証可能なゴールがあることを確認。無ければ**実装に入らず**壁打ち（grill-me）へ差し戻す。AI（特に Codex）は実装ファーストに走る傾向があるため、「何を作るか合意してから作る」をコマンドの構造で強制する。
1. `/goal <テーマ|計画>` — theme.md の目的・ゴールを読む。完了条件が曖昧なら grill-me を**3問以内**で挟んで確定（それ以上は聞かずに仮置きして進む）。
2. 計画が無ければ実行ライン方式で planning に起票 → `bucketctl promote` で active 昇格（上限に当たったら人間へ1行で裁定を仰ぐ。planning のまま実装先行はしない）。
3. 実行ラインを**上から直列に自律実行**。並列区間だけ subagent へ fan-out し、合流ステップで統合。
4. [SAVE] 到達時にまとめ評価1回 → FAIL なら修正1回 → 再評価 → 再FAILのみ人間へ。
5. 完了条件を照合し、全達成なら done へ。人間ゲートは既存規約の危険操作（削除・push・本番反映・DB migration 等）だけに限定する。
6. 完了時に theme.md の「派生計画」を更新し、テーマのゴールが全達成なら**テーマのクローズを人間に提案**する。

### 4. 評価の一本化（セーブポイント評価）

現行規約（plan-registry/AGENTS.md §2）を次のとおり改訂する:

| 項目 | 現行 | 改訂後 |
|---|---|---|
| 評価のタイミング | 計画/子ごとに独立評価 | **[SAVE] マーカー地点のみ**（既定は完了直前の1箇所。長い計画でも最大2箇所） |
| フルの差し戻し上限 | 2回 | **1回**（修正1回→再評価。再FAILで人間へ即介入） |
| Program の評価 | 子ごと独立評価＋親の統合評価 | 子の独立評価を廃止し、**親の [SAVE] に集約** |
| ライト | 独立評価1回・差し戻し1 | 変更なし（すでに理想形） |
| サクッと | 自己確認のみ | 変更なし |

- 評価ファイルは `評価/まとめ評価RR.md` に一本化（都度評価mdの新規作成を廃止）。
- 「毎計画に評価」ではなく「いいタイミングで終わった時のセーブポイントとして評価」を規約の文言として明記する。

### 5. 死に計画の回転

- **planning 上限5**（新設。themes/ が受け皿になるため成立）。超過時は bucketctl が拒否し、themes/ への差し戻しを案内する。
- **週次 sweep**（loop-creator 経由で登録する repo-local loop）: active/planning で14日更新なしの計画を検出し、「paused へ / テーマへ差し戻し / archive」の裁定リストを人間に提示する。移動自体は人間承認後に bucketctl で行う。
- **テーマ連動クローズ**: テーマを閉じる時、派生計画に終了区分（completed/superseded/cancelled…）を付けて一括で閉じる。「目的が消えたのに計画だけ生きている」状態を構造的に作れなくする。
- done→archive の人間ゲートは現状維持。

### 6. hooks・既存機構への影響と互換方針

**壊さないもの（不変を保証）**: `plans/{planning,active,paused,done,archive}` の名前と位置。これにより guard-plan-bucket-move / guard-plan-gate / plan-ops(bucketctl 等) / board DB(plan_slug) / daily-start は全て無傷。themes/ は**追加のみ**なので既存 hook に影響しない。

**変更するもの（最小限）**:
- bucketctl.sh: planning 上限5の追加（上限判定は active 用が既存なので同型で1箇所）。
- fetch-context.sh: 節名別名（`## 工程` | `## 実行ライン`）対応1行。
- guard-plan-bucket-move.py: themes/ 配下は対象外であることの確認（`plans/<bucket>` 判定なので原理上影響なしだが、テストケースを1つ足す）。

**ライトテーマ・テーマ照合の hook 実装方式（2026-07-24 設計）**:

役割分担は「hook＝決定的な注入と照合、判断＝セッション中のAI、書き込み＝board.py（spool再送つき）」。新規hookイベントはゼロで、既存3本の拡張に収める。

- **SessionStart（reconcile-and-notify.py 拡張）**: ライトテーマ一覧（board-theme-id＋名前＋照合キーワード）をコンテキスト注入。名前・IDは Turso（失敗時spool/キャッシュ）、キーワード・親目的はローカル `themes/_light/<名前>.md` から読む（ハイブリッド方針どおり）。
- **UserPromptSubmit（register-and-guide.py 拡張）**: 割り込みらしいプロンプト（または明示 `/light`）に対し、該当しそうなライトテーマ候補を1行注入するだけ。逸脱判断はAIが行い、`board.py todo-add --theme <light-id> --route light` で入れ子登録（警告・案内のみ、denyしない思想は guard-plan-gate と同じ）。
- **Stop / SubagentStop（mark-wait.py 系拡張）**: theme_id 未設定のセッションを、セッション要約×キーワードで機械照合し、合致すれば `sessions.theme_id` を「自動照合」マーカー付きで書いて入れ子化（UIから外せる形）。d3d45e1 の session routing 配線の上に乗る。合致しなければ未紐付けのまま。
- **自動起案・昇格は hook に入れない**: 即時性が不要なので daily-start（朝会）か週次loopで、未紐付けの溜まりから親目的1行のライトテーマ起案候補と、同類3件の昇格候補を提示（作成は人間OK後に theme-add＋`themes/_light/` 生成）。
- **Codex登録**: events/ 実体は runtime 非依存 Python のため、hooks-registry の Codex 登録表へ同イベントを追記するだけ（guard-plan-gate の Codex 未登録解消と同時に実施）。
- **データ側の変更**: themes テーブルに種別（normal / light）を1列追加。Focusmap は `sessions.theme_id` / `todos.theme_id` で入れ子表示し、「未紐付け」欄は照合に漏れたものだけが残る。

**実装ファースト制御の強化**:
- guard-plan-gate（計画に紐づかない実装への警告 hook）は現状 **Claude のみ登録・Codex 未登録**。Codex にも登録し、警告文面を「/goal 経由への誘導」に改訂する（警告のみ・deny はしない方針は維持）。
- guard-plan-bucket-move が正しく拒否しても「planning のまま実装先行」が起きた実例（ドッグフーディング課題#1）への対処は、hook の強化ではなく /goal の入場条件（方針3-0）で行う。

**ついでに直す既存の壊れ（調査で発見）**:
- focusmap: `.claude/settings.json` の Stop hook が存在しない `auto-commit.sh` を参照 → hook 削除か実体作成のどちらかに倒す。
- personal-os: guard-plan-bucket-move の登録状態の記述矛盾（「未適用」vs「登録済み」）を実機確認して片方に統一。
- focusmap: `docs/plans/README.md` のデッドリンク（completed/・gemini/）。

### 7. 各リポへの展開（横断・順次）

personal-os で方式を確立してから、同じ「themes/ ＋ 直列実行ライン ＋ [SAVE]評価 ＋ 週次sweep」を薄く移植する:

- **仕事（nextlevel-work）**: `plans/` 直下に `themes/` を追加。`領域/*/計画/` 12件を棚卸し（停滞7件は paused/停止/archive の裁定リスト提示）。`計画一覧.md` はテーマ一覧を兼ねる形に改訂し、台帳の手動同期を「テーマ単位」に減らす。
- **focusmap**: `docs/plans/`（旧系）を凍結アーカイブし `docs/ai/plans/` に一本化。NOW.md / MAP.md はアーカイブへ。task-board の6月残置3件と done_pending_archive 2件を裁定。
- **career-website**: `docs/企画/` をテーマ層として再定義（存在しない採用/不採用フォルダは規約から削除）。未着手9件・進行中の滞留2件を棚卸し。ROADMAP.md は「テーマ一覧＋今月の計画」に縮小して更新を再開するか、廃止して計画一覧に統合するかを人間が裁定。
- 各リポの AGENTS.md 該当節を同一作業内で改訂する（二重管理はしない。語彙の正本は personal-os 側）。
