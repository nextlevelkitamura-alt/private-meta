# 説明の型

メタ構造を人間が最短で理解できるHTMLの構成と見せ方。meta-explainの中核。
汎用のHTML規約（部品の選択ラダー・表の使用条件・節の型）は `html` スキルの `references/html-structure.md` が正本で、ここにはメタ説明に固有の型だけを書く。

## 1. 説明の原則

1. **全体 → 差分 → 本文の順**。いきなり中身を見せず、まず「どこの話か」を地図（構成図・関係図）で示す。
2. **現状を先に**。変更案は、現状が分かってから出す。現状抜きの変更案は理解できない。
3. **実ファイル根拠**。「今こう動いている」はパスと本文を根拠にし、読んで確認できなかった箇所は「推測」ラベルを付ける。
4. **本文は全文**。新規・変更ファイルのドラフトは省略しない。省略した瞬間に理解ゲートが形骸化する。
5. **人間の言葉**。作業中に生まれたコードネーム・略語・独自番号を説明なしで使わない。初出で定義する。
6. **1節1メッセージ**。見出しとリード1行だけ拾い読みして論旨が追える状態を保つ。
7. **分からせる責任はAI側**。理解されなかったら、同じ説明を繰り返さず見せ方を変える（図 → ウォークスルー → 具体例の順に切り替える）。

## 2. 固定7節型

変更前合意モードは7節フル。説明のみモードは③④⑤を省き、②を厚くする。

1. **① 結論**: 何の話か・何をどう変えるかを1〜2文で先出し。
2. **② 現状**: 構成図・関係図・「今こう動いている」。
3. **③ 変更計画**: before / after の対比と変更理由。
4. **④ ミクロ計画**: 新規・変更ファイルごとの本文ドラフト全文。
5. **⑤ 影響とリスク**: 触るもの・触らないもの・壊れうるもの。
6. **⑥ 未決事項**: 人間に決めてほしいことを質問形式で列挙（各質問に推奨案を必ず付ける）。ゼロなら「未決なし・OK待ち」と書く。
7. **⑦ 更新履歴**: v1, v2, … で「指摘 → どう直したか」を積む。

## 3. 見せ方マッピング（メタ内容 → 部品）

1. **フォルダ構成・正本と露出の関係** → tree図。新規=緑・変更=青の注釈を付け、「正本」と「symlink露出先」を必ず区別して書く。
2. **hook・loopの発火と流れ** → SVGフロー図。「いつ動くか（発火点）→ 何をする → 何が出る」の順。人間ゲート・承認点は色と形を変えて目立たせる。
3. **設定・登録・本文の差分** → before / after の2列対比。変わる行だけ強調し、変わらない部分は薄くする。
4. **ファイル本文** → codeblock全文。見出し行を強調色にし、行単位で指摘できる粒度にする。
5. **skill・registry・runtimeなど複数要素の関係** → 1枚のSVG関係図。矢印には必ずラベルを付ける（「参照」「symlink」「生成」「発火」）。ラベル無しの矢印を描かない。
6. **手順・運用フロー** → steps。実際のコマンド・実文をcodeblockで添えるウォークスルー型にする。
7. **選択肢の提示** → 案の列挙＋推奨pill。推奨を必ず1つ決めて先頭に置き、「どれでもいい」で人間に丸投げしない。

## 4. 理解ループの回し方

1. 毎版、⑥未決事項を質問形式で維持する。回答済みの質問は消さず「回答: …」を追記して閉じる。
2. 指摘を受けたら: ⑦更新履歴に「vN: 指摘 → どう直したか」を1行追記 → 該当節を直す → 同一URLで再公開する。新しいURLを増やさない。
3. 同じ箇所に2回指摘が来たら、直すだけでなく見せ方を変える（§1-7）。
4. 合意条件: 未決事項が全て回答済み ＋ 「理解した・これでいい」の明示。「はい」だけの返事は合意とみなさない。
