# 既存計画の整合監査と上限是正

## 目的

新しい遷移規約・終了区分と `active≤3 / paused≤3 / done≤8` を既存計画へ安全に適用し、実態とバケットが合わないもの・上限を超えるもの・終了記録が無いまま閉じられたものを可視化する。履歴を消さず、移動が必要な計画だけを人間が選べる状態にする。

着手は01（遷移・終了区分の機構）と11（ai運用pilotのArea標準適用）の後。同じarea `plans/` を触るため11と並行しない。

## 現状

ai運用には active、paused、done、archive の各バケットに複数の計画があり、過去の規約では done と archive の意味が現在提案と異なる。2026-07-13時点で paused=24（上限3）、done=18（上限8）であり、新規ルールでは即時に超過状態となる。直近でも archive 化された program に、子計画マップが `実装`・未チェックのまま残る例がある。これはその計画が未実装と即断する根拠ではないが、バケットだけでは人間が完了根拠を追えない状態である。

## 方針

1. active/paused/done/archive の全計画を読み取り専用で走査し、(a) 実装根拠、(b) 最終評価md、(c) 人間確認記録、(d) programの子計画マップの状態とチェックボックス、(e) バケット件数と上限超過、(f) archive配下の終了記録・終了区分の有無を一覧化する。出力はこの計画の `references/` に置き、正本の計画本文をコピーしない。
2. 一覧から「現規約なら done 候補」「archive だが確認根拠・終了記録不足」「子マップとバケットが矛盾」「paused/doneの上限超過」の候補を抽出する。機械検出（01のarchive lint・`bucketctl check` を含む）は候補提示に限り、状態の断定や移動を自動化しない。
3. 人間へ候補ごとに `維持 / planningへ戻す / activeへ戻す / doneへ戻す / archiveを承認（終了区分を記録） / pausedへ戻す` を提示する。承認が得られた対象だけ、01の遷移手順で移動し、programマップ・終了記録を同じコミットで更新する。削除・履歴改変はしない。先行資料 `2026-07-08-並列実装フロー` の `merged` close候補もここで提示する。
4. 少数の代表ケースで、planning→active→done→archive のE2Eを実演する。archive前は明示確認が無いと停止すること、session finishだけでは状態を動かさないことを確認する。
