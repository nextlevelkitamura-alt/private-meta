# SQL入門 — personal-osのTurso実例で学ぶ（2026-07-19）

SQLとDBの基礎を、このpersonal-osで実際に動いているTursoの実例だけで説明する学習用リファレンス。一般論の教科書ではなく「自分のデータがどう保存され、どう呼び出されるか」を分かるための文書。

## 1. Tursoとは何か

- **SQLite**: 1ファイルで完結する世界一使われているDB。スマホアプリの中身も大抵これ。
- **libSQL**: SQLiteのフォーク（互換拡張版）。
- **Turso**: libSQLをクラウドでホストするサービス。ローカルファイルの代わりに **HTTPSでSQLを送ると結果がJSONで返る**。だからMac・focusmap（Cloud Run）・エージェントのどこからでも同じDBを読み書きできる。

personal-osでは2つのDBを使っている: `personal-os-board`（セッション盤面）と `personal-os-inbox`（起票・やること箱）。

## 2. SQLが「どう呼び出されて、どう実行されるか」

session-boardの実際の流れ（board.py → turso/store.py）:

```text
board.py（Pythonスクリプト）
  │ ① デイリーmdを先に確定（flockで排他）
  ▼
turso/store.py
  │ ② macOS Keychainからトークン取得
  │    security find-generic-password -s turso-personal-os-board -w
  │ ③ SQLをJSONに包んでHTTPS送信
  ▼
POST https://<db名>.turso.io/v2/pipeline   （Bearer <トークン>）
  {"requests": [{"type": "execute",
                 "stmt": {"sql": "INSERT INTO session_logs (repo, entry, ...) VALUES (?, ?, ...)",
                          "args": [...]}}]}
  ▼
Turso側のSQLiteエンジンが実行 → 結果（行データ or 成功/失敗）がJSONで返る
  │ ④ 失敗したら state/turso-spool.jsonl に書き溜めて後で再送（spool）
```

ポイント:

- **SQLはただの文字列**。それをHTTPで送るだけなので、curlでも同じことができる。
- `?` は**プレースホルダ**。値を後から安全に埋める仕組みで、SQLインジェクション（値に細工したSQLを混ぜ込まれる事故）を防ぐ。
- 1クエリの往復は100〜300ms程度。

curlで手打ちする場合の形（トークンは表示・記録しない）:

```bash
TOKEN=$(security find-generic-password -s turso-personal-os-inbox -w)
curl -s -X POST "https://<inbox-db>.turso.io/v2/pipeline" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"requests":[{"type":"execute","stmt":{"sql":"SELECT title, status FROM todos WHERE do_date = date('"'"'now'"'"','"'"'localtime'"'"')"}}]}'
```

## 3. テーブルとは / 今あるテーブル

テーブル＝Excelのシートのようなもの。**列（カラム）の型と意味を先に決めて**、行（レコード）を貯める。

| DB | テーブル | 1行が表すもの |
|---|---|---|
| board | `sessions` | 今動いているAIセッション1つ |
| board | `session_logs` | 終わった成果1件 |
| board | `session_events` | 状態が変わった瞬間1回（run→wait等） |
| board | `daily_summary`（VIEW） | 1日ぶんの集計（後述） |
| inbox | `goals` | 起票された目標1件 |
| inbox | `todos` | やること1件（デイリータスク） |
| inbox | `repos` | 実行repoの選択肢1つ |

## 4. SQL文の基本4種を実例で

### CREATE TABLE — 箱を定義する

`todos` の実DDL正本はfocusmap repo `db/turso/migrations/20260717000000_todos_and_repos.sql`。骨子はこういう形:

```sql
CREATE TABLE todos (
  id          INTEGER PRIMARY KEY,   -- 主キー: 行を一意に指す番号
  title       TEXT NOT NULL,         -- NOT NULL: 空を許さない
  do_date     TEXT,                  -- やる日 'YYYY-MM-DD'（SQLiteの日付は文字列）
  due_date    TEXT,                  -- 〆切
  repo        TEXT,                  -- どのrepoの仕事か
  assignee    TEXT,                  -- '自分' | 'AI'
  status      TEXT DEFAULT 'open',   -- open | done | dropped
  ai_status   TEXT,                  -- 未検知〜完了（AIに任せた場合の進捗）
  created_at  TEXT,
  completed_at TEXT
);
```

### SELECT — 読む（今回の本丸）

「今日やること」を取るクエリはこれだけ:

```sql
SELECT title, repo, assignee, status
FROM todos
WHERE do_date = date('now', 'localtime')   -- 今日の日付と一致
  AND status = 'open'                      -- まだ終わってない
ORDER BY due_date IS NULL, due_date;       -- 〆切があるものを先に
```

- `WHERE` = 絞り込み、`ORDER BY` = 並び順。
- `date('now','localtime')` はSQLite組み込みの「今日」。タイムゾーンを付けないとUTCになる（queries/*.sqlにTZ注記があるのはこのため）。

### INSERT — 追加する

session_logsへの成果追記（store.pyがやっていること）:

```sql
INSERT INTO session_logs (repo, parent, entry, session_date, session_key)
VALUES (?, ?, ?, ?, ?);
```

sessionsは「同じセッションなら上書き」したいので **upsert**（あれば更新・なければ挿入）:

```sql
INSERT INTO sessions (session_key, goal, state, updated_at)
VALUES (?, ?, ?, ?)
ON CONFLICT(session_key) DO UPDATE SET goal=excluded.goal, state=excluded.state, updated_at=excluded.updated_at;
```

### UPDATE — 書き換える

タスク完了の書き戻し（Phase 2でエージェントがやること）:

```sql
UPDATE todos
SET status = 'done', completed_at = datetime('now', 'localtime')
WHERE id = ?;
```

`WHERE` を忘れると**全行が書き換わる**。UPDATEとDELETEは必ずWHEREとセット、が鉄則。

## 5. もう一歩: VIEW・migration・窓関数

- **VIEW**: 保存されたSELECT文に名前を付けたもの。`daily_summary` は実体データを持たず、`SELECT ... FROM daily_summary` と読んだ瞬間に session_events から日次集計を計算して返す。「集計ロジックをDB側に1ヶ所で持つ」ための道具。
- **migration**: スキーマ変更のSQLを日付付きファイルで積み上げたもの（例: `20260718_logs_session_key_and_daily_summary.sql`）。これが揃っていれば空のDBから同じ構造を再現できる。**現状boardの基盤3テーブルはCREATE文がmigrationsに無く手設置**＝再現できないのが弱点（turso現状調査参照）。
- **窓関数（window function）**: 行を跨いだ計算。`queries/session-durations.sql` は `LEAD(at) OVER (PARTITION BY session_key ORDER BY at)` で「次のイベント時刻」を各行に並べ、状態ごとの滞在時間（run何分・wait何分）を出している。「イベントの時系列から時間集計を作る」定番パターン。

## 6. 用語ミニ辞典

| 用語 | 意味 |
|---|---|
| PK（主キー） | 行を一意に指す列。`todos.id` など |
| DDL | CREATE TABLEなど構造を定義するSQLの総称 |
| upsert | あれば更新・なければ挿入（INSERT ... ON CONFLICT） |
| 論理削除 | 行を消さず `deleted_at` に日時を入れて「消えた扱い」にする（Supabase tasksが採用） |
| spool | 送信失敗をファイルに溜めて後で再送する仕組み（turso/spool.py） |
| RLS | Row Level Security。Supabase側で「自分の行しか見えない」を強制する仕組み。service_roleキーはこれをバイパスする |
