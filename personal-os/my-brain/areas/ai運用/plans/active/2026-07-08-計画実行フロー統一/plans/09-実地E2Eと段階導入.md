# 実地E2Eと段階導入
## 目的

子06〜08の新しい計画読込・評価・修正ゲートを、Claude/Codexとメイン/サブエージェントの実機で検証し、
既存作業を止めずにshadowから強制モードへ段階導入できる状態にする。

## 現状

- Hookの合成テストが通っていても、runtime trust・実イベントpayload・Orcaフック・利用上限でE2Eが止まる実績がある。
- ClaudeとCodexはStop注入、Hook型、trust、subagent起動方法が異なり、片側PASSだけでは共通運用を保証できない。
- 新構造は既存の平置き子計画と共存する必要があり、全計画の一括移動はリスクが高い。

## 方針

### 1. テストマトリクス

最低限、次を別ケースとして固定する。

- runtime: Claude / Codex。
- 実装者: メイン / subagent。
- 規模: サクッと / ライト / フル。
- 結果: 初回PASS / 初回FAIL→修正→PASS / PLAN_DEFECT / reviewer起動不能。
- 計画: 正常 / 未読 / SHA変更 / 必須節欠落。
- 差分: 許可内 / 3ファイル目 / 範囲外 / 未コミット。
- Hook: 正常 / 二重発火 / trust切れ / 内部例外 / stop再入。

### 2. テスト層

1. **unit**: plan解決、SHA、サクッと2ファイル上限、状態遷移、採点lint、fix-plan生成。
2. **hook fixture**: Claude/Codexの実payload fixtureでstdout JSON・exit code・再入防止を検証。
3. **integration**: 一時git repoでplan→編集deny/allow→評価→修正を通す。
4. **E2E**: 実Claude／Codex／Orcaペインで各1本、SubagentStopと通常Stopを実測。

### 3. 段階導入

- **Phase 0: dry-run** — 記録だけ。現在ならdenyする理由と想定遷移をログし、編集は止めない。
- **Phase 1: shadow** — サクッと/ライト判定、plan解決、レビュー要求生成を観察。偽陽性を収集。
- **Phase 2: plan-read強制** — ライト以上の編集前読込だけfail-closed。レビューは警告。
- **Phase 3: review強制** — 未レビューStopをblockし、評価／修正ラウンドを必須化。
- 各Phaseの昇格は人間承認。自動で次段階へ上げない。

### 4. 登録と人間ゲート

- Claude settingsへの新Hook登録、Codex hooks.json更新と`/hooks`再trustは人間ゲート。
- 登録前後の設定差分、対象イベント、戻し方を1セットで提示する。
- session-board既存Hookと同じイベントで併存するため、実行順と片方の失敗が他方へ影響しないことを実測する。

### 5. rollback

- 強制モードを1フラグでshadowへ戻せる。
- Hook登録を外しても、既存session-boardの開始／Stop／サブ体数が維持される。
- runtime cacheと未完了review状態の掃除手順を持つ。計画・評価・修正MDは履歴として削除しない。

### 6. 移行

- 新規ライト／フルから新フォルダ形式を適用する。
- 既存active計画は着手時に必要なものだけ移行候補を提示し、人間承認後に移す。
- done/archiveの履歴は移動・書換しない。lintはlegacy読取互換を維持する。
