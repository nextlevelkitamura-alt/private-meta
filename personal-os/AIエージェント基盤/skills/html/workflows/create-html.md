# HTML作成（1本道）

回答・作業結果・実装説明を、見て分かる1枚HTMLにする。モード選択は無い。

## 1. 対象を確認する

1. `/html` で指定された直前の内容、またはHTML化を依頼された内容を入力にする。
2. 正本md、デイリー、計画、Notion同期対象、継続編集する文書はHTMLで置き換えない。
3. バックエンドやデータ保存が必要なら、表示専用Artifactの対象外と伝える。
4. 実装・変更の説明かどうかを判定する。該当したら `references/impl-explain.md` を併読する（3点セット必須・合意まで対象を編集しない）。

## 2. サンドイッチ構造で組む

1. ページの並びは常に `references/html-structure.md` §2に従う: ①結論 → ②先に質問 → ③詳しい説明（図解が主役）→ ④もう一度質問（推奨つき）。
2. 人間に判断してほしいことが無ければ②④を省き、①③だけにする（レポート・調査まとめはこの形になる）。
3. 実装・変更の説明では、③に3点セット（どこを触る → こう変わる → 中身の意味）を必ず入れる。
4. 内容が軽ければ部品と図を減らして短くする。「軽く作るモード」は無い（構造と読みやすさのルールは常に同じ）。

## 3. HTMLを作る

1. `references/html-structure.md` の読者想定・Artifact制約・部品選択ラダー・表の条件・節の型に従う。
2. `assets/artifact-template.html` を骨組みにし、使う部品だけ残す。
3. 外部リソース・JS・バックエンドを使わず、self-containedな単一ページにする。
4. secret・token・credential・認証値を含めない。
5. ローカルへ保存する場合は `GLOBAL_AGENTS.md` §3–4を正本として、areaでは計画ごとの `explain/` と恒久知識の隣接HTMLを使い分ける。
6. ページ全体を白背景のライト単色にし、カード・図・コードブロックに暗い背景を使わない。表示環境による配色切替も作らない。

## 4. 公開前チェック

1. ページ全体が白背景で、図・カード・コードブロックに暗い背景・配色切替が無いか。
2. 見出しとリードだけ拾い読みして、論旨が追えるか。
3. 構造・流れ・関係を扱う節が文章だけになっていないか（図解判定の取りこぼし。主要節の過半で図が主役か・最低1枚あるか）。
4. コードブロックに「つまり」の1行訳、専門用語に初出の言い換えが付いているか。
5. 判断してほしいことが質問形式＋推奨案で②④に出ているか。無ければ「OK待ち」の明記か②④の省略になっているか。
6. 表が5条件（`references/html-structure.md` §6）を満たし、1ページに3箇所以上無いか。
7. PCで表示とレイアウト崩れを確認したか。

## 5. 提示と更新

1. チャット表題を付けたArtifact URL、またはローカルHTMLのリンクを提示する。
2. 指摘を受けたら同じHTML・同じURLを更新し、更新履歴にvN（指摘 → どう直したか）を積む。新しいURLを増やさない。
3. スマホ・ブラウザ・URL表示の明示依頼時だけ `workflows/mobile-preview.md` を実行する。HTTPS接続・HTTP 200・HTML title一致・tmux稼働を確認してからURLを提示する。
