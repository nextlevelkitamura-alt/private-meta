## 1. private-meta / personal-os（計画基盤の本丸）

### 構造の現状
- 正本: `my-brain/areas/AGENTS.md`（語彙・バケット・テンプレ）+ `AIエージェント基盤/plan-registry/AGENTS.md`（規模・評価・経路）。
- バケット: `plans/{planning,active,paused,done,archive}`。状態はフォルダが正本。上限 active=6 / paused=3 / done=8、**planningとarchiveは無制限**。
- 入口: kickoff（起票）→ inbox-triage / plan-triage(registryへ畳込み済み) → plan-create-review → `plans/planning/<日付-名前>/plan.md`。
- テンプレは `目的/非対象/現状/実行契約/方針/完了条件` 必須。**完了条件は作成時点でチェックボックス化した受入条件（＝評価項目の早期分割）**。フル規模は Program（親 program.md + 子計画 NN + 評価/）へ早期分解。
- 評価規約: サクッと=自己確認のみ / ライト=独立評価1回・差し戻し1 / **フル=独立評価・差し戻し2回（3回目FAILで人間）**。まとめ評価が既定だが、Programでは**子ごとに独立評価 + 親の統合評価**が走る。

### 死に計画の実態（ai運用 area）
- planning **9件**（無制限のため溢れの受け皿化）。優先◎の program が2〜3週間 promote されず滞留（07-02 repo構造標準化、07-09 デイリー運用刷新、07-11 Focusmap自動処理統合）。

- ドッグフーディング破綻の自認例: `planning/2026-07-17-worker役割別コンテキストと評価フォルダ分離/plan.md` 冒頭「状態: planningのまま実装先行（bucketctl が active上限で正しく拒否＝ドッグフーディング課題#1）」。

### テーマ・壁打ちの置き場

### hooks / データ送信の結合度

## 横断診断（問題の構造）

1. **早期分割が停滞を量産する**: 全リポで「作成時に Phase/子計画/評価項目へ分割 → 分割単位ごとに評価・状態管理が発生 → 重くて着手されない/途中で止まる」が再現している（仕事リポの Phase 0 据え置き3件、personal-os の planning 9件、career の未着手9件）。
2. **テーマ（目的）の熟成層が存在しない**: どのリポも「アイデア→即・計画の形」を強制する。目的が揺れている段階の考えを育てる場所がなく、計画だけが増えて目的が失われる。テーマは board DB の1行ラベルで本文を持てない。
3. **クローズの儀式が回らない**: done→archive、完了→移動、台帳同期がすべて手運用で、archive 60 vs done 2（personal-os）、5.5ヶ月滞留（career）、6月残置（focusmap）など死に計画が堆積する。
4. **評価の設計が重い**: フルは差し戻し2回、Program は子ごと独立評価+親統合評価。ユーザー体感の「評価やりすぎ」はここに由来。
5. **hooks はパス結合だが、バケット名を変えなければ安全**: `plans/{planning,active,paused,done,archive}` の名前と位置を維持する限り、既存 hook / plan-ops / board DB / daily-start は無傷。テーマ層の**追加**は既存機構に影響しない。
6. **修正すべき既存の壊れ**: focusmap の死んだ Stop hook、docs/plans/README のデッドリンク、guard-plan-bucket-move の登録記述矛盾、career の frontmatter 4系統混在。
