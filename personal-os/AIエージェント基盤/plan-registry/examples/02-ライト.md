# 実演例2: ライト（合成例）

## 入力

- やりたいこと: 「見張り番（watch.sh）のWAKE理由をSlack通知にも流したい」
- 対象repo見当: AIエージェント基盤（loops-registry / orca-cockpit まわり）

## 4判断

1. **規模**: 問い1=NO（`watch.sh` の通知呼び出し追加＋通知先設定ファイル追加の2ファイル超になる見込み・戻しは容易だが1〜2ファイルに収まらない） → サクッとではない。1レーンで完結し、完了条件が「WAKE時にSlackへ1通届く／既存の起床通知が壊れない／設定なしでも従来どおり動く」の3行で書ける → **ライト**。
2. **経路**: Private起点なのでrepo-registryで担当repoだけを決める。Global loopの変更は、目的に対応するAI運用ThemeがあればそのTheme最寄り `AGENTS.md` が宣言する `plans/planning/` を新規候補にする。Themeに属さない実装なら所有repoの計画箱を使い、AI運用直下の共通計画箱は作らない。宣言範囲の既存planが一意なら新規作成せず合流する。
3. **起動形**: 計画は子計画文書として最小plan.mdがあれば足りる通常ケース → **2ペイン既定**（実装/レビュー）。
4. **モデル**: 実装=sonnet5／レビュー=codex xhigh（既定）。このレーンが遅れても他レーンは並列で進められる（誰も待たない）→ 格上げなし。
