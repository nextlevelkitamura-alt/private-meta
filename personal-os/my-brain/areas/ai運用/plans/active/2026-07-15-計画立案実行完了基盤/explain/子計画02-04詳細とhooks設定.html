子02 遷移統制とplanctl同期 — 計画が「勝手に動かない・古くならない」仕組み

作るものは2つ。バケット移動の門番（bucketctl拡張）と、実装結果→計画の機械同期（planctl）です。まず門番が許す道はこれだけ：

planctlの6コマンド（何をするか）を見る
prepare
委譲の準備。run manifest＋Task Packet＋task worktreeを明示引数から生成（git管理外）
progress
programマップの子ブロック（状態・次・参照）を更新。対象以外はバイト不変
apply-evaluation
評価mdが全PASS＋文言完全一致の時だけ、[x]化・実装結果追記・マップ同期を機械実行。1つでも曖昧なら拒否
close
終了区分・人間確認を記録してbucketctl経由でarchiveへ。門番の検証は迂回できない
sync-check
結果・評価・チェックボックス・マップ・バケット・終了記録の不整合をJSONで返す（hookの見張りが使う）
rename
計画を大幅更新した時のフォルダ日付の最新化＋repo内参照の追従書き換え。--checkは「更新日と日付のズレ」を機械判定（mtime推測はしない）

子03 harnessとゴールコマンド — 委譲・worktree・完走のエンジン

部品は3層。delegate（1つの仕事をCodex/Claudeへ渡す）、roles（explorer/implementer/reviewerの役割定義）、そしてprogram-run（ゴールコマンド）。program-runが回すループはこれです：

roles 3種（worker人格の最小定義）を見る
explorer
read-only。実行経路・所有・依存を調べ、pathとsymbolを根拠に返す。編集しない
implementer
workspace-write。1つのTask Packetだけを最小変更で実装し、結果JSON必須。隣の仕事へ広げない
reviewer
read-only。完了条件とdiffを照合し、PASS/FAIL/対象外＋根拠。自己申告を根拠にしない

定義に入れないもの: 固定worktree・branch・Program固有の背景・モデルID・長い性格設定。/codex-implは中身をdelegate経由に差し替え（使い方は今のまま）。

子04 Prompt Submitとhookガード — 4つの見張りが「何を・いつ」見るか

hookは検知・案内だけで、計画もworktreeも編集しません。止める（継続を要求する）のは確定違反を検出した時に1回だけ。迷ったら必ず素通し（fail-open）なので、普段の作業は邪魔されません。

PreToolUseバケット移動の門番常時・Bashだけ監視

Bashコマンド内の計画バケットを対象にした生の git mv/mv を拒否し、bucketctl（02の門番）へ誘導。対象パスを含まない通常コマンドは無処理で通過。

SubagentStartworker起動時の割当照合

実装workerが割り当てられたworktree・base・branchで作業を始めるかをmanifestと照合し、ズレていたら編集前に警告。※両runtimeとも「開始を強制停止」はできない仕様のため照合・警告に限定（Codex調査で実機確認済み）。worktreeの作成・割当そのものはharness（03）の仕事。

SubagentStopworker終了時の結果検証

implementerが結果JSONを出さずに終わろうとしたら1回だけ差し戻し。reviewerの評価不足も検出。explorerは構造化結果があれば通す。未mergeのworktreeが残っていれば事実の案内だけ返す（削除はしない）。

Stopセッション終了時の同期・鮮度チェック

「レビュー全PASSなのに計画が未同期」の時だけ1回止めて planctl apply-evaluation を要求。一括レビュー待ちは止めずに件数を案内。計画を大幅更新したのに日付が古い場合は planctl rename --check の結果で案内（推測はしない）。あわせてPrompt Submitの計画案内文も新しい一本道＋レビュー方式に揃える。

実行される本体（Python）は1箇所だけに置き、CodexもClaudeも同じファイルを呼びます（runtimeごとに複製しない、が既存の規律）。緑＝今回の新設、青＝既存のまま。

共存の原則

既存のsession-board用hook（mark-wait等）と同じイベントに別handlerとして追加し、置換・改造はしません。お互いの出力に依存しないので、片方が失敗してももう片方は動きます。

hooksの登録方法 — 書く場所は2ファイルだけ

本体を置いただけでは動きません。runtimeごとの「登録表」に1エントリずつ追加して初めて呼ばれます。書く場所と反映のされ方が2つで違います：

  1. ①

    Claude Code → ~/.claude/settings.json の hooks 項目

    保存すれば即反映・trust不要。settings.jsonには他の設定も同居しているので、hooks項目だけを直接編集します（repoへのコピーやsymlink化はしない、が規律）。追加する1エントリの形：

    // "Stop" イベントへの追加例（既存のmark-waitとは別グループで共存） { "hooks": [ { "type": "command", "command": "~/.claude/agent-hooks/events/session-end/guard-plan-closeout.py --runtime claude", "timeout": 10 } ] }

    PreToolUse だけは "matcher": "^Bash$" を付けてBashコマンドの時しか起動しないようにします（他のツール操作を邪魔しないため）。

  2. ②

    Codex → hooks-registry/codex/hooks.json（repoが正本）

    ~/.codex/hooks.json はこのファイルへのsymlinkなので、repo側を編集すれば実体は1つ。ただしClaudeと違い、変更後にあなたが Codex の /hooks 画面で「再trust」する1操作が必要です（中身が変わるたびにhashが変わる仕組み）。追加エントリの形は①と同じで、パスが ~/.codex/agent-hooks/...・--runtime codex になるだけです。

有効になるまでの手順 — 実装と登録は分離、登録はあなたの承認後

Wave 4で本体とテストまでを作り置きし（ここまで人間なし）、登録の3操作だけがあなたの承認後です。★があなたの出番。

  1. 1

    本体＋テストを実装（Wave 4・承認不要）

    新設4ファイル＋共通エンジン＋fixtureテスト。fail-open・1回だけblock・既存hookとの共存を合成データで確認。

  2. 2

    登録差分を承認セットへ積む（承認不要）

    ①settings.jsonへ足すJSON、②hooks.jsonへ足すJSON、を差分の形で用意。この時点ではまだ何も動かない。

  3. ★3

    あなた: 承認セットで登録差分をOKする

    program完走後の最終一括確認に含まれます。

  4. 4

    ①Claudeへ登録 → 即反映

    settings.jsonのhooks項目へ追加。保存だけで有効。

  5. 5

    ②Codexへ登録 → repo正本を更新

    codex/hooks.jsonへ追加（symlink経由で実体は1つ）。

  6. ★6

    あなた: Codexの /hooks で再trust（1回）

    変更済みhookの一覧を見てtrust。これでCodex側も有効。

  7. 7

    E2Eで総点検

    既存5イベントが従来どおり動く／通常Bashは素通り／生git mvは拒否→bucketctl案内／PASSなのに未同期だけ1回止まる…を両runtimeで実測。registered.sh（診断スクリプト）でも登録状態を読み取り確認。

未確認が1つだけ残っています（決め打ちしません）

PLAN_RUN_MANIFEST（実行時情報の環境変数）がsubagentのhookまで届くかは公式仕様に保証がなく、harness実装後の実機E2Eで確認します。届かない場合、SubagentStartの照合は「manifestの存在確認と案内」まで縮退します（それでもStopガードと結果検証は機能します）。
