# 評価と計画遷移を進める

## 起動条件・入力・期待出力

- 起動条件: 「計画を評価したい」「修正を差し戻したい」「done/archiveへ進めたい」。
- 入力: 対象planまたは子計画、規模、親programの人間確認方針、子の人間ゲート、実装diff、事前に定義した完了条件、既存の評価・修正文書。
- 期待出力: 範囲を明記した評価MD、必要時の修正MD、親マップまたはplanへの結果、次の段階または人間確認事項。

## 手順

1. 規模に応じた評価範囲を `plan-registry/AGENTS.md` で確認する。サクッとは自己確認、ライトは1パス、フルは探索を含み修正上限2回とする。
2. evaluatorが、planの完了条件を対象ファイル・節・期待状態に照らして採点する。`plan-ops` の `scripts/check-section.sh` は節内の機械確認だけに使い、合否を自動決定させない。
3. 評価結果をarea規約の命名・置き場でMDに残す（programの子は `<program>/評価/NN-子名-評価RR.md`、単発planは plan.md 隣接。2026-07-17より前の旧配置は読み取り互換で移動しない）。FAILなら次の `修正NN.md` を評価と同じ場所に置き、対象、現状、期待状態、具体的な修正、触らない範囲を記してimplementerへ渡す。
4. 全PASSなら、子計画マップまたはplan本文へ評価結果と次段階を集約する。`人間ゲート: なし` のprogram子は `完了（全PASS・全体人間確認待ち）` にして閉じ、親の統合評価へ渡す。単発planの最終完了は人間確認へ進む。危険操作を持つ子は、その操作を実行する前だけ `人間確認（操作名）` へ止める。programは `program-lint.sh` でも整合を確認する。
5. done、archive、卒業、移動、push、main反映、runtime露出は人間ゲートを確認してから実行する。`bucketctl` はplanning→activeだけであり、他の遷移を自動化しない。

## 人間ゲート・失敗時・戻り先・完了確認

- 人間ゲート: 移動・削除・卒業・runtime変更・push・main反映は明示承認なしに実行しない。単発フルplanは最終完了前に人間確認を待つ。programの通常子は全PASSで閉じ、親の統合評価後に一度だけ人間確認を待つ。
- 失敗時: 完了条件が無い、対象範囲が曖昧、修正上限超過、承認未取得なら完了・archiveに進めず人間へ戻す。
- 戻り先: 実装内容の変更は対象planのimplementer、子の構成変更は `manage-program.md`、新規計画の経路は `create-or-join.md`。
- 完了確認: 評価MDが全PASS、必要な個別人間ゲート、親マップ/plan本文への集約、状態と物理バケットの一致を確認する。programでは「子ごとの承認待ち」が残らず、親だけが最終人間確認になっていることを確認する。
