# マルチ指揮官体制（3指揮官×各3レーンを衝突なく回す）

## 目的

オーケストレーションを見る指揮官3〜4人が、それぞれ常時3レーン（実働9〜12体）を並列で回せる体制にする。人間の確認キャパ内に収め、Fable退役後（Opus4.8指揮官）でも回る構造にする。決定ログ#3（1プロジェクト=1指揮官・16体構想）の具体化。

## ユーザー裁定

2026-07-03未明（思考フォークでの壁打ち）:

- **管轄共有が最重要**: 指揮官同士が「どこを管轄しているか」を共有し合う仕組みを最優先部品にする。
- サブスク環境: Claude Max $200＋Codex $100。スケール・収益化が見えたら OpenCode 経由の Kimi K2.6／GLM系を実装・レビューの受け皿に使ってよい（切替スキル導入済み）→ レート天井は当面設計の主役にしない。
- 人間キャパ: 確認ストリーム2〜3本は回せる（自己申告）。かつAIの評価軸が固まるほど人間判断は減らす方向（段階的自律）。
- 約1週間後（2026-07-10頃）に Fable→Opus4.8 へ戻る。指揮官業務が Opus4.8 で回る支援構造を先に整える。

2026-07-03朝（指揮官Aチャットでの壁打ち・本program化の起点）:

- 指揮官は基本3本（メイン2＋サブ1）。このチャット=指揮官A。起動プロンプトは残り2枚でよい。
- 1日の型を朝会で始める: 全repo進捗把握→repoごとの壁打ち→指揮官数決定→役割分担→貼り付けプロンプト配布（子01）。
- レビューは規模連動ゲートにする: 小さい修正は速度優先（実装時間以上をレビューに使わない）、大きい実装ほど厳格に（子02）。
- 依頼インボックスは「空白=下書き（回収しない）→立案済=回収対象→回収済み」へ変更。回収依頼はrepo判定して振り分け、サブ指揮官が既定の掴み手（手すきの指揮官も可）（子03）。
- サブスク構成表を持つ（毎月変わるため。置き場=指揮官ロースター内の節）。
- 各プロジェクトrepoに個別デイリーは作らない（デイリーは my-brain 1枚が正本・ダイジェストがrepo別集計済み。契約§5二重管理禁止）。

2026-07-03午前（指揮官Aチャット・子01壁打ち）:

- 指揮官はサブ含め最大4体（人間が管理できる上限）。既定は3。
- 指揮官モデルは基本Opus4.8へ寄せる予定。朝会冒頭でusage残を都度確認する（Claude Code=/usage・Codex CLI=/status）。
- 今日のTODOはAIが更新してよい: 繰越・朝会結論の追記は基本AI委託（（朝会）タグ・既存行の削除書換は人間）。人間もこれまで通り書ける。
- 朝会の結論・サマリは全てデイリーへ集約する（専用ファイルを作らない）。
- 動いているrepo/プロジェクトは毎日確認する: 朝会＋18:30・23:30の夜会（ダイジェスト3本に同期）。
- 貼り付けプロンプトは保存しない（初回分のみreferences残置）。ロースター「今日の編成」は毎朝書き換え。
- skill名は morning-routine。「おはよう」等でも発火。既存ゴール/朝夜ルーティン.mdは吸収してポインタ化（類似発火スキルのスキャン=削除対象なし）。
- 指揮官の無報告停滞を防ぐ設計を後続で持つ（方針8新設。claude /loop非依存・sonnet5等の安価サブエージェントに現状調査させる）。
- skill-creator-custom の監査・改善は指揮官A預かり（当日TODOに起票・後日計画化）。

## 全体像（目指す1日の型）

1. 朝会（子01）: 前日デイリー＋計画ボード＋repo概要で全体把握→人間と壁打ち→指揮官数・役割決定→ロースター更新→プロンプト配布。
2. 各指揮官: 持ち場の計画を人間と壁打ちで詰め→plan-triage（規模/経路/起動形/モデル）→orca-cockpitでレーン起動→cockpit-supervisorでWAKE対応。
3. レビュー: 規模連動ゲート（子02）。サクッと=AIレビュー省略＋人間へ即diff報告／ライト=1パス／フル=現行の厳格レビュー。
4. 依頼の入口: Notionインボックス（子03）。立案済→回収→振り分け→run-card→手すき指揮官がclaim。
5. 状態共有: 管轄の正本=説明書/指揮官ロースター.md（朝会で更新・節目に「今」を1行更新）。実況はレーン実況（Notion/デイリーauto区画）が担う。
6. 夜: ダイジェスト（23:30最終レンダ）と明日へ節（隣接計画）で翌朝へ繋ぐ。

## 方針（起票時7項の現在地）

1. **管轄の正本（最優先）**: (b)指揮官ロースター1ファイル と (d)思考フォーク=監督権なし明記 → **子01へ**。(a) cockpit.sh の up/send イベントとレーンタイトルへの owner フィールド追加、(c) レーン実況（デイリー＋Notion）と keeper 通知への管轄列 → 後続候補（子01の後段）。
2. **同一repo同時レーンの衝突防止**: 構成カードに「触るパス」を宣言→ up 時に active レーンの宣言との交差を警告。main マージは担当指揮官のみ・repoごと直列・マージ前 `git log main..`（運用契約へ1行追記）→ 後続候補。当面はロースターのキャパ規約＋起動プロンプトの管轄宣言で手動運用（本日3指揮官並行がその実地試験）。
3. **キャパ規約**: 指揮官1人=常時3レーン目安・レビュー同時実行は全体で2本まで → **ロースターに記載（子01）**。ダイジェストの所要時間統計に「人間確認待ちの滞留時間」を追加 → 後続候補。
4. **段階的自律（人間ゲートの動的降格）**: 種別ごとに実績条件を満たしたら「事前承認→事後報告」へ降格候補として提示し、人間が承認して初めて降格。事故時は即昇格で戻す。降格・昇格は決定ログに1行 → 後続候補（子02のレビューゲートが最初の適用面になる想定）。
5. **エンジン段構え**: 既定は現行（実装=sonnet5／レビュー=codex gpt-5.5 xhigh）。OpenCode（Kimi K2.6等）を非クリティカルレーンで統計比較してから広げる → 後続候補。
6. **Opus4.8指揮官の支援構造**: 指揮官の判断を型に外在化するほどモデル依存は下がる。(a)起動定型（読む順・状態再構築）→ **子01の朝会＋起動プロンプトが実装**。(b)裁定チェックリストの明文化 → 子02がレビュー分の明文化を担い、残りは後続候補。(c)重い裁定の逃げ道（人間へ） (d)Opus fastモード活用 → 後続候補。
7. **実地評価セッション**: OrcaでOpus4.8指揮官を起動し実地でオーケストレーションさせ、課題リスト化→本programへ反映 → 後続候補（Fable切替前の2026-07-10頃までに実施）。
8. **無報告停滞の取りこぼし防止（エンジン非依存・2026-07-03追加）**: 指揮官やレーンが報告なしで止まっても人間が気づける層を、claude /loop に依存しない形で持つ。watch-keeper の拡張として、定期巡回で orca ps とロースターを照合し、安価なサブエージェント（sonnet5等）に現状レポートを作らせて通知・デイリーへ流す → 後続候補（実装正本は ../2026-07-02-見張り番キーパー/plan.md へ寄せ、本programは要件だけ持つ）。
