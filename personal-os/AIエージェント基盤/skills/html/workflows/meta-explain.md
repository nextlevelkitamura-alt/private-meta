# メタ説明

メタ領域を、人間が理解しないまま進めないための理解ゲート。
「現状こうなっている（→ こう変える → 新しい本文はこれ）」を1枚のHTMLにまとめ、人間が「理解した・これでいい」と言うまで同じHTMLを更新する。合意前は対象を実装しない。

## 1. 対象とモードを特定する

1. 説明のみ（現状理解）か、変更前合意かを特定する。
2. 何の話かを1〜3行で復唱し、曖昧なら短く確認する。
3. 対象の正本は、最寄りの `AGENTS.md` → `personal-os/AIエージェント基盤/AGENTS.md` → 該当registryの順で特定する。
4. 推測で正本を決めない。

## 2. 現状を調査する

1. 正本・導線・runtime露出・関連hook / loop / Skillを実ファイルで確認する。
2. 「今こう動いている」をpathと本文を根拠に整理する。
3. 確認できなかった箇所は「推測」と明記する。
4. この段階では説明対象を編集しない。

## 3. 説明HTMLを作る

1. 構成・見せ方は `references/meta-explain-layout.md` に従う。
2. 共通のHTML規約は `references/html-structure.md`、品質は `references/mode-full.md` に従う。
3. 変更前合意では、新規・変更ファイルの本文ドラフト全文をHTML内に保持する。
4. スマホ・ブラウザ・URL表示の明示依頼時だけ `workflows/mobile-preview.md` を使う。

## 4. 理解ループを回す

1. 表題付きでHTMLのURLまたはローカルリンクを提示する。
2. 指摘ごとに同じHTMLを更新し、更新履歴へvNを積む。新しいURLを増やさない。
3. 未決事項への回答が全部そろうまで、合意とみなさない。
4. 同じ箇所に2回指摘が来たら、説明内容だけでなく見せ方を変える。
5. 「未決事項が全て回答済み」かつ「理解した・これでいい」の明示で合意とする。

## 5. 合意後につなぐ

1. 説明のみモードは、理解の明示で終了する。
2. 変更前合意モードは、合意内容を計画正本へ反映することを提案する。
3. 人間の承認後だけ計画を書く。
4. 実装・削除・移動・改名・symlink変更はこのworkflowでは行わず、実装レーンへ渡す。

## 6. 安全方針

1. 手順1〜4は読み取りとArtifact公開のみ。説明対象を編集しない。
2. 計画反映は人間の承認後だけ行う。
3. secret・token・credential・認証値をHTMLに載せない。
4. 本workflowは自発起動しない。
