# 01 現役自動処理台帳と責務境界

## 目的

GlobalとFocusmapに散らばるhook / loop / service / agent内部巡回を全件列挙し、「何を現役とみなすか」「誰が書くか」「どこが正本か」を固定する。UIやfolder変更より先に、現実と文書の差をなくす。

## 現状

### Global

- runtime hook: session-board 1機構。SessionStart / UserPromptSubmit / Stop / SubagentStart / SubagentStop。
- scheduled loop: board-reconcile（5分）、board-sweep（60分）、daily-notion-sync（30秒）、session-record-prune（月水金18時）。
- Claude settingsにはhooks-registry外のOrca系hookも存在し、Global一覧には載っていない。
- `registered.sh` はruntime診断、`loops-registry/実行一覧/personal-os.md` はloop一覧だが、hook / loop / serviceを横断する台帳はない。

### Focusmap

- repo-local runtime hook: Claude / Codexのauto-commit、push guard、Claudeの編集後lint / 記録。
- service: Codex app-server。実機は旧labelだけloaded。
- focusmap-agent内部: heartbeat、task claim、command、Codex monitor。
- Web: task progress、ai_tasks、runner status、詳細activity、watchの複数poll。
- legacy: `scripts/task-runner.ts` と再導入導線。Mac appは通常これを止めてagentへ寄せる設計。
- `src/hooks/` 52ファイルはReact custom hookであり、runtime hook台帳には入れない。運用上のpollだけserviceのconsumerとして記録する。

## 方針

初期inventoryは1 unit 1項目とし、次を**暫定採取項目**にする。恒久contractの正本は子04で作る `自動処理契約.md` だけとし、本子計画をschema正本にしない。

- `id` / `kind`（hook / loop / service / internal-loop）/ `scope`（global / repo-local）/ owner。
- 意図状態（enabled / paused / legacy）と実機状態（loaded / running / stopped / unknown）。
- purpose / trigger / period / runner / entrypoint / runtime登録先。
- reads / writes / source_of_truth / mirrors / UI consumer。
- lock / idempotency / retry / timeout / failure mode / non-blocking。
- state / log / healthcheck / tests / verify / stop / rollback。
- secretは値でなくpointer種別のみ。
- human gate / plan backlink / replacement（legacyの場合）。

inventoryは実装本文をコピーせず、実体pathと契約の短い要約だけを持つ。意図状態と実機状態を混ぜない。

## 初期分類で必ず含めるunit

1. session-board runtime eventsと共有本体。
2. Global 4 loops。
3. Focusmap `.claude` / `.codex` repo-local hooks。
4. Codex app-server / Focusmap Mac supervisor。
5. focusmap-agentと内部4 loop。
6. 旧task-runnerとinstaller / setupの再導入経路。
7. Focusmap Webの主要poll / watch consumer。
8. Turso / Supabase writerとsnapshot consumer。

## 完了条件（レビュー項目）

- [ ] inventoryに上記8群が重複なく入り、各unitの実体path・runtime登録先・停止方法が解決している。
- [ ] `src/hooks/`のReact hookがruntime hookとして誤分類されず、運用pollだけがconsumerとして結び付く。
- [ ] 全unitで意図状態と実機状態が別々に記録され、旧labelだけloadedという現状を表現できる。
- [ ] 各state / table / fileにwriterが1つ定義され、複数writerは「競合」または「mirror」として明示される。
- [ ] secret値を含まず、pointer種別だけが記録される。
- [ ] 既存計画へのbacklinkがあり、新計画へ仕様本文を二重コピーしていない。
