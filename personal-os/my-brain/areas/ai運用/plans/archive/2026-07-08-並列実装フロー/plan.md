# 並列実装フロー（スキャン→分岐→回収→監督）

## 目的

実装を「1チャット直列」だけでなく、安全に並列化できる型へ拡張する。
計画から並列可能なレーンを見つけ（スキャン）、worktree で分岐し、テストゲート付きで必ず main へ
回収し、監督者が統合検証する——この一連を毎回手作りせず再利用できる運用の型にする。

## 現状

- 実装は1チャット直列がメイン。並列化は 2026-07-08 の子03実装（計画実行フロー統一）で初めて
  手作りした（worktree 3レーン・自己完結プロンプト・メイン監督。学びは方針に反映済み）。
- session-board に目的別ビュー（同一目標にN体）が入り、計画列（子03）も実装中で、
  「誰がどの計画を拠り所に並列で動いているか」の可視化基盤は整いつつある。
- 並列化の判断基準・分岐/回収の規約・監督の役割定義はどこにも無い（毎回ゼロから設計になる）。

## 方針（未確定・2026-07-08時点の構想）

型は4段: **スキャン → 分岐 → 回収 → 監督**。

1. **スキャン（並列性の分析）**
   - 計画mdの実装手順から依存グラフを作り、レーンに割る。判定3軸:
     ①ファイル所有の非交差（レーン間で同一ファイルを触らない担当マップが作れるか）
     ②手順の依存方向（Aの出力にBが依存するなら同一レーンで直列のまま）
     ③本番稼働物への接触（触る工程は並列にせずメイン専有）。
   - 並列化する/しないの判定: 非交差レーンが2本以上作れて、クリティカルパスが実際に短くなる時だけ。
     1〜2ファイルの小改修は直列のまま（`GLOBAL_AGENTS.md` §7の規模判定と接続）。
2. **分岐（worktree規約）**
   - 並列開始前に**基点コミット**を必ず作る（計画md・テスト・仕様が HEAD に揃った状態。
     worktree は HEAD から切るため、未コミットの計画はワーカーに渡らない——実地の教訓）。
   - branch 命名 `wt/<テーマ>-<役割>`／worktree 置き場 `~/wt-<名前>`（`~/Private` 直下に作らない）。
   - レーンごとに担当ファイルマップを宣言。共有ホットスポット（program.md・実ボード・本番フック等）
     はメインが専有し、ワーカーのプロンプトで明示的に禁止する。
3. **回収（マージプロトコル）**
   - ワーカーの完了手順: テスト全緑 → パス指定コミット → main側の担当パスがクリーンか確認 →
     `merge --no-ff` → スモーク → `worktree remove`（ブランチは人間確認まで残す）。
   - コンフリクトは abort して報告（強行しない）。push は人間の明示依頼時のみ。
   - 本番稼働物（フック・loop等）を含むレーンは「マージ=デプロイ」と扱い、直後スモークと
     `git revert -m 1` の巻き戻し手順をプロンプトに埋め込む。
4. **監督（オーケストレーション）**
   - 監督者=メインチャット。役割: レーン設計・自己完結プロンプト生成・統合検証（実地確認・
     docs↔実装照合・lint）・人間への確認。**監督者は実装しない**（書いた本人と別の目線で検証）。
   - ワーカー指示書は自己完結プロンプト（worktree作成〜回収まで埋め込み）。正本仕様は worktree 内の
     計画mdを読ませ、プロンプトに仕様本文をコピーしない（二重管理禁止）。
   - 可視化: session-board の目標合流（目的別ビュー「この目標にN体」）＋計画列 `--plan`。
   - 完了合図: 当面は人間が中継（ワーカー完了→メインへ一言。全レーン揃いで1回が基本）。

裁定済み（2026-07-08 ユーザー）:

- **成果物の形: Global Skill 化はしない。** Skill は AI が説明を見て選ぶ仕組みで、フックのような
  発火保証が無い（現に hooks が確実に効いているのは md を注入・参照させているから）。型の正本は
  md で置き、フック等の確実に読み込まれる導線から参照させる。置き場と導線の設計は実装時（完了条件へ）。
- **モデル選定: 基盤に「AIモデル一覧」md を新設。** 使用中のサブスクと、どの用途にどのモデルを
  使うかをシンプルに1枚（決定ログ#11 のレーン規約もここへ集約）。本フロー含め各所はコピーせず
  そこを参照する（単一正本）。置き場=基盤直下 `AIエージェント基盤/AIモデル一覧.md` に確定
  （07-08・骨組み作成済み。契約サブスクの実態記入は人間待ち）。
- **ai-jobs: 棲み分けの明文化ではなく縮小方向。** 現アーキテクチャ（session-board＋対話ワーカー）
  で機能しておらず、実態も ready/running/review 全空・done 1件（07-02）のみ（`review`/`reviewing`
  の重複フォルダも放置状態）。本フローが実行の主経路になる。縮小の実行（areas/AGENTS.md §4.2 の
  書き換え・残カード整理・フォルダ畳み）は別計画。ただし `AIJOBS_RUN` ガード（headless をボードに
  載せない・common.py）はキューと独立に有効なので残す（将来の自動実行でも使う）。
- **監督の自動化: 段階ロードマップで人間の手を徐々に離す。**
  フェーズ1（いま）= 監督がプロンプト生成 → 人間が新チャットへ貼る → 完了は人間中継。
  フェーズ2（型が安定後）= Orca CLI 等で起動〜コミット〜main マージまで自動化し、
  **push の最終確認だけ人間に残す**。Orca は既に worktree 方式で動いており親和性が高い
  （`~/orca/workspaces/` に Private の worktree が現存）。まず本計画で基盤（型md・AIモデル一覧・
  規約）を固めるのが先。

- AIモデル一覧の中身充足（契約サブスク実態の記入は人間待ち・骨組みは作成済み）。
