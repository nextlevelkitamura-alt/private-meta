# 儀式分配設計（morning-routine → Standup / Wrap / Shutdown）

親計画: `../program.md` ／ 子計画: `../plans/03-儀式の自動実行.md`
位置づけ: 設計書 兼 **第2波の実装指示書**（第1波W4作成・2026-07-09）。
第1波の実装済み範囲は `skills/standup/SKILL.md`（骨格・未配線）と本書のみ。wrap/shutdown スキル・daily-shutdown loop・scheduled task 登録・morning-routine 縮退はすべて第2波。

## 1. 分配マップ（morning-routine の各手順がどこへ行くか）

分配元: 基盤 `skills/morning-routine/SKILL.md`（朝会モード⓪〜⑥・夜会モード）。読み方: 「元手順 → 行き先（扱い）」。

朝会モード:

- ⓪ usage残確認・今日の指揮官モデル決定 → **残す（指揮官編成側）**。編成判断の入力なので Standup には入れない（責務分離）。
- ① 前日デイリー・当日朝転記区画を読む → **Standup ①②**（封印確認・継承リスト収集）へ吸収。
- ② 計画ボード×repo-registry×projects/active の壁打ち → **Standup ③④**へ置換。計画ボードは renderer 廃止（2026-07-04）で消滅済みで、後継の面は月間計画（今月の的・週の大枠・予定）。repo編成のための深掘り（どのrepoに指揮官を張るか）は編成側に残る。
- ③ 今日の指揮官数の決定 → **残す（指揮官編成側）**。
- ④ ロースター「今日の編成」記入 → **残す（指揮官編成側）**。
- ⑤ 指揮官起動プロンプト生成（`references/commander-prompt.md`）→ **残す（指揮官編成側）**。references/ も編成側スキルが持ち続ける。
- ⑥ 今日のTODO更新＋インボックス振り分け → **Standup ④⑥**へ置換（「今日のTODO」節は8節テンプレで「今日すること」業務行に代わる）。インボックス未処理行の振り分けは kickoff／inbox-patrol 経路（子06）へ。

夜会モード:

- 18:30 状況報告・TODO消し込み → **Wrap（18:20）**へ: 仕事系業務の締め（%確定の作用は board.py 系・判断はAI）・質問キュー消化・今夜の1行確認。
- ロースター「今」の照合・更新 → **残す（指揮官編成側）**。Wrap の手順からは外す（Wrap は対人の仕事締めに限定）。
- 23:30 worktree-sweep 実行 → **第2波で判断（人間ゲート）**。候補a: Shutdown の Python 枠に「検知のみ・結果はログへ」で載せる／候補b: 編成側スキルに残す。down/削除の実行は従来どおり常に人間承認。
- 23:30 「明日へ」記入の促し → **Shutdown（23:00）**へ: LLM が明日へを作文（stdout契約・§3）。翌朝 Standup ② が受け取る。
- 23:30 逆算Alignmentチェック → **Standup ⑤（ズレ調停）＋月曜Weekly拡張**へ吸収。①今日の業務がladderに繋がるか＝ズレ調停、②放置目標・④動かない計画＝Weekly/Monthly の照合、③人間確認待ち滞留＝Wrapの質問キュー消化が受け皿。

morning-routine 本体の扱い（**廃止/縮退の最終判断は人間ゲート**・第2波）:

- 案A（推奨）: 指揮官編成専用へ縮退。上で「残す」とした⓪③④⑤＋ロースター照合だけにし、名前も編成専用（例: `commander-briefing`）へ改名。朝会・夜会の発火語（おはよう・朝会・夜会・締め・おつかれ 等）は description から削除する。旧発火語を standup/wrap へ機械的に付け替えることはしない（「おはよう」で儀式が暴発する誤発火を避ける。儀式の入口は /standup 等の明示語と scheduled task に限定）。
- 案B: 廃止し、編成手順を `説明書/指揮官ロースター.md` へ吸収（morning-routine は skill-delete で削除）。
- どちらの案でも skill-creator-custom →（削除時は）skill-delete の正規ゲートを通す。子03レビュー項目「morning-routine側に旧発火語・重複記述が残っていない」をここで判定する。

## 2. Desktop scheduled tasks 登録手順（人間実行・第2波）

前提: Claude Code Desktop 2.1.201+（公式 scheduled-tasks / desktop-scheduled-tasks 機能）。制約: アプリ起動中＋Mac稼働時のみ発火。スリープで逃した回は wake 時に直近1回キャッチアップ発火（→ ガード文が必須）。

登録する task は2本:

1. **Standup**: 平日（月〜金）朝。時刻は人間が決める（例 08:00。起床後にMacを開いている時間帯）。
2. **Wrap**: 平日（月〜金）18:20。

登録方法: Desktop のチャットから scheduled task を作成する（「毎平日8:00にこのプロンプトを実行して」等。公式ドキュメント code.claude.com/docs の desktop-scheduled-tasks を参照）。プロンプト文案（そのまま登録・ガード文込み）:

Standup 用:

```
/standup で朝の儀式を実行してください。
手順正本: ~/Private/personal-os/AIエージェント基盤/skills/standup/SKILL.md（本文をここへコピーしない）。
ガード: 現在時刻が17:00以降なら当日ドラフトは作らず、前日の封印状態と当日ボードの要約だけ報告して終了してください（スリープ復帰のキャッチアップ発火対策）。
```

Wrap 用:

```
/wrap で夕方の締めを実行してください。
手順正本: ~/Private/personal-os/AIエージェント基盤/skills/wrap/SKILL.md（本文をここへコピーしない）。
ガード: 現在時刻が21:00以降なら締め作業はせず要約のみ報告して終了してください（封印はShutdownに委ねる）。
```

規律:

- 登録で `~/.claude/scheduled-tasks/<task>/SKILL.md` が生成される場合、**薄いポインタに保つ**（基盤 skills/ への参照＋ガード文のみ。手順本文のコピー禁止）。登録直後に中身を確認し、厚ければ人間が削ってよい。
- 手順正本が基盤 `skills/standup|wrap/SKILL.md` のみであること（子03レビュー項目5）をここで確認する。
- 時刻・曜日の変更は task 側だけで完結させ、SKILL.md に時刻をハードコードしない（Wrap 18:20 のような固定裁定は program.md が正本）。

## 3. Shutdown loop（`loops-registry/loops/daily-shutdown/`・新設・第2波）

型: `loops-registry/loops/session-record-prune/`（launchd 型 loop の最新雛形）を踏襲する。

構成:

```text
loops-registry/loops/daily-shutdown/
  loop.md                             # 実行スペック。frontmatter 稼働状態=draft/未ロード→有効化は人間ゲート。設計backlink=子03
  com.kitamura.daily-shutdown.plist   # StartCalendarInterval 毎日 23:00（Weekdayキー無し）・RunAtLoadなし
  scripts/
    shutdown.sh                       # launchd入口（shutdown.py --apply を1回だけ）
    shutdown.py                       # Python枠（決定的・冪等）
    ensure-daily.sh                   # rendererから救出（§4）
    carry-tomorrow.sh                 # rendererから救出（§4）
  tests/
    test_shutdown.py                  # env sandbox・LLM stub・冪等/失敗経路
  output/logs/                        # daily-shutdown.{out,err}.log
```

plist は session-record-prune と同型: `/bin/zsh -lc 'cd <loopフォルダ> && scripts/shutdown.sh'`・EnvironmentVariables（HOME/PATH/TZ=Asia/Tokyo）・StandardOut/ErrPath は `output/logs/` 配下・RunAtLoad なし・symlink 方式で `~/Library/LaunchAgents/` へ露出し `launchctl bootstrap`/`enable` で有効化（停止は `bootout`）。有効化・停止したら `loops-registry/実行一覧/personal-os.md` を同じ作業で更新。発火はシステムTZ基準・逃した回は wake 時に1回（launchd 仕様）。

責務分割 — **Python=枠（記録を落とさない）／LLM=意味づけ（stdout契約のみ・ファイルを書かない）**:

shutdown.py（Python枠）の処理順:

1. 対象デイリーの特定。env で差し替え可能にする（`GOAL_BASE` / `SESSION_BOARD_DATE` — session-board テスト規約と同じ語彙）。テストは実ファイル非接触。
2. **board.py と同一の flock を取ってから書く**（子04と同じ絶対条件。デイリーへの並行書き込みで行を落とさない）。
3. 冪等ガード: 当日 `## クローズ` に `closed` 行が既にあれば no-op で正常終了（同一入力の2回連続実行は差分ゼロ＝子03レビュー項目）。
4. LLM 呼び出し: env `DAILY_SHUTDOWN_LLM_CMD` で差し替え（既定 `claude -p`・タイムアウト付き）。非ゼロ終了・タイムアウト・stdout の parse 不能は全て「LLM失敗」に分類（半端な取り込みをしない）。
5. LLM成功時のみ: ⏸最終判定を既存の作用器（`board.py finish`・`[auto]` プレフィックス＋根拠必須＝契約§5）経由で反映し、「明日へ」作文を当日 `## 明日へ` へ機械行として追記。
6. 達成率確定: 当日「今日すること」業務行の最終%を読み取り統計（完了n・持越n・追加n・⏸残n）を計算する。%の推定そのものは子04エンジンの仕事で、shutdown は**読むだけ**（書き換えない）。
7. 封印: `board.py close` を呼ぶ（クローズ節・closed 行・統計の書式の正本は board.py 側＝子02段階4。shutdown.py に封印書式を二重実装しない）。
8. 週大枠書き戻し: 当月 `月間計画.md` の `## 照合記録` へ当日1行を書き戻す（日付キーで冪等 upsert・マーカー運用。人間の的の文面・`## 今月の的` 本文には触らない＝機械は専用行のみ）。
9. 翌日デイリー生成＋継承転記: `ensure-daily.sh`（テンプレから生成・既存なら不触）→ `carry-tomorrow.sh`（当日「明日へ」を翌日ファイルへ転記・§4）。
10. **LLM失敗時**: 手順5だけをスキップし、6〜9 を完遂して封印まで終える。クローズ節へ「LLM未処理」フラグ1行を残す（翌朝 Standup ① が検知して意味づけを追補できる）。
11. git commit / push はしない（子03レビュー項目）。全体失敗時は exit 非0（launchd ログで検知）。

LLM stdout 契約（案・第2波で凍結）: LLM はファイルを書かず、stdout に次の2ブロックのみを出す。

```text
明日へ:
- <持ち越し・申し送り 1行ずつ>
⏸判定:
- <s:KEY> 完了|残す ｜根拠: <台帳名 or 証跡1行>
```

- 契約外の行が混ざったら全体を「LLM失敗」扱いにする（parse できた分だけ取り込む中途半端をしない）。
- ⏸判定の一次担当は子05（board-sweep）。Shutdown の判定は「その日最後の残り」への最終確認のみで、`[auto]`＋根拠の語彙は契約§5 に従う。

tests/test_shutdown.py（Python・session-record-prune / session-board のテスト流儀）:

- env sandbox（GOAL_BASE 等）で仮デイリー・仮月間計画を組み立てる。LLM は `DAILY_SHUTDOWN_LLM_CMD` に echo スクリプトを差して stub。
- 最低限のケース: 封印冪等（2回実行差分ゼロ）／LLM失敗経路（フラグが残り封印は完了）／週大枠書き戻し冪等／翌日生成＋転記／closed 済み no-op／commit されないこと。

`/shutdown` スキル（`skills/shutdown/`・第2波新設）: 同じ shutdown.py を対話から叩く薄い窓。手順本文を持たない（正本一元）。
