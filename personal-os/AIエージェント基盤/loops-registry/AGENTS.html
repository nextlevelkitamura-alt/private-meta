まず押さえる3つの正本

実装の正本

global または所有repo

実行の正本

launchd / plist

実機でいつ動くかはplistと launchctl が決めます。基盤のsymlinkは実行経路にしません。

全体の正本

現在は一覧、将来はTurso

移動は別作業

既存の scripts/、仕事repoの業務自動化、Focusmap Agentを、標準root導入と同時に移動しません。対象・launchd backup・rollbackを決め、人間が承認した移行だけが実体を動かします。

loopでないものを混ぜない

自動で動くものでも、発火点と所有責務が違えば置き場所を分けます。

loop 時刻・間隔で繰り返す処理

置く場所: globalなら基盤の loops/、repo固有なら将来のrepo loops/。目的・発火・停止手順を loop.md に残します。

hook runtimeイベント直後の処理

置く場所: hooks-registry/ またはrepo-local hook。定時実行ではないためloop棚へ置きません。

手動コマンド / Skill 人の依頼で動く処理

置く場所: 所有Skillまたはrepoの scripts/。繰り返し自動発火に成熟するまではloopにしません。

AI実装・レビュー・采配 人の方向修正が前提の作業

置く場所: Skill / orchestration / 可視ペイン。無人の定期実行にする前に、まず対話的な運用で安定させます。

Loop Creatorが新設を一連の作業にする

作成、登録、リンクを別々にすると漏れるため、将来のGlobal Skillは一つの完了条件として扱います。

  1. 1
    repo-registryで所有repoを確定する

    repoの役割とcanonical pathをここで選び、名前や絶対パスを推測しません。

  2. 2
    対象repoのAGENTS.mdでloop rootを確認する

    loops/<loop-id>/ の宣言がなければfail-closed。勝手に scripts/ を探索しません。

  3. 3
    必要最小限の定義を作る

    必須はloop.mdだけです。必要な時だけplist、scripts、gitignore対象のlogsを追加し、空の標準フォルダは作りません。

  4. 4
    基盤にrepo rootをリンクする

    implementation-links/<repo-id> → <repo>/loops/ の相対directory symlinkを検証します。個別loopのリンクは不要です。

  5. 5
    registryへsource referenceを登録する

    移行期間は一覧、切替後はTurso definitionへcanonical pathを登録します。launchdの有効化は人間承認後だけです。

symlinkは入口であり、実行や削除の基準ではない

リンクを便利な抜け道にしないことで、repo所有権と実行の安定性を守ります。

実行

launchdはcanonical path

plistは所有repoまたは基盤の実体を指します。基盤リンクの改名・再作成が稼働中の実行を壊しません。

削除

リンクを辿って削除しない

削除前に実体・依存・loaded状態を確認します。リンクを消しても実体は消えず、実体削除は別の人間ゲートです。

実行時データ

lockは/tmp、ログは必要時だけ

lockは原則/tmp、ファイルログは必要な時だけgitignoreされたlogs/またはrepo外へ置きます。永続stateと成果物は目的に合う正本を明示します。
