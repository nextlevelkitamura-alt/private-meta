# html-structure（共通の土台）

`html` Skillの共通規約。型は1つで、モード選択は無い。部品のCSS・雛形の実体は `assets/artifact-template.html`、実装説明に固有の必須要素は `impl-explain.md`。

## 1. 読者想定（全ルールの前提）

1. 読む人は「コーディングしないが、この内容のGO / NGを判断する人」。迷ったら常に判断者目線に倒す。
2. 専門用語は初出でその場で言い換える（例: symlink（＝実体を別の場所から参照するショートカットのようなファイル））。ページ末尾の用語集は作らない。
3. コード・設定・コマンドを見せたら、直後に「つまり: 」の日常語1行訳を必ず添える。
4. コードネーム・略語・独自番号は初出で定義する。

## 2. サンドイッチ構造（ページの並び）

どの内容も上からこの並びで作る。

1. **① 結論**: 1〜2文・専門用語なし。ここだけ読めば概要が分かる。
2. **② 先に質問**: 人間に判断してほしいことを1行ずつ・番号つきで予告する。
3. **③ 詳しい説明**: 図解が主役（§5の選択ラダー）。節の型は§7。実装・変更の説明では `impl-explain.md` §1の3点セット（どこを触る → こう変わる → 中身の意味）を必ず入れる。
4. **④ もう一度質問**: ②と同じ質問に推奨案を付けて、ここで答えてもらう。
5. 判断してほしいことが無ければ②④を省き、①③だけにする（レポート・調査まとめ・ダッシュボードはこの形）。合意待ちだけなら④に「未決なし・OK待ち」と書く。
6. 指摘を受けたら同じHTMLを更新し、更新履歴にvN（指摘 → どう直したか）を積む。

## 3. Artifact制約（守らないと公開が壊れる）

1. 外部リソース禁止(CSP): CSS/JSはインライン、画像は data URI、フォントは system。
2. バックエンド無し: フォーム保存・view時API・複数ページは不可。表示専用と割り切る。
3. 単一ページ: 相対リンクは解決しない。ページ内アンカーを使う。
4. サイズ: 16MiB以下。大きな画像を埋め込まない。
5. ファイル形式: `.html` / `.htm` / `.md`。

## 4. 見やすさの基本

1. 文章は結論を先に出す。
2. 内容は部品に置き換える。「詳細＝表」ではない。詳細の既定の受け皿は `.deflist`（スタック定義ブロック）。
3. 1項目の説明は2行まで。超える説明は、図に描き直すか `details` へ落とす。
4. 色は意味に割り当てる（推奨=緑、注意=赤・琥珀、強調は1色だけ）。状態は色だけでなく文言・pillでも示す。
5. 1枚で完結させ、節が5つ以上なら冒頭にアンカーチップを置く。
6. 仕組み・定義・設定の説明は、逐条解説より「入力→処理→出力」の実文サンプルで追わせる方が伝わる。原文全文が必要な場合は `details` に折りたたみ、ページの主役はウォークスルーにする。

## 5. 部品の選択ラダー（毎回、上から順に判定して最初に当たった部品を使う）

1. **図解判定（1段目・既定）**: 主題が構造・流れ・関係・層・差分・経緯のどれかに当たる → **図が主役**。`diagram-recipes.md` §1の対応表でレシピ（R1〜R6）を選ぶ。フォルダ構成はボックス型ツリー図（R1）が主役で、テキストの `.tree` は details内の補足・全量列挙用。量の目安と逃げ道（長い理由・逐条規約はテキストでよい・装飾だけの図は禁止）は同 §2。
2. 同じ型の短い事実が行×列に揃い、§6の5条件を全部満たす → 表。
3. キーと値が1対1で短く揃う（スペック・設定値・メタデータ） → `.kv`。
4. 順序に意味があるが図にするほどの分岐・関係がない → `.steps`。
5. 実行フロー・コマンドや対話のやり取りを再現する → `.steps` の各stepに `.codeblock` で実文（入力・コマンド・出力のサンプル）を入れる（ウォークスルー型）。
6. ちょうど2案の対比（Before/After・案A/案B）で構造差でなく文言・条件の差 → `.compare2`。3案以上は `.cards`。
7. どれでもない＝説明が主役（理由・背景・条件が混ざる項目の列挙） → `.deflist`。3件以下で各1〜2行なら `.cards` でもよい。
8. いつでも併用: 本筋でない補足・ログ・長い根拠 → `details`（折りたたみ・初期は閉じる・結論は隠さない）／リスク・前提・例外 → `.callout`（1節1個まで）。

## 6. 表の使用条件（1つでも外れたら `.deflist` へフォールバック）

1. セルは名詞句1つ。目安25字以内。文章・セル内改行を書かない。
2. セルに接続詞（「ため」「しかし」「場合は」等）が出たら、それは表でなく `.deflist` の内容。
3. 行数は8行まで。超えるならグルーピングして分割するか `.deflist` 化。
4. 列数は4列まで。「項目/判定/理由」の3列型を機械的に複製しない（「理由」列は大抵条件2に違反する）。
5. 表は1ページに2箇所まで。3箇所目が必要になったら設計を疑う。

## 7. 節の型（ページ後半でも崩さないフラクタル構造）

1. どの節も「h2見出し → リード1行 → 主役部品1つ → 補足は `details`」の順にする。既定の主役部品は**図＋figcaption**（§5の1段目に当たるとき）で、当たらない節だけテキスト部品が主役になる。目安は主要節の過半で図が主役・1ページ最低1枚（`diagram-recipes.md` §2）。
2. 見出しは「詳細」でなく結論・内容が分かる文言にする。見出しとリードだけ拾い読みして論旨が追える状態を保つ。
3. リード1行は結論から書く。「〜について説明します」のような前置きを書かない。
4. 主役部品は1節に1つ。部品の重ね置き（表＋deflist併置など）をしない。
5. 同じ主役部品を3節連続で使わない。特に表の直後の節に表を置かない（視覚リズム）。
6. 節が2画面分を超えそうなら、節を分割するか後半を `details` へ落とす。

## 8. 書き方（スキャンで読める文）

1. 箇条書き・カード・セルは先頭に太字キーワードを置き、太字だけ拾い読みできる形にする。
2. 本文の行長は日本語で40字前後に抑える（コンテナの max-width で制御。テンプレ既定で担保）。
3. 番号付き見出し（01/02…）は本当の手順・順序にだけ使う。飾りにしない。

## 9. トークンの土台

1. 図はラスター画像でなく SVG か HTML+CSS で描く（軽い・self-contained）。雛形の実体は `assets/artifact-template.html` の図解レシピ雛形（R1〜R6）だけに置き、md側へ複製しない。
2. JSは書かない。開閉はネイティブ `details/summary` で足りる。タブ・カルーセルは使わない。
3. 大きなデータは全量埋め込みでなく、要約して載せる。

## 10. 見た目の一貫性

1. 色・フォント・余白・部品のトークンは `assets/artifact-template.html` を基準にする。
2. 揃えたい場合は `CLAUDE.md` にデザイントークンを置く（Anthropic公式のビルトイン design skill も同じ場所を見る）。
3. 配色は必ず白背景のライト単色にする。ページの画布は `#fff` とし、カード・図・コードブロックを含めて暗い背景を使わない。
4. 表示環境による暗色切替を書かない。どの内容・どの経路でも例外はない。

## 11. いつHTML / いつMarkdown・正本が変わったら

1. HTML（このSkill）: 見て終わり / 共有して終わりの成果物。
2. Markdownのまま: 正本、継続編集するもの、git diff を見るもの、Notion同期対象（デイリー・計画）、短い返信。
3. 迷ったら: AIが後で読み書き・同期するなら md、人間が一度見るだけなら html。
4. HTMLは正本にしない・手で継続編集しない。正本（md/コード）が変わったら作り直す。追従は寿命で3方式から選ぶ: 都度その場生成（既定）／手動再生成（恒久資料の現実解・忘れると陳腐化）／ビルド時自動生成（頻繁に更新する重要ビューだけ・構築コスト要）。
5. AIに読ませる・編集させるのは md 側にする（生HTMLは読解精度が落ち、部分編集も壊れやすい）。生成HTMLはgitの正本にしない。残すなら派生物として分離する。
