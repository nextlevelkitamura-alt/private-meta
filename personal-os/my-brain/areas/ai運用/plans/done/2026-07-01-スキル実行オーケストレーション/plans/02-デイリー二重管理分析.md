## 二重管理の具体箇所

共有コントラクト §5 の基準「索引/ビューは可、状態の複製は不可」で仕分けした。実データ上まだ0件で
「実害は未発生だが構造的に発生する」ものは（構造）と付す。

1. ダッシュボード.md ↔ plans/フォルダ ＋ ai-jobs/フォルダ（計画active・AIジョブ件数）（構造）
   - ダッシュボードは索引だが**手更新**で、自動再生成（dashboard.sh）が未実装。単一 writer が無いため、
     フォルダ実体（正本）と索引がズレる＝実質的な状態複製になりやすい。契約上はビューだが writer 未単一化。
   - 現状は両方0件で一致しているだけ。カードや計画が動いた瞬間にズレるリスクが構造的にある。

2. デイリー `auto:goal` ↔ `3年計画.md`／`年間計画/<年>.md`（的の転記）（構造）
   - 的をデイリーへ表示するビューだが、朝loop（生成器）が未実装。現状は手コピー＝的が2箇所に並ぶ
     状態複製になりやすい。3年/年間側が更新されてもデイリー側は自動追随しない。

3. デイリー内部 `auto:log` ↔ `auto:done`（同一ファイル内の多段ビュー）
   - `auto:log`(hook) が残した当日 commit の sha を、`auto:done`(夜loop) が件名解決して再掲する。
     writer は別だが情報源は同じ `auto:log`。同じ commit が「ログ（自動）」に sha、「実行したこと」に
     sha＋件名で**2回**出る。状態の複製ではない（多段ビュー）が、読み手には冗長。

4. 「実行したこと」人間行 ↔ `auto:done`（実績の意味的重複）（構造）
   - 同一セクションに、人間手書き実績（`- `）とAI自動要約実績（`auto:done`）が同居する。人間が
     「◯◯やった」と書き、夜loopが同じ作業を commit から要約すると、同じ実績が2箇所に並ぶ。マーカーで
     物理分離されているので消し合いはしないが、内容は重複しうる。

5. デイリー `auto:done` の done card 列挙 ↔ ai-jobs/done フォルダ（揮発ビュー）
   - `collect-done-cards.sh` は done カードを「完了カード: <base>／担当／出所」の1行で列挙する。状態の
     正本は依然フォルダ位置なので契約上OKなビュー。ただし**当日 mtime 依存**で、archive へ動くと消える等、
     レンダリング条件が状態遷移に縛られる。現状 done 0件で実害なし。二重管理というより「揮発ビュー」。

判定サマリ: 「状態の複製（禁止）」に該当するものは現時点で**無い**（すべてビュー／索引の形は保っている）。
ただし 1・2・4 は「単一 writer と正本が確立していない索引/ビュー」で、運用が始まると実質的な二重管理へ
転落する。3・5 は冗長/揮発だが正本は単一。

## 論点

1. 何を単一正本にするか。
   - AIジョブ状態＝ai-jobs フォルダ、計画状態＝plans/ フォルダ、の2つは契約で確定済。争点は
     ダッシュボード/デイリーが**手でそれを写す**現状（単一 writer 化＝dashboard.sh 待ち）。
   - 人間の日次「TODO」「実績」は、いまデイリーが**唯一の置き場＝正本**。ここにはレンダリング元が無い。
     つまりデイリーは純粋なビューではなく「人間の日次 state 保管庫」でもある。

2. デイリーを"ビュー"に寄せると何が壊れ/良くなるか。
   - 良くなる: 状態が plans/ai-jobs に集約され、デイリーはいつ再生成しても同じ（冪等）。§3 と整合。
   - 壊れる: 人間の「今日やる」を計画/ai-jobs に書く必要が出る。だが ai-jobs は「human 作業を入れない」
     規律（ai-jobs/AGENTS §4）で、日次 TODO は job でも計画でもない。TODO の正本を新設しない限り、
     デイリーからは剥がせない。朝に手でサッと書く軽さも失われる。

3. 未実装マーカー（`auto:goal`/`auto:carry`/`auto:progress`＋朝loop）をどうするか。
   - 生成器を実装して「デイリー＝夜loop/朝loop が描くビュー」を完成させるか、空マーカーを削除して
     実装済み範囲（log/done/align）に割り切るか。放置は「空の見出し」を生む。

4. 「レンダラは夜loop（単一）」と言い切ってよいか（§3 の文言問題）。
   - 実際は hook(auto:log) と 夜loop(auto:done/align) の2 writer。§3 の単一レンダラ前提は現実と不一致。

## 選択肢（最低2案）

### 案A: デイリー＝daily-digest が描くビューに寄せる（純レンダラ化）

- 内容: 人間 TODO/実績の状態正本をデイリー外（計画／ai-jobs／専用 TODO store）へ移し、デイリーは
  それらをレンダリング。`auto:goal`(朝loop)/`auto:carry`/`auto:progress` の生成器も実装する。
- 利点: 共有コントラクト §3 と完全一致。状態が plans/ai-jobs に集約され二重管理が構造的に消える。
  デイリーは冪等な再生成物になる。
- 欠点: 日次 TODO の正本を新設する必要（ai-jobs は human 作業禁止と衝突）。生成器4系統の実装コストが大。
  朝に手で書く軽さを失う。
- 移行コスト: 大（TODO 正本の新設＋朝loop/carry/progress 実装＋既存運用の切替）。

### 案B: デイリーは「人間の日次 state ＋ 自動ログの追記」に割り切り、§3 を現実に合わせて修正

- 内容: デイリーは人間の TODO/実績を第1状態（正本＝デイリー）として持つと認め、自動マーカーは
  log/done/align の参考ビューに限定。未実装の `auto:goal`/`auto:carry`/`auto:progress` マーカーは**削除**。
  計画/ジョブ状態はダッシュボードが索引を担い、デイリーはそこへリンクするだけ（状態を持たない）。
- 利点: 実装済み範囲で完結し移行が軽い。日次運用が軽いまま。デイリーに計画/ジョブ状態を持ち込まない
  ことで二重管理を回避。§3 を「自動部分はビュー／人間 TODO・実績はデイリーが正本」と正直に書き直せる。
- 欠点: §3 の「第2状態保管庫にしない」を緩める（人間 TODO/実績はデイリーが正本＝state であると認める）。
  デイリーは完全な冪等レンダラにはならない。
- 移行コスト: 小（空マーカー削除＋契約文言修正＋ダッシュボード役割の明文化）。

### 案C（段階案・A と B のブリッジ）

- 内容: 当面 B で運用しつつ、先に `dashboard.sh`（単一 writer）を実装して「計画/ジョブ状態＝script 再生成の
  ビュー」を確立。goal/carry/progress は必要になった時だけ生成器を足して A に近づける。
- 利点: 最大リスク（手更新ダッシュボードのズレ）を先に潰せる。段階的で低リスク。
- 欠点: 中途状態が続く。どこまで A へ寄せるかの判断を都度求められる。
- 移行コスト: 中（dashboard.sh は子02-B に既存依存）。

## 共有コントラクト §3 の判定

§3「デイリーは その日のボード状態＋done ログのレンダリング（ビュー）であって第2の状態保管庫にしない。
レンダラは夜loop daily-digest」は、現実と**部分的にしか一致しない**。

- 不一致1: デイリーは人間の TODO/実績の**第1状態保管庫**でもある（レンダリング元が存在しない情報）。
  §3 の「第2状態保管庫にしない」は自動部分には効くが、人間 TODO/実績には当てはまらない。
- 不一致2: 自動部分の writer は夜loop **だけでなく hook もある**（`auto:log`）。「レンダラは夜loop」の
  単一化は現実と食い違う。
- 不一致3: 6マーカー中4つ（goal/carry/progress＋朝loop）は**生成器未実装**で、ビューが空 or 手埋め。
  「デイリー＝夜loop が描くビュー」は現状 done/align の2マーカーでしか成立していない。

結論: §3 を「不変の前提」として扱うと現実の運用と乖離する。§3 の文言修正（案B）か、実装で §3 に
寄せる（案A）かは設計判断であり、ここでは決めない（未決事項へ）。

## 実行機構の棚卸しと並行制御の原則（会話2で追記・要指揮官/契約確認）

分析の過程でユーザーと確認した、実行機構の集約仮説と near-term 運用方針。§1（状態＝ai-jobsフォルダ）・
§7（人間ゲート）と Stream1（分配設計）に跨るため、ここでは方向性として記録し、契約改訂は指揮官/別フェーズで判断する。

### 実行機構は実質2つに集約できる
- ① ループ（`loops-registry/loops`）: 定期・自動・自己完結。loop.md に「何を・どこで・どのスクリプトで」書けば
  完結する。ai-jobs は不要。
- ② ai-jobs（6フォルダの箱）: 実行状態をフォルダ位置で持つ**並行制御レイヤー**。ロック（mv）が価値を出すのは
  「無人で複数ワーカーが同じ列から同時に取る」場合のみ。
- ③ 計画駆動: 人間が計画（＋レビュー項目）を作り、指揮官(Orca)監督で実装→レビュー。デイリーが「今やるべき」の
  正本。人間ゲートは**入口（着手判断）と出口（最終納品チェック）**の2点のみ、中間の実装/レビューは AI/Orca。
- 現状評価: ②は箱を先に広げたが中身（安定稼働するスキル/ループ）がまだ育っていない。→ **当面 ② は休眠、
  ①＋③の2本立て**で簡素に回す（ユーザー判断）。

### 並行制御の原則（フォルダmv vs テキスト状態）
- 二重実行を防ぐ本質は「検知のしやすさ」ではなく「**claim（掴む）が原子的か**」。
- フォルダ mv は「見て奪う」が1手＝原子的ロック。テキスト状態は「見る→書く」の2手で、隙間に別ワーカーが
  割り込むと二重実行する（速さでは防げない＝同時に読めば両方 ready を見る）。
- ただし競合が起きるのは「複数ワーカーが同じ列から同時に取りに行く」時だけ。
  - **単一の割り振り役（指揮官）が各タスクを1つのworktreeへ配る → 競合しない＝テキスト状態で十分**
    （フォルダmv不要）。指揮官が実質のロック。
  - ワーカーが自分で同じ列から取りに行く（＝headless無人並行）→ フォルダmvが要る。
- near-term は前者（指揮官が配る）なので、デイリー/計画のテキストで状態遷移（ready→running→done 等）を
  書いてよい。将来 headless 並行へ戻す時に ② のフォルダmvロックを再有効化する。
- ガード: 機械が書くのはマーカー内側のみ・writer は1つ、人間はマーカー外。同時全書き込みはファイル破損しない
  （`set-marker-block.sh` の mktemp+mv 原子置換）が片方の変更を取りこぼす可能性は残る＝個人OS規模では許容し、
  冪等な再生成で吸収（`session-daily-log/README.md` の既結論と同じ）。

### near-term 運用方針（要指揮官確認）
- headless（無人並行AI実行）は当面止める＝§7 の人間ゲートを閉じたまま。ai-jobs は消さず**休眠**。
- ①ループ（自動・自己完結）＋ ③デイリー正本→指揮官が監督付き Orca へ1つずつ配る、の2本立て。
- 状態遷移はデイリー/計画のテキストで持ってよい（単一割り振り役のため）。

## 各テーマの課題感（実装前の再整理・会話4）

実装カードに落とす前に、各テーマの「本当の課題・未解決点・テストすべき点」を先に出す。これを計画（指揮官/
専用チャット）へ渡し、**課題感の合意 → その上で会話3の配布パッケージを再設計して着手**する（いきなり実装しない）。

### T1. 夜loop 出力フォーマットの読みやすさ（実物観察）
- コミット無しセッションは「repo(branch) / session=UUID」の空ポインタ＝人間には中身ゼロ（今日の実物で
  対話行の一部がこの状態）。設計/調査/判断系は成果がコミットに出ず"空セッション"に見える。
- `session=UUID` は人間可読情報ゼロのノイズ。ポインタとして要るが表出の価値は低い。
- **時系列ソートされていない**（auto:log の並び順のまま）。日次振り返りは時系列で読みたい。
- 「対話セッション」プレフィックス反復が冗長。「実行したこと」で人間 `- ` 行と `[auto]` 行が混在し境界が曖昧。
- 根本: コミット＝成果の代理は"コード作業"に効くが、思考系には効かない（＝T5に接続）。format単独では解けない。
- テスト/分析: 同分複数セッション競合の取りこぼし、当日0件、commit有無混在、repo消失時のsha表示。
  受け入れ条件＝「誰が・何分で・何を読み取れれば良いか」を先に決める。

### T2. 契約 §1/§7 の2モード化
- 「単一割り振り役ならテキスト状態でOK」を契約化する時、**headlessへ戻す切り替え条件**（＝同じ列を同時に
  取り合うワーカーが2以上になる瞬間）を明記しないと事故る。
- §7「入口と出口だけ人間ゲート」は「計画にレビュー項目あり＝自動実行GO」と誤読される破壊力（会話2の懸念）。
  中間の自動進行を、破壊的操作（push/main/削除/launchd登録）と非破壊とで線引きする必要。
- Stream1（分配設計）が同契約に依存 → 2モード化が分配前提を変えるので矛盾確認（指揮官マター）。
- テスト/分析: 「単一割り振り役＋テキスト状態」で1計画を実走し、二重実行なし・状態遷移が追えるを実証（program B）。

### T3. デイリー空マーカー（goal/carry/progress）
- keep/delete は format でなく**逆算の前提に依存**する判断。
- keep の壁: goal/progress は 3年/年間ゴールが正本だが**今は未確定だらけ**→自動化しても中身が出ない。carry
  （未完了TODO由来）は比較的意味を持つが、TODOの正本がデイリー自身＝循環に注意。
- delete の壁: 逆算ビューの設計意図が消える→消すなら「なぜ今は持たないか」を1行残す。
- 判断連動: **ゴールの埋まり具合**次第。空なら carry のみ実装・goal/progress は時期尚早、が妥当な仮説。
- テスト/分析: 先に 3年/年間ゴールの現在の埋まり具合を確認してから keep/delete を決める。

### T4. ドキュメント実態乖離
- 単発修正(Card1)で直しても**再発する**。稼働状態を単一真実源（launchctl/settings.json）から引く/定期照合する
  仕組みが無い＝構造的再発。今回は手修正でよいが「再発防止」を別課題として残す。

### T5. 思考・判断の記録（横断・冒頭の不安の核）
- コミットに残らない思考（今回の設計会話等）は、誰かが doc 化しない限り消える。transcript自然文要約
  （夜loop TODO）は誤要約リスクで未実装。
- 未設計: どの粒度で・誰が(人/AI)・どこに残すか。これが「どう管理してるのか定かじゃない」の正体。

### 配布パッケージ（会話3）の見直し方針
- T1 を**新カードに昇格**（format改善＝まず"読みやすさの受け入れ条件"を決める分析が先、実装は後）。
- Card3(marker triage) は T3 のとおり**ゴール埋まり具合の確認を前段**に付ける（でないと keep/delete を誤る）。
- Card2(契約draft) は T2 の切り替え条件・破壊操作の線引き・Stream1整合を盛り込む。
- Card1(doc乖離) は T4 の再発防止メモを add。
- 結論: 会話3の3カードは**そのまま着手せず、この課題感を織り込んで再設計**してから ready へ。
