## 方針

- テーブル: `todos`（id / title / note / do_date / due_date / repo / assignee(自分|AI) / status(open|done|dropped) / ai_status(未検知|検知|立案中|実行中|確認待ち|完了) / source(web|chat|cli) / goal_ref / created_at / updated_at / completed_at）＋ `repos` マスタ（repo-registryからの参照コピー・「なし（私用）」を含む）
- DB置き先（board / inbox / 統合）は実装冒頭に設計判断し、判断理由をこの計画の方針へ追記してからmigrationを書く（適用は人間承認後）
- 画面: 今日ボード=やること最上段（AI検知バッジ・確認待ち行に承認ボタン）＋既存データ区画（エージェント/終わったこと/サマリは既存クエリ・集計SQLを流用）。右下FAB→起票画面へ遷移（タイトル/いつ/実行repo/任せる=必須、締切・メモ・的紐付けは「詳しく▾」に格納）
- 画面更新はポーリング/再検証で実現（Tursoにrealtime pushが無いため）。間隔は実装時に決めてこの計画へ追記

### 実装時の設計判断（2026-07-17追記）

- **DB置き先: personal-os-inbox**。board dbはfocusmapから読み取り専用トークンしか持たず書けない。inboxは既存の`goals`追加フォームと同じくfocusmapがWeb書込トークンを持つため、`todos`/`repos`もここに新設した（`db/turso/migrations/20260717000000_todos_and_repos.sql`）。子02のAI検知ループ（Mac側）がai_statusを更新する際は、Mac側がKeychain経由でinbox書込権限を追加すれば足り、focusmap側の変更は不要という設計判断。boardとの「統合」は取らなかった（board db側のスキーマ変更は session-board 実装の正本を触ることになり、変更禁止範囲に抵触するため）
- **ポーリング間隔: 20秒**。sessions画面が日付ナビ+手動更新なのに対し、今日ボードはai_status変化（子02の検知結果）を待つ画面のため、isToday時のみ軽量クライアントコンポーネント（`_components/board-poller.tsx`）で`router.refresh()`を20秒間隔実行。Turso読み取りコストと体感速度のバランスで20秒を採用（他画面のfast-watch 1秒〜Target Refresh 60秒の間の中庸値）
- **画面配置**: `/dashboard/board`・`/dashboard/board/add` は `dashboard/workspace/`（business向けspace選択UIを持つ）ではなく `dashboard/layout.tsx` 直下の独立ページとした。既存 `src/components/today/*`（カレンダーTodo機能）と概念が異なるため命名衝突を避け、認証チェックは `ai-todos/page.tsx` と同じ自前パターンを踏襲。触ってよい範囲が `src/app/` に限定されているため、起票フォームのUIも新規Server Component＋ネイティブHTML（radio pill・details）だけで組み、既存 `src/components/` への追加・変更はしていない
- **「いつ」の暫定マッピング**: schemaが`do_date`単一カラムのみのため、今日=当日/明日=翌日/今週=今週日曜(JST)/来週=来週月曜(JST)/日付...=カスタム日付、として実装。週バケット表示の正式対応は子02（2週間/月ビュー）に委ねる
