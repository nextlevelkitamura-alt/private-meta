# 運用契約（Personal OS 不変の背骨）

Personal OS で作業する人間・AI・スクリプトの全員が従う恒久ルール。運用モデルの現在地は同フォルダ `README.md`。
改訂は人間承認＋決定ログ（`my-brain/areas/ai運用/決定ログ.md`）への追記とセットで行う。
2026-07-02 に計画フォルダ内の `共有コントラクト.md` から昇格（決定ログ #2）。以後この1枚が正本。

## 1. 状態の持ち方（2モード）

- モードB（テキスト状態・単一指揮官）＝既定: 単一の指揮官（人＋中央Claude、orca-cockpit）が直接配る間は、テキストで状態を持つ。writerが1人なので同時書き込み衝突が無く、フォルダロック不要。
- モードA（フォルダmv）: ai-jobs run-card のフォルダ位置が正本（`ready/running/review/reviewing/done/archive`）。無人・並行（複数AIが自分から同じ列を取りに来る）時だけ使う。現在は休眠（フォルダ・dispatcher実装・branchは温存）。
- モードAへの活性化条件（すべて満たしたら人間承認の上で昇格）:
  1. モードBで安定した定型処理が2本以上、同じ列を同時に取り合う（無人並行でないと詰まる）状態になった。
  2. ai-jobs-dispatcher のスモークテスト（実ワーカー1枚の実走）に合格した。
- 不変: どちらのモードでも状態は1箇所（§5）。B→A昇格時はテキスト状態をrun-cardへ移す（両方に持たない）。

## 2. 作業パイプライン（段階語彙）

- フル: 企画 → 計画 → 計画レビュー → 実装 → 実装レビュー →（修正⇄再レビュー）→ 人間確認 → 完了。
- ライト: 計画 → 実装 → 実装レビュー → 完了（人間確認は事後報告）。ライトでも plan.md は必須（最小行数でよい）。
- サクッと: 計画書なしの即実行（指揮官直または単発ペイン）→ 事後報告。判定は構造3条件のみ＝①変更1〜2ファイル ②容易に戻せる ③人間ゲート事項を含まない（時間の目安は使わない。2026-07-02裁定=決定ログ#5）。入口判定の手順書は基盤 `skills/plan-triage`。
- 規模は plan.md 冒頭の `規模:`（フル/ライト。省略時フル）で宣言する。
- 人間ゲートは入口（企画・計画レビュー）と出口（人間確認）の2点＋任意の途中確認。中間（実装⇄実装レビュー）はAIが自動で進めてよい。
- ただし破壊操作は段階・規模に関係なく常に人間ゲート: 削除・履歴改変・hook/launchd登録・`git push`・main反映・外部公開・**本番データの書き換え/削除・DB migrationの適用**（2026-07-03追記=決定ログ#8。アプリrepoでDB・本番が絡む場合の穴を塞ぐ）。
- cockpitペイン対応: 計画+監督ペイン=計画/計画レビュー、実装ペイン=実装/修正、レビューペイン=実装レビュー。
- 実行方式の語彙: ①キュー（Orca経由・見える）/ ②headless（定時・無人）/ ③hook（イベント）。迷ったら①、安定で②へ卒業（正本: 基盤 `loops-registry/references/loop-types.md`）。
- **規模別レビュー方式（レビューゲート・決定ログ#9）**: レビューの重さを変更規模に連動させる。「軽くする」のは範囲・回数・モデルで、異系統クロスチェック（実装=claude系 × レビュー=codex系）は各段で維持する。
  - サクッと: AIレビューなし。実装者の自己チェック＋人間へdiff要約を即報告（構造3条件が保険）。
  - ライト: AIレビュー1パス・差し戻し上限1。範囲=差分×完了条件の照合＋変更行のバグ・境界まで（触っていない周辺コードは掘らない）。モデル=codex gpt-5.5 high。目安=レビュー時間≦実装時間。
  - フル: AIレビュー（探索込み）・差し戻し上限2。モデル=codex gpt-5.5 xhigh。大きい・破壊的・複数ファイル横断ほどフルへ（迷ったらフル）。
  - 実行中の昇格: サクッと/ライトで始めた変更が構造3条件を外れたら（2ファイル超・戻しにくい・人間ゲート該当）、実装者は`_DONE`前に申告し指揮官が上位規模へ引き上げる。
  - モデル具体構成の正本はロースター/plan-triage。将来の非クリティカルレーンのレビューエンジン差し替え（kimi等）は親計画方針5で統計比較後。

## 3. ビューと記録（デイリー=1枚原則）

- 人間が見る・書くページは当日デイリー1枚（`my-brain/ゴール/デイリー/<年>/<月>/<日>.md`）。ダッシュボードを独立ページとして持たない。
- デイリーの役割は2つで、混ぜない:
  1. 人間の日次記入の正本: 今日のTODO・依頼インボックス・実績の手書き行（マーカー外）。ここはレンダリング元の無い第1情報。
  2. ビュー: `auto:*` マーカー内はレンダラだけが書く（冪等・全置換）。計画・実行状態の第2保管庫にしない。
- 記録の保証はpull: 事実の正本は各サーフェスのネイティブ記録（Claude transcript / Codex sessions / 各repoのgit履歴 / ai-jobs）。hookとcockpitイベントは引き金（速報）であって正本ではない。レンダラは何度走っても同じ結果になる（冪等）。
- 実装の無い `auto:*` マーカーをテンプレ・実ファイルに置かない。

## 4. スキルの正本

- Skill本文の正本は基盤 `skills/<skill>/SKILL.md`。registry/logs/plansに本文を複製しない。
- skill注入は名前参照（runtime symlink経由）を第一とし、非対応engineのみパスpointerでfallbackする。

## 5. 二重管理の禁止

- 同じ情報（状態・本文・スケジュール）を2箇所に持たない。索引/ビューは可、状態の複製は不可。

## 6. 既存資産（重複を作らない）

- `coding-task-orchestrator`（コーディング特化）と重複させない。汎用分配はコーディング作業をそこへ委譲する。
- 自動実行は ai-jobs レーン / loop-runbook の上に乗せる。schedule は基盤 loops-registry の loop entry を単一正本にし、新しい schedule registry を作らない。

## 7. セキュリティ / ゲート

- secret / `.env` / 認証値は出力・記載・コミットしない。
- 破壊的変更・hook/launchd登録・`git push`・main反映・削除は人間ゲート（§2の破壊操作と同一）。

## 8. 正本変更の追従

- 語彙・規約・構成の正本を変えたら、同じ作業単位で関連 `AGENTS.md` / README / テンプレ / 説明書 をgrepして追従し、決定ログに1件追記する。
