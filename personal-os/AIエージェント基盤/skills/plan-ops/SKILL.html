Skill説明書 · 手続き

plan-ops

計画の内容を決めずに、雛形生成・子計画マップ更新・静的検査・WIP上限付き昇格を、固定pathのscriptで安全に行う入口です。

要点workflowscript範囲人間ゲート置き場確認

最初に押さえる3点

計画の判断、機械手続き、レビュー判断を混ぜないことで、AIの動作を追える状態に保ちます。

判断は外へ委譲

置き場と規模はplan-triage、目的と内容は計画の所有者が決めます。

手続きだけを固定化

このSkillは6つの既存CLIを選んで呼び、本文の意味を解釈しません。

状態とゲートはplan本文に残す

親の 人間確認方針、子の 人間ゲート、子計画マップと各plan本文が正本で、第2の台帳は作りません。

判断から結果までの流れ

機械化する範囲を中央に閉じ、前後の判断責任を明確に分けます。

目的ごとにworkflowを選ぶ

コマンドを探す前に、やりたい手続きから読むファイルを一つに絞ります。

雛形と更新

workflows/scaffold-and-update.md
plan / program / 子計画を作る。既存NNのstate・next・refを更新する。

検証と昇格

workflows/validate-and-promote.md
programをlintし、レビュー項目の節を確認し、planningからactiveへ昇格する。

自動・手動の境界

references/script-map.md
6つのCLI、固定path、書込み条件、手動に残す判断を一望する。

6つのscriptは「事実」だけを自動化する

各CLIができることと、人が判断したまま残すことを並べて確認できます。

script自動化すること人が決めること
new-plan.shplan / program雛形の新規生成置き場、目的、完了条件
new-child.sh子雛形と親への相対backlinkマップへの新規NN追加、内容
progctl.sh既存NNのstate / next / refを冪等更新状態・次の一手の意味
program-lint.sh親子対応・backlink・状態語彙の静的検査内容とレビュー合否
check-section.sh指定節だけの表示・pattern照合一致の意味と合否
bucketctl.shplanning→activeのdry-run / 移動優先順位、退避、archive・卒業

書込みはdry-runと人間ゲートの後にだけ行う

移動・commitの自動実行は、計画の優先順位や安全性を代わりに決めるものではありません。

既定は安全側。 templatesは親の 人間確認方針: 最終一括 と子の 人間ゲート: を置きますが、scriptはその意味も承認有無も判断しません。progctl.sh と bucketctl.sh はまずdry-runで差分を見ます。bucketctl --apply / --commit は移動を伴うため、人間の明示承認が必要です。activeは原則3件で、2026-07-14に承認されたai運用の限定例外だけ、指定programがactiveの間は4件です。

実体は既存の置き場に残す

入口文書を分けても、CLI・テンプレ・テストのpathは変わりません。

入口と実装のずれをテストで防ぐ

CLIを変えた時だけでなく、説明を書き換えた時も実装との突き合わせを行います。

bash __tests__/run.sh

script本体の変更時は回帰テストを実行します。入口文書だけを変えた時も、workflow / referenceの説明とCLIの実装を突き合わせます。
