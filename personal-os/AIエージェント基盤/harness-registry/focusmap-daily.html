合意して実装した5点

この5点は設計案ではなく、Hook・Skill・追加DDLの現行契約です。本番Turso適用は完了し、Focusmap UI接続が次段階です。

Q1. 毎回の分類基準はSKILL.mdを強制読込するか
しない。 固定方針は各SessionStartイベントで session-classification-policy.md を注入し、resume/compact後にも復元する。毎promptは現在地と候補だけを短く渡し、Skillは未分類を再評価する明示ワークフローにした。
Q2. repo-registry外の新しいrepoもメニューへ出すか
推奨: 出す。 全トップレベルセッションから自動検出し、未登録 バッジを付ける。ただしrepo-registryへは自動登録しない。フルpathは通常表示せず、必要時だけマスクして見せる。
Q3. すぐ終わる計画外修正をThemeに入れるか
推奨: Themeに貢献するなら入れる。 ただし偽のPlanにはせず、Theme直下の「テーマ内作業」帯に置く。Theme進捗%は正式Plan工程だけで計算し、単発は別件数で表示する。
Q4. AIはどこまで自動確定してよいか
推奨: Planカードから起動、handoffにIDあり、同sessionで人間確定済み、ユーザーが明示、の時だけ自動紐付け。意味が似ているだけなら提案に留める。
Q5. 計画箱を新しく作るか
推奨: 作らない。 既存の「当日ボードSQL化」programへ、Global Skill・Hook/DB・UI・人間操作・統合評価の子10〜14として合流する。

Script・Hook・Markdown・Skillの使い分け

AIがScriptを読むのではありません。HookがScriptを実行し、Scriptが作った短いMarkdown本文をAIが読みます。

Hook

発火タイミング

UserPromptSubmit、SubagentStart、Stopなど「いつ実行するか」を決めるセンサー。

Python

事実と順番

ID・時刻・cwd・repoを記録し、DB取得と注入文の組立を決めた順に実行する。

Policy MD

開始イベントで戻す固定境界

既存Plan・Theme内作業・新候補・未分類の境界。SessionStartでPythonがplain textとして注入する。

Skill

人が明示する整理手順

未分類のまとめ直し、誤分類監査、Theme・Plan候補の再検討。必要な時だけ全文を読む。

同一イベントのHookを分けて順番に動かすことはできません。 Codex・Claudeとも一致する複数Hookは並列です。DB記録 → policy読込 → 候補取得 → context返却の順序は、1本の統括Scriptの中で保証します。
非推奨

SKILL.md本文だけを毎回注入

技術的には可能ですが、Skillの発動ではありません。frontmatter・明示呼出し・Skill内hook等が効かず、同じ文書に2つの実行経路が生まれます。

採用済み

PolicyとSkillを役割で分ける

開始時の固定基準は短いpolicy。毎promptは動的packet。人が明示して行う再分類だけ session-routing Skillが担当します。

イベント順と、1本のScript内部の順番

SessionStartは実行場所の事実を捕捉し、分類の主入口はUserPromptSubmitに固定します。

SessionStart
session/runtime/cwdを観測し、repo・worktree・branchをbest-effortで記録。分類はまだ行わず、その開始イベントで固定policyを返す。resume/compactでも方針を復元する。
UserPromptSubmit
1本の register-and-guide.py が直列に、①入力正規化 → ②session登録/復帰 → ③route-prepareでContext＋pendingを同一batch → ④今日＋対象repoの候補取得 → ⑤1つの短いContext作成、まで行う。
メインAI
注入された基準と候補を読み、既存Plan・Theme内作業・新Plan候補・新Theme候補・独立単発・判定待ちのどれかを構造化提案する。自然言語SQLは使わず専用CLIへ書き戻す。
SubagentStart / Stop
親sessionの分類を継承。agent_id・役割・開始/終了時刻・状態を機械記録する。subagentごとにDailyの独立sessionを増やさない。
Stop
turn終了とrun→waitだけを扱う。分類を決めない。完了確認は人間の節目で行い、単発Todoのチェック後にarchiveへ移す。
[FOCUSMAP ROUTING CONTEXT] turn: <turn-id> repo: focusmap / main current: theme=<id> / plan=<slug> 未所属時だけ、今日＋対象repoのTheme候補・参照Plan候補を最大3件ずつ追加。 required: 5分類をroute-proposeへ1回記録。 write-back: board.py route-propose ...
重要: AIへ渡すのは生プロンプト全文や長いPlan本文ではありません。判断に必要な短い候補だけに絞り、Hook出力の上限と対話ノイズを抑えます。

実装したSkillの役割は「再分類と監査」

毎promptの自動受付はHookが所有し、Skillは人間が「未分類を整理して」と頼んだ時だけ使います。CodexとClaudeは同じ正本をsymlink経由で読みます。

skills/session-routing/ ├─ SKILL.md … 57行の再分類手順・5分類・専用CLI └─ SKILL.html … 白背景の人間向け説明書 ~/.agents/skills/session-routing → 正本（Codex） ~/.claude/skills/session-routing → 正本（Claude）
発動関係: HookがSkillを呼ぶのではありません。Hookはpendingと判断材料を必ず用意し、通常のメインAIがその場で提案します。Skillは後から人間が明示して整理する別の入口です。

先に潰すべき失敗

順序崩れ

同一イベントの複数Hookへ分割

並列実行されるため、DB受付とcontext生成を同じ統括Script内で直列化する。所有者の違うHook同士は相互依存させない。

偽Skill発動

SKILL.md本文の注入をSkillと呼ぶ

plain context注入とSkill invocationを明確に分ける。自動処理はpolicy、明示整理はSkill。

repo混同

basename衝突・worktree重複

Git common-dir由来repo_keyとworktree root・実cwdを分離。未登録と非Gitを明示し、remote URLは保存しない。

進捗水増し

単発をPlan工程へ混ぜる

Theme内作業帯へ分離し、Plan進捗%と単発完了数を別指標にする。
