# Handoff Plan Supervisor

断片的な依頼、メモ、スクリーンショット、既存ドキュメントを、他チャットがそのまま実装に入れる引き継ぎパッケージへ変換する。親チャットは計画・プロンプト・戻り報告レビューを担当し、実装は必要に応じて別チャットへ渡す。

## 基本方針

- まず目的、成功条件、非目標、制約を固定する。
- 実装前に全体像、依存関係、検証方法、完了判定を明確にする。
- 写真、スクリーンショット、添付画像がある場合は、根拠資料として引き継ぎ書に入れる。
- 添付画像が見当たらない場合は、画像を作ったり推測したりせず「写真なし」と明記する。
- 別チャット用プロンプトは、必ずチャット本文に出力する。ファイル作成だけで終わらない。
- 別チャット用プロンプトの正本は `../plan-ops/templates/実行指示.md`。このSkillは独自テンプレ本文を持たず、Task Packetへ具体値を埋める。
- 長い背景、詳細仕様、画像一覧、調査ログはプロンプト本文に詰め込まず、引き継ぎ書や既存ドキュメントを読ませる形に逃がす。
- 戻り報告は鵜呑みにせず、目的・差分・検証・残リスクで評価する。

## 出力契約

- 初回応答では、最低限 `全体プラン`、`引き継ぎ書の要点`、`別チャット用プロンプト`、`監督ボード` をチャットに出す。
- 引き継ぎ書ファイルを作った場合でも、別チャット用プロンプトはチャット本文に掲載する。
- プロンプトはTask Packetの必要箇所だけを埋め、長い背景はplan/referencesへ逃がす。
- 説明は、何を作るかだけでなく「なぜやるか」「完成後に何が変わるか」「どこまでやらないか」を先に書く。
- 実装担当が最初に読むべき順番を明記する。例: `1. AGENTS.md 2. 引き継ぎ書 3. 対象ファイル`。

## ワークフロー

1. **入力整理**
   - ユーザーの依頼、対象リポジトリ、関連ファイル、添付画像、既存計画を確認する。
   - リポジトリ作業なら `AGENTS.md` / `CLAUDE.md` / `README` / `git status` / 進行中計画を確認する。
   - 不明点は、実装分解に影響するものだけ短く質問する。推測で進める場合は「仮定」に分ける。

2. **素材確認**
   - 会話で渡された画像、指定パス、`docs/**/assets/`、`output/`、スクリーンショット保存先など、依頼に関係する範囲だけ確認する。
   - ローカル画像を本文に表示する場合は、絶対パスの Markdown 画像タグを使う。
   - 引き継ぎ書には、画像ごとに「何の参考か」「実装で見るべき点」を1行で添える。

3. **全体プラン作成**
   - ゴール、成功条件、非目標、前提、フェーズ、タスク、依存関係、リスク、検証、完了条件を整理する。
   - タスクは `調査 -> 実装 -> 検証 -> 統合 -> 完了判定` の順で追える形にする。
   - 大きい実装では、並列化できる範囲と直列にすべき範囲を分ける。
   - 冒頭に、目的と実装の全体像を5文以内で説明する。

4. **引き継ぎ書作成**
   - 他チャット、人間、後日の自分が読んでも現在地が分かる Markdown として作る。
   - 画像、関連リンク、対象ファイル、禁止範囲、検証コマンド、完了報告フォーマットを含める。
   - ファイルを作る場合は、対象プロジェクトのドキュメント規約に従う。規約がなければ `docs/ai/plans/active/` かユーザー指定先を使う。

5. **別チャット用プロンプト作成**
   - `../plan-ops/templates/実行指示.md` を参照し、目的、非対象、読む順番、対象repo、base、変更可能/禁止範囲、依存成果、実装内容、検証、停止条件、result packetを具体化する。
   - 作業場所とbranchはハーネス割当済みであり、workerが作成・切替しないことを明記する。push、merge、deployは対象repo規約に従い、明示依頼なしには行わない。
   - 実装範囲が大きい場合は、調査用、実装用、レビュー用、統合用にTask Packetを分ける。
   - 長くなる場合は、詳細を `参照ドキュメント` に置き、Packetには読む順番と要点だけを書く。

6. **監督・戻りレビュー**
   - worker の戻り報告を受けたら、差分、commit、検証ログ、未解決リスク、目的達成度を確認する。
   - 不足がある場合は、追加確認または再開プロンプトを作る。
   - 完了可否は `local main`、`origin/main`、production、検証、ドキュメント更新を分けて判定する。

## プロンプト圧縮ルール

プロンプトが長くなる場合は、次の順に削る。

1. 詳細背景を引き継ぎ書への参照に変える。
2. ファイルごとの細かい説明を「対象ファイル一覧 + 期待する変更」に圧縮する。
3. 画像説明は「参照画像パス + 見るべき点」だけ残す。
4. 調査ログ、検討した代替案、会話経緯はプロンプトから外し、引き継ぎ書に置く。

プロンプトに残す必須情報:

- 目的
- 完成後のユーザー体験または運用変化
- 最初に読むドキュメント
- 触ってよい範囲 / 触らない範囲
- 実装タスク
- 受け入れ条件
- 検証
- 完了報告形式

## 自己評価チェック

出力前に次を確認し、不足があれば直してから出す。

- 目的が一文で分かるか。
- 実装の全体像が、背景を知らない実装者にも分かるか。
- プロンプトが長すぎず、詳細は参照ドキュメントへ逃がせているか。
- 別チャット用プロンプトがチャット本文に実際に出ているか。
- 写真や添付素材の有無を明記しているか。
- 完了報告を見れば、親チャットが評価できる形になっているか。

## Task Packet記入例

`実行指示.md` の該当欄だけを次のように埋める。本文の構成・規律・役割別追加指示は必ず正本テンプレをそのまま参照する。

```text
Task ID: plan-ops-01
役割: implementer
対象repo: /repo
対象計画: /repo/plans/active/example/plan.md
base commit: abc1234
目的: 実行契約を満たすplan-lintを追加する。
読む順番: 対象repoの最寄りAGENTS.md → 親program → 対象計画 → references → 対象ファイル
変更可能: skills/plan-ops/scripts/ と __tests__/
変更禁止: runtime設定、hook登録、symlink露出
依存成果: なし
検証: bash skills/plan-ops/__tests__/run.sh
停止条件: 既存program-lint fixtureの互換を保てない時
完了時: 実行結果.jsonのresult packetを返す
```

## 戻り報告レビュー

判定は「合格 / 条件付き合格 / 差し戻し」、根拠は目的達成度・確認した証拠・変更path・検証結果・問題・未解決リスク・次の指示で返す。再開時も独自書式を作らず、`実行指示.md` のTask Packetへ前回結果と今回の修正範囲を埋める。

## 副作用レベル

- L1/L2。基本は計画、引き継ぎ書、プロンプト、レビューを作る。
- 実装、commit、merge、push、deploy はユーザー指示と対象リポジトリ規約に従う。
- 破壊的操作、secret 表示、production 変更、無断 deploy はしない。
