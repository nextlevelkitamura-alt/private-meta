# 03 Focusmap自動処理管理UI

## 目的

manifestとverifyが返す状態をFocusmapへ集約し、hook / loop / serviceを「現役か」「本当に動いているか」「最後に成功したか」で確認できるread-only画面を作る。

## 現状

- FocusmapのAutomation設定画面はMac agent接続状態が中心で、Global hook / loop一覧ではない。
- Web UIにはTurso snapshot / heartbeat / activityを読む資産がある。
- runner statusやtask progressのpoll間隔は複数hook / componentに分散し、3秒 / 5秒 / 30秒 / 45秒 / 90秒窓の記述ドリフトがある。
- Global verifyはJSON snapshotをまだ出さず、hook verifyも存在しない。

## 方針

### 情報設計

- AI運用の配下に既存「今日」と新規「自動処理」を置く。
- 自動処理画面は次の4ビューを持つ。
  1. 全体: 現役 / 異常 / 停止 / legacyの件数。
  2. 自動処理: unitごとの意図状態、実機状態、trigger、owner、last success / failure。
  3. 異常: stale、意図と実機の不一致、entrypoint欠落、mirror parity不一致。
  4. 接続: Mac agent、Codex app-server、Turso、runtime露出。

### データ経路

- 子07の状態snapshot APIだけを読む。publisher、認可、保持、retryは子07の責務で、本子計画へ重複実装しない。
- snapshot失敗時は前回値＋stale表示。空配列で正常に見せない。
- UIはsnapshotのconsumerであり、manifest・plist・settingsを直接読んだり編集しない。
- pollは共通hook / queryへ寄せ、画面ごとの独自周期を増やさない。前面表示・異常あり・通常idleで段階化する。

### 初期スコープ外

- 任意shell、launchctl、runtime settings編集。
- unitの停止 / 再開 / 再実行。
- raw log全文やsecret pointerの具体値表示。
- MD正本の置換。

## 完了条件（レビュー項目）

- [ ] Global / Focusmap unitが同じ表示schemaで読み込まれ、scopeとownerを区別できる。
- [ ] 意図状態・実機状態・最終成功・最終失敗・staleが別フィールド / 別表示で確認できる。
- [ ] loaded想定だがprocess無し、旧labelだけloaded、entrypoint欠落を異常として表示できる。
- [ ] snapshot失敗時に前回値＋staleとなり、空配列やオンライン表示へ変換されない。
- [ ] raw log全文、token、credential、環境変数値がAPI payload / UIへ出ない。
- [ ] 絶対path、host名、PID、runtime登録path、log / state pathがAPI payload / UIへ出ない。
- [ ] Webから任意command、launchctl、settings / plist変更を実行する経路がない。
- [ ] poll周期が共通化され、同じrunner状態を複数箇所から重複取得していない。
- [ ] feature flag OFFで既存sessions / automation settingsに回帰がない。
