# programと子計画を管理する

## 起動条件・入力・期待出力

- 起動条件: 「子計画を追加したい」「programにまとめたい」「子の状態・依存を更新したい」。
- 入力: 親programの絶対path、既存子の一覧、新しい子の目的、依存・並列可否・レビュー方式、対象repoと計画箱の解決結果。
- 期待出力: 子計画本文、親programの子計画マップ、lint結果、次に実装またはレビューする子。

## 手順

1. `plan-registry/AGENTS.md` でprogram化の基準を確認する。独立に卒業する子が2本以上でなければ、`create-or-join.md` の単発planへ戻す。
2. 親programの子計画マップと既存子を読んで、NN、責務、依存、並列可否、レビュー方式、人間ゲートを人間・指揮官が決める。親には `人間確認方針: 最終一括（危険操作は実行前に個別承認）` を書く。通常子は `人間ゲート: なし`、危険操作を実行する子だけは操作を特定する。新しいNNブロックの追加は手動で行い、子の雛形は `plan-ops` の `scripts/new-child.sh` を使う。
3. 子の目的、現状、方針、完了条件を埋める。子の状態は子専用バケットでなく親マップにだけ持つ。
4. 既存NNの状態、次の一手、参照repo@hashだけを変える時は、まず `plan-ops` の `scripts/progctl.sh` のdry-runで差分を確認する。書込みと定型commitを一体で許可された場合だけ `--commit` を使う。
5. `program-lint.sh` で親子対応、backlink、状態語彙、完了条件を検査する。違反は内容を推測して直さず、該当子の所有者へ戻す。

## 人間ゲート・失敗時・戻り先・完了確認

- 人間ゲート: 親programまたは子の移動・削除・卒業・runtime変更は明示承認なしに行わない。
- 失敗時: NN競合、親不在、lint違反、依存の矛盾はマップを推測で更新せず停止する。
- 戻り先: 新規/既存planの経路解決は `create-or-join.md`、評価・状態遷移は `review-and-transition.md`、実装監督は `cockpit-supervisor`。
- 完了確認: 子のbacklink、親マップの依存・次の一手・人間ゲート、`program-lint` の違反なしを確認する。通常子の人間確認を親programの最終確認へ二重に積んでいないことも確認する。
