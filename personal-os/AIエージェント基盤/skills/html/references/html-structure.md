# html-structure（共通の土台）

`html` Skillの全モード共通の規約。まずここに従い、次に選んだモード（`mode-quick.md` / `mode-full.md`）に従う。モードの選び方は `SKILL.md` §4。部品のCSS・雛形の実体は `assets/artifact-template.html`。

## 1. Artifact制約（守らないと公開が壊れる）

1. 外部リソース禁止(CSP): CSS/JSはインライン、画像は data URI、フォントは system。
2. バックエンド無し: フォーム保存・view時API・複数ページは不可。表示専用と割り切る。
3. 単一ページ: 相対リンクは解決しない。ページ内アンカーを使う。
4. サイズ: 16MiB以下。大きな画像を埋め込まない。
5. ファイル形式: `.html` / `.htm` / `.md`。

## 2. 見やすさの基本

1. 文章は結論を先に出す。
2. 内容は部品に置き換える。「詳細＝表」ではない。詳細の既定の受け皿は `.deflist`（スタック定義ブロック）。
3. 色は意味に割り当てる（推奨=緑、注意=赤・琥珀、強調は1色だけ）。状態は色だけでなく文言・pillでも示す。
4. 1枚で完結させ、節が5つ以上なら冒頭にアンカーチップを置く。

## 3. 部品の選択ラダー（毎回、上から順に判定して最初に当たった部品を使う）

1. 同じ型の短い事実が行×列に揃い、§4の5条件を全部満たす → 表。
2. キーと値が1対1で短く揃う（スペック・設定値・メタデータ） → `.kv`。
3. 順序・時系列に意味がある → `.steps`。分岐・状態遷移があるなら SVG フロー図。
4. 階層・包含関係がある（フォルダ・構造） → `.tree`。
5. ちょうど2案の対比（Before/After・案A/案B） → `.compare2`。3案以上は `.cards`。
6. どれでもない＝説明が主役（理由・背景・条件が混ざる項目の列挙） → `.deflist`。3件以下で各1〜2行なら `.cards` でもよい。
7. いつでも併用: 本筋でない補足・ログ・長い根拠 → `details`（折りたたみ・初期は閉じる・結論は隠さない）／リスク・前提・例外 → `.callout`（1節1個まで）／関係の図解が要る → SVG。

## 4. 表の使用条件（1つでも外れたら `.deflist` へフォールバック）

1. セルは名詞句1つ。目安25字以内。文章・セル内改行を書かない。
2. セルに接続詞（「ため」「しかし」「場合は」等）が出たら、それは表でなく `.deflist` の内容。
3. 行数は8行まで。超えるならグルーピングして分割するか `.deflist` 化。
4. 列数は4列まで。「項目/判定/理由」の3列型を機械的に複製しない（「理由」列は大抵条件2に違反する）。
5. 表は1ページに2箇所まで。3箇所目が必要になったら設計を疑う。

## 5. 節の型（ページ後半でも崩さないフラクタル構造）

1. どの節も「h2見出し → リード1行 → 主役部品1つ → 補足は `details`」の順にする。
2. 見出しは「詳細」でなく結論・内容が分かる文言にする。見出しとリードだけ拾い読みして論旨が追える状態を保つ。
3. リード1行は結論から書く。「〜について説明します」のような前置きを書かない。
4. 主役部品は1節に1つ。部品の重ね置き（表＋deflist併置など）をしない。
5. 同じ主役部品を3節連続で使わない。特に表の直後の節に表を置かない（視覚リズム）。
6. 節が2画面分を超えそうなら、節を分割するか後半を `details` へ落とす。

## 6. 書き方（スキャンで読める文）

1. 箇条書き・カード・セルは先頭に太字キーワードを置き、太字だけ拾い読みできる形にする。
2. 本文の行長は日本語で40字前後に抑える（コンテナの max-width で制御。テンプレ既定で担保）。
3. 番号付き見出し（01/02…）は本当の手順・順序にだけ使う。飾りにしない。

## 7. トークンの土台

1. 図はラスター画像でなく SVG か HTML+CSS で描く（軽い・self-contained）。
2. JSは書かない。開閉はネイティブ `details/summary` で足りる。タブ・カルーセルは使わない。
3. 大きなデータは全量埋め込みでなく、要約して載せる。

## 8. 見た目の一貫性

1. 色・フォント・余白・部品のトークンは `assets/artifact-template.html` を基準にする。
2. 揃えたい場合は `CLAUDE.md` にデザイントークンを置く（Anthropic公式のビルトイン design skill も同じ場所を見る）。
3. 配色はライト単色に意図的にコミットする。dark分岐（`prefers-color-scheme`・`data-theme`）は書かない。quick/fullどちらのモードでも同じ。

## 9. いつHTML / いつMarkdown

1. HTML（このSkill）: 見て終わり / 共有して終わりの成果物。
2. Markdownのまま: 正本、継続編集するもの、git diff を見るもの、Notion同期対象（デイリー・計画）、短い返信。
3. 迷ったら: AIが後で読み書き・同期するなら md、人間が一度見るだけなら html。
