Plan Registry / Human Guide

計画管理は、ここから読む

Plan Registryは「規約と責務地図」の入口です。個別の計画、状態、履歴、session-boardを集める場所ではありません。置き場そのものは、対象repoまたはareaが持ち続けます。

規模とレビューprogram最終一括確認危険操作の個別承認責務地図手動 / script / hook計画本文は移動しない
最初の基準: Plan Registryがサクッと／ライト／フルの基準を定義し、plan-triageが依頼へ適用します。plan-create-reviewは解決済みの基準とrouteを使って、作成・合流・program管理・評価・終了（archive）を案内します。3条件がすべてYESの時だけサクッと実行できます。

誰が何を持つか

Plan Registry

規模・段階・レビューの基準と責務地図を定義します。個別の計画本文、状態、履歴は持ちません。

plan-triage

基準を依頼へ適用し、対象repo・既存plan・宣言済み箱の書込みなしrouteを返します。

plan-create-review

利用者の目的を作成・合流・program管理・評価・終了（archive）へ振り分け、triageとopsへ委譲します。

人間 / 指揮官

人間ゲートと、既存planへ合流するか新規起案するかの最終判断を担います。

対象repo / area

計画箱、物理バケット、テンプレ、実際の計画本文と状態を持ちます。

plan-ops

確定した入力だけを雛形、子計画マップ更新、静的lintの手続きとして処理します。

hook / session-board

実行記録と通知だけを担い、計画本文・状態・repo・計画箱・合否を決めません。

依頼から完了までの境界

規模と人間ゲート

規模進め方レビュー
サクッと3条件が全YESの時だけ、計画書なしで実装実装者の自己確認
ライト計画 → 実装 → 実装レビュー → 事後報告1パス、差し戻しは最大1回
フル単発planは段階的に最終人間確認へ。programは通常子をAIレビューで閉じ、親の統合評価後に一度だけ人間確認へ進める探索を含む。差し戻しは最大2回

programの通常子

実装 → 独立レビュー → 必要な修正 → 再レビューが全PASSなら、完了（AIレビュー済み・全体人間確認待ち） として閉じます。子ごとに人間確認は待ちません。

親programの最終確認

全子の完了条件と統合評価がPASSになってから、人間が全体を一度確認します。親には 人間確認方針: 最終一括 を明記します。

削除・移動・改名・履歴改変・hook/launchd登録・push・main反映・外部公開・本番データ変更・DB migrationは、規模を問わず実行前に人間の明示承認が必要です。これは親の最終確認へ繰り延べません。

手動と自動の分け方

人間は二種類の判断をする

危険操作は該当する子で実行前に承認します。通常のprogram子はAIレビューで閉じ、全子を統合した最後にだけ人間が全体を確認します。

scriptは確定入力だけを処理する

plan-ops は雛形、子計画マップ更新、静的lintを担います。規模基準、置き場、優先度、レビュー合否は決めません。

hook / session-boardは記録・通知だけ

開始・入力・終了の記録と通知だけです。計画本文・状態を持たず、repoや計画箱、レビュー合否を決めません。

runtime露出は承認制

plan-create-review の正本は基盤配下で、5runtimeへsymlink露出済み（2026-07-16人間承認）。以後の露出変更も人間承認ゲートです。
