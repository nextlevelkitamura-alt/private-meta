# 計画の回転と期分割運用

起案: 2026-07-21（AI討論3案×反対尋問の結論＋人間の追加アイデア2件を統合）。図解正本は `explain/plan.html`。

## 目的

「ひとつのprogramが終わらないと次の計画ができない」詰まりを、階層の追加ではなく**回転の改善**で解消する。具体的には次の5点を1企画として束ねる。

1. doneハード上限（8）の撤廃と、滞留可視化・ソフト閾値警告への置換
2. activeの無活動検知と paused 降格の**提案化**（自動移動はしない。検知→提案→人間ワンタップ→bucketctl）
3. 期分割規約: 6〜7子を超える構想は program を継ぎ足さず、後継 program（第N期）へ割る
4. テーマ（大課題テーマ）とプログラムの境界の明文化（参照で束ねる傘 vs 所有して束ねる実行単位）
5. 週次クローズ儀式（承認セット方式での done 一括クローズ＋保留承認操作の実行）の定着

## 非対象

- program の上への md 新層の追加（2026-07-21 AI討論で不採用。テーマとの二重傘・出口の詰まりに作用しないため）

- 進行中「2026-07-19-計画とカスタムエージェント整理」の Lane A 対象パス（plan-registry/**・計画系3 Skill）への先行変更
- バケット移動の完全自動化（本企画は検知・提案まで。移動は人間タップ＋bucketctl 検証のまま）
- focusmap 側 UI（バッジ・ワンタップ）の本実装（本企画は仕様定義まで。実装は子07/09 クローズ後の follow-up 計画）

## 方針

### 1. done上限の撤廃（ハード→ソフト）

- done=8 のハード上限を撤廃し、bucketctl の done 移動ブロックを警告に変える。
- 根拠: 上限の意図は「満杯にして人間に閉じさせる背圧」だが、ソロ＋AI運用では AI が承認を代行できないため、背圧は AI の実行だけを止めて人間を速くしない（三重満杯・07-15滞留の実測）。
- 代替の圧力（セット導入が採用条件）:
  - (a) done 各計画の滞留日数を可視化（デイリー/ボードに「done N件・最古M日」）
  - (b) done 件数がソフト閾値（初期値10・人間裁定）を超えたらデイリーで警告
  - (c) 週次クローズ儀式（方針5）
- done→archive の「人間確認＋終了記録」ゲートは不変（閉じる質は落とさない）。active=5・paused=3 は本企画では変更しない。

### 2. activeの無活動検知（自動移動はしない）

- 判定を2種に分ける:
  - **承認待ちstale**: 人間ゲート（migration・push等）で N日停止 → paused に落とさず、承認セットへ積み「承認待ちN日」バッジで督促する。視界から消すと承認がさらに遅れるため降格対象にしない。
  - **無活動stale**: 実行セッションも評価も計画フォルダへのコミットも M営業日ない → paused 降格を**提案**。理由・再開条件を自動起草し、人間のワンタップ承認後に bucketctl が検証つきで移動する。
- 閾値: 発案は「1日」だが、週末・ゲート待ちの誤検知を避けるため初期値 M=2〜3営業日 を推奨（人間裁定で確定。設定値は1箇所で持つ）。
- 自動移動をしない理由（規約整合）: (a) hook は計画を作成・移動しない／満杯時も自動退避しない既存原則 (b) paused は理由と再開条件の記録が必須で無人では品質担保できない (c) paused 上限3が自動降格で即溢れる。
- データ源: session-board DB（sessions.todo_id/theme_id）＋ git log。検知の実行形（loop/launchd か デイリー朝会内の1手順か）は loop-creator ゲートで決める。

### 3. 期分割規約（継ぎ足さず、割る）

- plan-registry §2 へ追記: 「同一目的で6〜7子を超える構想は、OS例外（10前後）を常用せず、後継 program（第N期）へ割る。期同士は superseded/merged の終了記録＋後継リンクで接続する」。
- 進行中 program の期分割は、静かな縮小ではなく人間ゲート付きの計画大幅更新（rename＋explain 再提示→合意→bucketctl）として行う手順を同節に明記する。完了条件の分母を黙って削らない。
- program lint に「子8本以上で警告（10超はエラーでなく要理由注記）」を追加する。

### 4. テーマとプログラムの境界の明文化

- areas/AGENTS.md へ1節: **テーマ＝DB正本・参照で束ねる長寿命の傘**（計画を所有しない・評価/Wave/人間ゲートを持たない・ボードで人間が編集）、**プログラム＝md正本・所有して束ねる有期の実行単位**（子計画を物理所有・Wave/評価/完了ゲートを持つ・gitで編集）。
- テーマ→計画は plan_refs の一方向参照のみ・本文コピー禁止・進捗の重複描画禁止（子09の裁定を規約語彙へ昇格）。「次の次」の構想アイデアの既定の置き場は「テーマの plan_refs へ登録＋必要になったら planning 起案」とし、既存 program への合流は実装中かつ密結合の時に限る。

### 5. 週次クローズ儀式

- 週1回、承認セット方式（07-15 program が規定済みの一括判断文書）で done 一括クローズ＋保留承認操作の実行をルーティン化する。
- 所有先（デイリー儀式に組み込むか、独立 Routine にするか）は人間裁定。判断の質は落とさず、判断のタイミングだけを束ねる。

### 6. UI追従の仕様定義（実装は後続）

- doneバッジ・承認待ちバッジ・paused 提案ワンタップの表示仕様を本企画で定義し、実装は「当日ボードSQL化」子07/09 クローズ後の follow-up 計画（focusmap側）として起案する。md/規約/script を先行し、UI を待たない。
