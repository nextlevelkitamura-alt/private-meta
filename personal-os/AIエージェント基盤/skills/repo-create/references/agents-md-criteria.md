# AGENTS.md criteria

`AGENTS.md` は、人間とAIが最初に読む入口。人間が分からない構成を、AIにだけ分からせようとしない。

## 共通構成

1. `# repo名`
2. 冒頭1〜2文: このrepoが何をする場所か。
3. `## 1. このrepoでやること`
4. `## 2. 絶対ルール`
5. `## 3. フォルダ概要`
6. `## 4. 作業の始め方`
7. `## 5. 更新ルール`
8. `## 6. 確認方法`

不要なら見出しを減らしてよい。大事なのは、目的、禁止、フォルダ、開始手順、更新手順が分かること。

## 書き方

1. 端的に書く。
2. 1項目に1判断だけ書く。
3. 抽象語だけで終わらせない。
4. 「どこに置く」「何をしない」「何を確認する」まで書く。
5. 長い手順、過去ログ、diff全文、作業メモは置かない。
6. `CLAUDE.md` は同階層の `AGENTS.md` への相対symlinkにする。

## 業務運用repo

Personal OS、docs、registry、automation、計画管理のrepo。

1. 正本、計画、履歴、現在状態が分かれている。
2. フォルダ概要が上の方にある。
3. 変更時に `AGENTS.md`、plans、logs、catalog、registryの整合性を見る。
4. 外部送信、削除、secret、定期実行、本番データ変更は人間承認にする。
5. 実行済み事実はlogs、これからやる判断はplansへ分ける。

## coding repo

アプリ、ライブラリ、CLI、実装作業のrepo。

1. install、dev、test、lint、buildの確認方法が分かる。
2. 触ってよい場所、作業前に読む場所が分かる。
3. deploy、migration、production data、push to `main` は人間承認にする。
4. 既存実装パターンを読んでから変更する。
5. UIやruntime変更では、確認方法を明記する。

## 評価

1. 初見でrepoの目的が分かるか。
2. 危険操作が止まるか。
3. フォルダ概要が分かるか。
4. 作業の始め方が分かるか。
5. 更新時に何を直すべきか分かるか。
6. 人間が読んで分かりづらくないか。
