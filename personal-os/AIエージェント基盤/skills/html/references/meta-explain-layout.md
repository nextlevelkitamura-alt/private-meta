# メタ説明の型

メタ構造を人間が最短で理解できるHTMLの構成と見せ方。
汎用のHTML規約は `references/html-structure.md` が正本で、ここにはメタ説明に固有の型だけを書く。

## 1. 説明の原則

1. **全体 → 差分 → 本文の順**。いきなり中身を見せず、まず「どこの話か」を地図で示す。
2. **現状を先に**。変更案は、現状が分かってから出す。
3. **実ファイル根拠**。「今こう動いている」はpathと本文を根拠にし、確認できなかった箇所は「推測」と書く。
4. **本文は全文**。新規・変更ファイルのドラフトは省略しない。大きい場合は差分と要点を主表示にし、全文はファイル単位で折りたたむ。
5. **人間の言葉**。コードネーム・略語・独自番号は初出で定義する。
6. **1節1メッセージ**。見出しとリードだけ拾い読みして論旨を追える状態にする。
7. **分からせる責任はAI側**。理解されなければ、図 → ウォークスルー → 具体例の順で見せ方を変える。

## 2. 固定7節型

変更前合意モードは7節フル。説明のみモードは③④⑤を省き、②を厚くする。

1. **① 結論**: 何の話か・何をどう変えるかを1〜2文で先出しする。
2. **② 現状**: 構成図・関係図・「今こう動いている」。
3. **③ 変更計画**: before / afterの対比と変更理由。
4. **④ ミクロ計画**: 新規・変更ファイルごとの本文ドラフト全文。
5. **⑤ 影響とリスク**: 触るもの・触らないもの・壊れうるもの。
6. **⑥ 未決事項**: 人間に決めてほしいことを質問形式で列挙し、各質問に推奨案を付ける。ゼロなら「未決なし・OK待ち」と書く。
7. **⑦ 更新履歴**: v1, v2…で「指摘 → どう直したか」を積む。

## 3. 見せ方マッピング（図の実体は `diagram-recipes.md` の対応表を正とする）

1. フォルダ構成・正本とruntime露出 → ボックス型ツリー図（R1）。正本とsymlink露出先を塗り／破線で区別する。
2. hook・loopの発火と流れ → 横フロー図（R2）。人間ゲート・承認点を強調する。
3. 設定・登録・構造の差分 → before / after対比図（R5）。文言・条件だけの差は `.compare2`。
4. ファイル本文 → codeblock全文。
5. Skill・registry・runtimeの関係 → 関係図（R3）。層の話ならレイヤー図（R4）。
6. 手順・運用フロー → stepsと実文のウォークスルー。経緯・時系列はタイムライン（R6）。
7. 選択肢 → 推奨案を先頭に置き、「どれでもいい」にしない。

## 4. 理解ループ

1. 毎版、未決事項を質問形式で維持する。回答済みも「回答: …」を追記して残す。
2. 指摘を受けたら、更新履歴へvNを追記し、該当節を直して同じURLで再公開する。
3. 同じ箇所に2回指摘が来たら見せ方を変える。
4. 未決事項が全て回答済みで、人間が「理解した・これでいい」と明示した時だけ合意とする。
