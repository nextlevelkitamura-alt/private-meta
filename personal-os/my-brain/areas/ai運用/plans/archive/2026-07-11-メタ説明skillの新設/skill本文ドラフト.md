# meta-explain

メタ領域を、人間が理解しないまま進めないための理解ゲート。
「現状こうなっている（→ こう変える → 新しい本文はこれ）」を1枚のHTMLにまとめ、
人間が「理解した・これでいい」と言うまで同じHTML（同一URL）を更新し続ける。実装はしない。

## 1. 手順

1. **対象とモードを特定する**: 説明のみ（現状理解）か、変更前合意か。何の話かを1〜3行で復唱し、曖昧なら短く確認する。対象の正本は 最寄りの `AGENTS.md` → `personal-os/AIエージェント基盤/AGENTS.md` → 該当registry の順で特定する。推測で正本を決めない。
2. **現状を調査する（読み取りのみ）**: 正本・導線・runtime露出・関連hook / loop / Skillを実ファイルで確認し、「今こう動いている」をパスと本文を根拠に整理する。確認できなかった箇所は「推測」と明記する。
3. **説明HTMLを作る**: 構成・見せ方は `references/説明の型.md` に従う。出力は `html` スキル（mode-full）の規約に従う。
4. **理解ループを回す**: HTMLのURLを表題つきで出す → 指摘ごとに同一URLを更新し、更新履歴にvNを積む → 「理解した・これでいい」まで回数無制限。未決事項への回答が全部そろってから合意とみなす。
5. **合意を正本へつなぐ（変更前合意モードのみ）**: 合意内容を計画正本（plan.md / program.md 等）へ反映することを提案し、人間の承認後に書く。実装は実装レーン（別セッション）へ渡す。

## 2. 安全方針

1. 手順1〜4は読み取りとArtifact公開のみ。対象ファイルを編集しない。
2. 手順5の計画反映は人間の承認後だけ。実装・削除・移動・改名・symlink変更はどの段階でもしない。
3. secret・token・credential・認証値をHTMLに載せない。

## 3. 関連

1. 説明の構成・見せ方（このSkillの中核）: `references/説明の型.md`。
2. 出力規約: `html` スキル（mode-full / html-structure）。規約本文をコピーしない。
3. 近接: `naiyou-suriawase`（軽い事前すり合わせ）/ `grill-me`（問いで詰める壁打ち）/ `plan-triage`（入口判断）。
4. 窓口: `skill-creator-custom`。
