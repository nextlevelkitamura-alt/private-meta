# safety-rules — 副作用とガード

`/sns-post` スキルが行う副作用と、それぞれのガード手順。

## 副作用一覧

| 操作 | 影響範囲 | リスク |
|---|---|---|
| スプシ書き込み（追記・更新） | 該当リポのスプシ | 行ずれ・誤上書き |
| Buffer API 投稿予約 | 外部公開（フォロワー全員） | 誤投稿 |
| Buffer 既存予約の本文編集 | 同上 | 誤編集 |
| 画像生成（Gemini） + Drive アップロード | Drive 容量・公開 | 容量浪費・公開漏れ |
| Threads API 数値取得 | 読み取り専用 | 影響なし |

## 7装置チェック

### 1. ~~disable-model-invocation: true~~ → 解除済み（2026-05-04）
スマホからの利便性のためモデル自動起動を許可。代わりに以下を厳守：
- 起動後は必ず Step 0/1/2 の AskUserQuestion を経由（自動実行しない）
- 副作用操作（投稿・スプシ更新・予約編集）は装置 #2〜#7 で多層ガード

### 2. submit-guard（送信直前確認）
Buffer 投稿（モード4）の前に必ず dry-run → ユーザー確認 → 実行。

```bash
# Step 1: dry-run
npx tsx src/publish.ts --dry-run

# Step 2: ユーザーに対象N件を提示してOKをもらう

# Step 3: 実行（--yes は必須）
npx tsx src/publish.ts --post --yes
```

### 3. --dry-run オプション
publish.ts / edit-post.ts に `--dry-run` を必ず実装。送信内容のみ表示。

### 4. 送信履歴追跡（二重送信防止）
スプシの A列（投稿予約）・B列（予約元）で状態管理：
- B列="Buffer" の行は publish.ts で自動スキップ
- 同一 dueAt + 同一 channelId への投稿はBuffer側でも重複チェック

### 5. ポストアクションセクション（各 mode-*.md 末尾）
モード完了後にユーザーに次の選択肢を提示：
- 投稿作成完了 → 「Buffer予約しますか？」
- Buffer予約完了 → 「次の投稿準備しますか？/ 一覧確認しますか？」

### 6. ユーザー確認文面（対象・内容・影響の3つ明記）
送信系操作前のメッセージは必ず以下を含む：
- **対象**: どのアカウント・何件
- **内容**: 投稿日時・本文プレビュー
- **影響**: 「フォロワー全員に公開」「dueAt は変更なし」等

### 7. 認証情報の保護
- `.env.local` は .gitignore 対象
- BUFFER_TOKEN / THREADS_ACCESS_TOKEN_* をログ出力しない
- スプシに認証情報を書き込まない

## 操作別ガード詳細

### スプシ書き込み（追記）
- 範囲指定を必ずシート名込みで（`{sheetName}!A:G`）
- 既存行を上書きしないよう `+append` を使う
- 1投稿1行（複数の投稿を1行にまとめない）

### スプシ書き込み（更新）
- 行番号を必ずユーザー確認
- 範囲は1セル単位（D14、E14 等）
- 複数行の一括更新は dry-run で内容表示してから

### Buffer 既存予約編集（モード7）
- 編集前に Buffer から最新の dueAt と本文を取得
- 編集対象の確認をユーザーから2回もらう（時刻 + 本文プレビュー）
- 編集後は dueAt が維持されたかを Buffer GraphQL で確認

### 画像生成
- 一度に大量生成しない（5件以上はユーザー確認）
- Drive アップロード後、公開URLが取得できたかをスプシ書き込み前に確認

## エラー時の挙動

- スプシ書き込み失敗 → 投稿案をターミナルに表示してリトライ案内
- Buffer API エラー → 失敗行のみスキップ、成功した分は正常完了
- 認証切れ → ユーザーに `--show-browser` 案内 or .env.local 更新案内
