# 06 サブエージェント個体記録

## 目的

Claude Code / Codex の各サブエージェントについて、「何体いるか」だけでなく、**誰が・何を依頼され・いつ動き・何を報告して終わったか**を個体単位で追えるようにする。

当日デイリーmdは現在状態の軽い画面のまま保ち、個体履歴は新規Tursoテーブルを唯一の構造化保存先にする。実際に何が行われたかの第1証拠は各runtimeのネイティブトランスクリプトとし、Tursoは検索・集計できる索引兼報告台帳として扱う。

## 現状

- `SubagentStart` / `SubagentStop` はClaude・Codexともruntime登録済みで、親sessionの `sub_n` と🔵/🟢を自動更新する。
- Turso `session_events` が持つのは、親sessionが `run|wait` から `sub` へ入った時と、最後のサブ終了で `run` へ戻った時の**親状態遷移**。2体目以降の開始、途中終了、agent_id、依頼内容、個別報告は履歴に残らない。
- 2026-07-11実データでClaude/Codex双方の `sub-start` / `sub-end` は確認済み。受け口E2Eも複数体の参照カウントを含め全PASS。
- Codex公式の `SubagentStart` payload は `agent_id` / `agent_type` / `turn_id` を持つが、**依頼プロンプト本文を持たない**。`SubagentStop` は加えて `agent_transcript_path` / `last_assistant_message` を持つ。トランスクリプト形式は公式に安定interfaceではないと明記されている。
- Claudeも開始・終了イベントと個体識別情報は利用できるが、runtime差がある。実装前に両runtimeの合成fixtureと、秘密値を除いた実payloadのキー一覧だけを再確認する。
- 現在のsession-boardは「MD確定→Tursoベストエフォート送信」。今回の個体記録は既存mdからの移行対象ではなく新規データなので、mdへ同じ本文を増やさずTursoから開始する。

## 境界

### 対象

1. 各サブ個体の開始・終了、親sessionとの帰属、runtime、agent type。
2. サブが自分の依頼を1行で言い直した `task_summary`。
3. 終了時の構造化報告（Summary / Evidence / Risks / Next）と取得成否。
4. 多重開始・多重終了・終了取りこぼし・Turso障害に耐える冪等な記録。
5. 個体履歴を確認する保存SQLまたはread-only CLI。

### 対象外

1. 当日デイリーmdへ個々の依頼文・報告本文を展開すること（現行の `↳ 🔵 サブN体` は維持）。
2. サブの思考過程・chain-of-thought・生の全プロンプト・生トランスクリプトをTursoへ複製すること。
3. v1でFocusmap UIを実装すること。UIは実データが安定してから別計画で判断する。
4. 過去トランスクリプトの一括backfill。新しい記録経路の安定を優先する。
5. 報告が薄いサブを最初からHookで停止・再実行させること。

## データ契約（計画時点の推奨）

新規 `subagent_runs`（仮称）を1サブ個体1行で持つ。`session_events` は親状態の時間計測という既存責務を変えない。

- 識別: `run_id`（`runtime:parent_session_id:agent_id` から決定的生成）、`runtime`、`parent_session_key`、`agent_id`、`agent_type`。
- 時刻: `started_at`、`ended_at`、`updated_at`。Hook到着順が前後しても早い開始・遅い終了を保持する。
- 状態: `status=running|finished|orphaned`、`stop_seen_n`、`report_state=missing|captured|invalid`。成功/失敗をHookが推測しない。
- 依頼: `task_summary`（1行・上限を設ける）、`task_source=self-register|final-report|missing`。
- 報告: `summary`、`evidence`、`risks`、`next_action`。v1は短い構造化テキストに限定し、生の全文は保存しない。
- 証拠導線: runtimeとagent_idを基礎にした `transcript_ref`。絶対pathを外部表示へ出さず、保存要否と表現を実装前レビューで確定する。
- 安全: 生プロンプト・推論・secret・認証値を保存しない。各テキスト列に長さ上限と最低限のredactionを置く。

確定前の論点:

1. `transcript_ref` をTursoへ保存するか、runtime＋IDからローカル解決するか。v1は絶対pathを保存せず、runtime＋親session＋agent_idからローカル解決する方式を推奨。
2. `evidence` を1列テキストにするか、子テーブルへ分けるか。v1は1列を推奨。
3. `orphaned` をreconcileで何分後に確定するか。v1は現行🔵閾値30分の2倍である60分を推奨し、実測後に調整する。

## 記録フロー

### 開始時（機械事実＋自己登録）

1. `SubagentStart` Hookが `agent_id` / `agent_type` / 親session / runtime / `started_at` をupsertする。
2. 同Hookがサブへ短い報告契約を `additionalContext` で注入する。
3. サブは依頼を読んだ最初の節目で `sub-describe --agent-id ... --task "<1行>"` 相当を呼び、`task_summary` を自己登録する。
4. 自己登録が来なくても開始自体は成功扱いにし、`task_source=missing` のまま可視化する。Hook本体を止めない。

`SubagentStart` payloadに依頼本文が無い以上、開始時の依頼内容を100%機械取得する設計にはしない。非公開・不安定なCodex内部DBやトランスクリプト形式への依存も避ける。

### 終了時（構造化報告＋補完）

1. サブの最終報告形式を `Task / Summary / Evidence / Risks / Next` に揃える。Taskは依頼の1行再掲。
2. `SubagentStop` Hookが `agent_id` で同じrunを更新し、終了時刻と構造化報告を保存する。
3. 開始時の自己登録が欠けていれば、最終報告のTaskで `task_summary` を補完する。
4. 報告が欠落・parse不能でも終了記録は残し、`report_state=missing|invalid` とする。v1ではサブをblockしない。
5. `last_assistant_message` は必要な構造化節だけを抽出し、全文保存しない。`agent_transcript_path` は証拠参照にのみ使う。

## 段階導入

### 段階0 — 契約固定とfixture

- Claude/Codexの開始・終了payloadについて、値を保存せずキー・型・欠損パターンを確認する。
- `run_id`、状態語彙、文字数上限、redaction、冪等更新規則をテスト先行で固定する。
- DB migration案は作成するが、本番適用は人間ゲートで止める。

### 段階1 — 観測のみ

- `subagent_runs` を本番へ追加し、開始・終了の機械項目だけを記録する。
- task/reportが無くても警告・blockしない。既存 `sub_n` と `session_events` の挙動を変えない。
- 10〜20個体で開始/終了対応、重複、孤児、Hook遅延を観測する。

### 段階2 — 自己登録と終了報告

- Start注入で `task_summary` 自己登録と終了フォーマットを案内する。
- Stopで構造化報告を取り込み、欠損率・invalid率を集計する。
- 注入を守らなくても処理継続。既存作業へのレイテンシと失敗影響を測る。

### 段階3 — 可視化と運用判断

- 保存SQL/read-only CLIで「親目標→session→各サブ→依頼→報告」を確認できるようにする。
- 欠損率、平均稼働時間、孤児件数を運用レビューする。
- 20個体以上の実測後、報告欠損を1回だけblockして再要求するかを**人間が判断**する。自動強制はこの子計画の既定に含めない。

## 失敗時の扱い

1. Turso送信失敗でサブの開始・終了を妨げない。Hookは非ブロッキングを維持する。
2. `subagent_runs` の失敗再送は個体upsertを冪等にする。既存events/logsスプールとの統合か専用キューかは実装前に決め、同じ事実を二重送信しない。
3. Stopだけ先に届いた場合も行を作り、後着Startで補完できる。
4. StartだけでStopが無い行は即削除せず、reconcileで `orphaned` にする。元トランスクリプトは削除しない。
5. 同一agentのStop再発火は `stop_seen_n` を増やし、最終時刻・報告を安全に更新する。Hook継続による再Stopを別個体として数えない。
6. redactionで確信が持てない本文は保存せず `report_state=invalid` とし、証拠参照だけ残す。

## 人間ゲート

1. Turso本番のテーブル/index作成と権限変更。
2. Codex Hook変更後の `/hooks` 再trust、Claude user設定変更が必要になった場合の適用。
3. 欠損報告をSubagentStopでblockする運用への昇格。
4. 過去データbackfill、既存md正本の縮退、Focusmap UIへの公開。

## 実装順と担当境界

1. 契約・fixture・SQL builder。
2. Turso migration案とread-only確認SQL。
3. 共有 `common.py` / `board.py` の個体run操作。
4. Claude/Codexの薄いシムから共通操作へ渡すpayload整形。
5. 観測のみE2E → 人間ゲート → 本番小規模実測。
6. 自己登録・構造化報告 → 欠損集計 → 強制有無の人間判断。

同じ共有ファイルとDB契約を触るため、実装は並列分割せず直列を既定とする。計画レビュー後の実装/レビューは2ペインでよく、実装=Claude Sonnet 5、レビュー=Codex gpt-5.5 xhigh。DB migration適用とruntime trustは指揮官ではなく人間ゲートで止める。
