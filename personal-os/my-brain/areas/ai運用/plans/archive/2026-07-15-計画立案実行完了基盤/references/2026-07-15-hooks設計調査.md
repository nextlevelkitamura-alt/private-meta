責務は次の通りとする。

| 受け口 | runtimeイベント | 役割 | 計画本文の編集 |
|---|---|---|---|
| `guard-plan-bucket-move.py` | `PreToolUse` | Bash入力中の生 `mv` / `git mv` と計画バケット対象パスを検出し、`bucketctl` へ誘導してdeny | しない |
| `guard-plan-closeout.py` | `Stop` | manifest phase、同期状態、レビュー待ち件数、日付rename要否を検査。未同期の `review_passed` だけ一度継続要求 | しない |
| `verify-plan-worker.py` | `SubagentStart` | write taskならmanifestのworktree指定・base・対象roleを照合し、欠落時は警告・文脈注入 | しない |
| `verify-plan-worker.py` | `SubagentStop` | implementerのresult packet、reviewerの必須評価、explorerの構造化結果を検証し、欠落時だけ一度継続要求 | しない |
| `shared/plan-closeout/common.py` | 共通 | manifest読込、phase判定、result packetの存在・schema検査、`planctl rename --check` 結果の解釈、runtime別出力用の決定を返す | しない |

`mark-wait.py` と `sync-subagent-status.py` は変更してplanロジックを持たせない。各イベントでは既存handlerと新handlerが並列実行されても相互依存しないようにする。

日付陳腐化はhookが「目的・子構成の大幅変更」を推測してはいけない。`planctl rename --check` が、manifestまたは計画メタデータの明示的な「大幅更新日」とフォルダ日付を比較して `rename_required` を返す契約を先に子02で定める必要がある。ファイルmtimeだけでの判定は誤検知するため不採用とする。

また、`SubagentStart` はworktreeを作成・割当できない。manifest上の指定と実際の `cwd` の一致がruntime payloadで観測できる場合だけ照合し、不一致なら警告する。実際の作成・割当はharnessの責務である。

## 3. runtime差

根拠:

- `personal-os/AIエージェント基盤/hooks-registry/references/claude-hooks.md`
- `personal-os/AIエージェント基盤/hooks-registry/references/codex-hooks.md`
- `personal-os/AIエージェント基盤/hooks-registry/references/claude-vs-codex-hooks.md`
- 実機確認: Codex CLI `0.144.1`、Claude Code `2.1.207`
- [Codex Hooks公式仕様](https://developers.openai.com/codex/hooks)
- [Claude Code Hooks公式仕様](https://code.claude.com/docs/en/hooks)

| 論点 | Claude Code | Codex CLI | 設計判断 |
|---|---|---|---|
| `PreToolUse` | 存在。Bash等をmatcherで絞り、denyできる | 存在。Bash・`apply_patch`・MCPを対象にできる | 生 `mv` / `git mv` 対策は両方で `^Bash$` |
| `SubagentStart` | 存在。agent typeをmatcherにできる | 存在。agent typeをmatcherにできる | worktreeは照合・案内のみ。開始阻止はしない |
| `SubagentStop` | 存在。継続要求可能 | 存在。継続要求可能 | result packet・評価不足の一回だけの再実行要求に使用 |
| `Stop` | 存在。継続要求可能 | 存在。継続要求可能 | `review_passed` 未同期だけを一回継続させる |
| Stop / SubagentStopへの追加文脈 | `additionalContext` は使わず、`decision:block` と `reason` を使う | `additionalContext` は使えない。JSONの `decision:block` と `reason` を使う | 両runtime共通でblock形を採用 |
| 登録 | `~/.claude/settings.json` を保存すれば反映。trust不要 | `hooks.json` を読む。内容変更後に `/hooks` でtrustが必要 | 実装と登録を分離し、承認後のみ適用 |
| hook並列性 | 同一eventの複数hookが実行される | 同一eventの複数command hookが並列起動する | `mark-wait` とcloseout guard、subagent同期と検証を依存させない |

stdout契約は次で統一する。

- `PreToolUse` deny: `hookSpecificOutput.permissionDecision: "deny"` と理由をJSONで返す。
- `Stop` / `SubagentStop` 継続: `{"decision":"block","reason":"..."}` をJSONで返す。
- 通過: exit 0・stdoutなし。
- 案内のみ: Claudeは追加文脈またはsystem message、Codexは `systemMessage` に限定する。CodexのStop後は追加文脈を次ターンへ注入できないため、重要な未同期だけをblock理由にする。

`SubagentStart` は、ClaudeでもCodexでも開始を停止する強制点としては使えない。Codex公式仕様では `continue:false` はSubagentStartを止めず、Claude公式仕様でも開始時のexit 2は開始阻止にならない。従って、worktree割当の担保をhookへ委譲してはいけない。

未確認事項は、`PLAN_RUN_MANIFEST` がharnessからClaude/Codex本体、およびそのsubagent hookプロセスへ継承されるかである。公式仕様はhookのstdin payloadを定義しているが、この任意環境変数の継承を契約としては保証していない。harness実装後、両runtimeの実機E2Eで確認する必要がある。

## 4. 規律整合

根拠:

- `personal-os/AIエージェント基盤/hooks-registry/AGENTS.md`
- `personal-os/AIエージェント基盤/hooks-registry/shared/session-board/common.py`
- `.../plans/04-PromptSubmitとhookガード.md`
- [Codex Hooks公式仕様](https://developers.openai.com/codex/hooks)
- [Claude Code Hooks公式仕様](https://code.claude.com/docs/en/hooks)

「内部失敗で本体を止めない」と「未同期なら継続要求」は矛盾しない。blockは、検証が成功して明確な契約違反を検出した時だけ発生させる。manifest読込失敗、JSON不正、schema不一致、対象外role、`PLAN_RUN_MANIFEST` 不在では必ずfail-openにする。

判定表:

| 条件 | 動作 |
|---|---|
| `PLAN_RUN_MANIFEST` 不在 | 常に通す |
| manifest不正・読込失敗・必要キー不足 | 通す。必要ならsystem messageだけ |
| `running` / `implemented` | 通す。一括レビュー待ちなら件数だけ案内 |
| `review_passed` かつ未 `synced` | 一度だけ継続要求。`planctl apply-evaluation` または `sync-check` を理由に示す |
| `synced` / `closed` | 通す |
| `blocked` | 通す。blockerを案内するがhookは解消しない |
| implementerでresult packet欠落 | 一度だけSubagentStop継続要求 |
| reviewerで必須評価欠落 | 一度だけSubagentStop継続要求 |
| explorer | 構造化結果があれば通す。結果の意味評価はしない |

連続block上限は1回とする。理由は、両runtimeの `stop_hook_active` が「すでにStop hookが継続させたか」を示すためである。`stop_hook_active=true` なら、同じ論点で二度目のblockはせずfail-openする。hook自身が永続カウンタを持つと、状態所有境界を不必要に増やす。

`hooks-registry/AGENTS.md` への追記文案:

> - hookは原則非ブロッキングとし、内部失敗・不正入力・対象外・manifest不在ではexit 0で通す。  
> - 明示manifestを正常に検証でき、かつ実行契約の未同期または必須result欠落を確定検出した時だけ、Stop / SubagentStopで一回だけ継続を要求できる。  
> - 継続要求は `stop_hook_active` がfalseの時だけ行う。trueなら再blockせず、理由を案内して通す。hookは永続カウンタ、計画本文、計画バケット、result packetを変更しない。  
> - `PLAN_RUN_MANIFEST` が無い通常セッションにはplan-closeout処理を作用させない。  
> - runtime固有差は登録表とstdout JSON形式に閉じ、判定ロジックは `shared/plan-closeout/` に集約する。

| 段階 | hook | harness / program-run | 人間ゲート・注意 |
|---|---|---|---|
| worktree作成 | manifestの指定、存在、base、`cwd`を照合できる範囲で案内。作成しない | 明示baseからtask-scoped worktreeを作成し、manifestへ記録 | `SubagentStart` は開始を止められないため、作成責務を置かない |
| 実装 | 監視・意味判断をしない | Task Packetに従いworkerを起動、対象path・write laneを管理 | hookが計画・コード・manifestを編集しない |
| result packet回収 | SubagentStopで存在・schema・role別必須項目を検証し、一度だけ再実行要求 | result packetを回収・schema検証し、review queueへ載せる | packetがないままPASS扱いにしない |
| レビュー | reviewerの評価不足だけを検出 | reviewerを起動し、PASS/FAILをplanctlへ渡す。一括レビュー待ちはqueueで管理 | `implemented` の一括待ちはStopで止めない |
| 統合branchへのmerge | mergeしない。状態案内のみ | 現行の子03契約は「push・mergeをしない」としているため、現時点では統合候補・差分提示まで | 「program-runが統合branchへmergeする」とするなら、子03の禁止事項と矛盾するため、先に計画を明文化して人間承認が必要 |
| worktree削除 | 削除しない | cleanup候補と安全条件を提示する。自動削除しない | Global AGENTSの削除ルールにより、実削除は明示承認が必要 |

特に「worktree作成→レビュー→統合branch merge→削除を人間なしで完走」と「子03のpush・merge禁止、Global AGENTSの削除は明示承認」は両立していない。現行計画に従うなら、program-runの完走ラインは「実装・レビュー・承認セット作成」までで、merge・削除は承認後の別適用である。
