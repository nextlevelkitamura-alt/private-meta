# project-slide-workflow

プロジェクト型のスライドを「壁打ち → 内容確定 → 1枚ずつ確認 → 生成/プロンプト化 → PDF化」するためのグローバルSkill。

## 基本方針

- いきなり全ページを作らない。まず全体コンセプト、対象者、付加価値、ゴール、利用場面を固める。
- スライドは1枚ずつ進める。各ページで「見出し・本文・入れる要素・デザイン案」を提示し、ユーザー確認後に生成へ進む。
- ユーザーが修正した文言を最新版として扱う。修正前の画像・文言は最終成果物に入れない。
- 資料の用途を確認する。参加者向け資料ならCTAを入れすぎない。広告/LP/LINE向けならCTAを入れる。
- 文章は上から目線にしない。「教える」「変えさせる」より「一緒に整理する」「一緒に進む」「相談できる」を優先する。

## 起動条件

ユーザーが以下のように言ったら使う。

- 「資料を作りたい」「スライドを作りたい」「画像でスライド作って」
- 「このプロジェクトの打ち出しを固めたい」
- 「壁打ちしてからスライド化したい」
- 「images gen2で資料を作りたい」
- 「Claude/Codexどちらでも使える資料作成フローにしたい」
- 求人・キャリア支援・営業企画・LP・LINE配信・Instagram投稿に転用する資料を作りたい

## 環境分岐

### Codexで画像生成ツールが使える場合

- ユーザーが画像生成を許可したページだけ、画像生成へ進む。
- 画像生成前に、必ずそのページの内容案をテキストで提示する。
- 生成後にユーザーが修正したら、修正版を生成し、最終PDFには修正版だけを入れる。
- 画像生成物を別パスへ移す場合も、元画像は削除しない。
- PDF化するときは、採用版の画像だけを順番に並べる。ページ数とファイルサイズを確認する。

### Claudeなど画像生成できない環境の場合

- 画像は生成しない。
- 各スライドの確定内容と、画像生成ツールへ渡すプロンプトを作成して止める。
- 「この環境では画像生成せず、プロンプト化までにします」と明示する。
- 1枚ずつ確認する流れはCodexと同じ。

## 標準ワークフロー

### 1. 目的と用途を確認

最初に不足している場合だけ、以下を短く確認する。

- 誰に見せる資料か
- 何枚構成か
- 参加者向けか、広告/LP/LINE向けか
- ロゴやブランド素材の有無
- 使うテンプレ・UIイメージの有無
- 最終成果物は画像、PDF、プロンプト、PPTXのどれか

ユーザーが既に十分な情報を出している場合は、質問で止めずに仮案を出す。

### 1.5. テンプレを選ぶ

既存テンプレを使うか、今回の資料から新規テンプレ化するかを確認する。  
テンプレ候補と保存ルールは [references/templates.md](references/templates.md) を読む。

提示例:

```markdown
今回のUIはどれで進めますか？

1. NEXTLEVEL CAREER 白×赤ミニマル
2. 既存スライドの見た目を踏襲
3. 今回だけ新規で作る
```

選んだテンプレについて、最低限以下を明示してから進める。

- ロゴ: 使用するロゴのパス
- 比率: 16:9など
- 色: ベース色、本文色、アクセント色
- UI: カード、ロードマップ、線画アイコンなど
- 禁止: 入れない要素、CTA有無、写真の有無

### 2. 全体コンセプトを固める

以下を整理して提示する。

- 資料名
- 対象者
- 解決したい悩み
- 用意する付加価値
- 資料全体のトーン
- 避ける表現
- 最終的に伝えたい一文

求人・キャリア支援系では、誇大広告に見える表現を避ける。

避ける例:
- 「必ず」「保証」「誰でも」「絶対」
- 「すぐ正社員になれる」
- 「誰でも高収入」
- 実求人とズレる夢だけの訴求

使いやすい表現:
- 「目指せます」
- 「相談できます」
- 「一緒に整理します」
- 「条件が合えば」
- 「できる場合があります」
- 「まずは現実的な一歩から」

### 3. スライド構成を作る

5枚構成の基本形:

1. 表紙: プロジェクト名、ロゴ、短い説明
2. 共感: 対象者の悩み
3. 整理: 現在地チェック、付加価値
4. 行動: ロードマップ、流れ、開始までのステップ
5. 伴走: 参加後サポート、相談できること、次の可能性

用途によって変える。

- 参加者向け: CTAを控えめにし、安心感と流れを重視する
- 広告/LP向け: CTAを入れ、相談導線を明確にする
- 社内共有向け: 運用ルール、提供価値、注意点を入れる

### 4. 1枚ずつ確認する

各ページは、作成・画像生成・PPTX反映の前に必ず構成案を提示する。  
ユーザーが「作って」「OK」「これで進めて」など明示するまで、実ファイル生成へ進まない。
構成案には最低限以下を含める。

- そのスライドの役割
- 見出し案
- 本文案
- 入れる図解・グラフ・表・アイコン
- デザインイメージ
- ユーザーに確認したいポイント

各ページごとにこの形式で出す。

```markdown
## {n}枚目案

**役割**
...

**見出し**
...

**本文**
...

**入れる図解・グラフ・表**
- ...
- ...

**下部コピー**
...

**デザインイメージ**
...

**確認ポイント**
...
```

ユーザーが「作って」と言うまで画像生成しない。  
ユーザーが「次の内容を見せて」と言ったら、次ページの内容案だけを提示する。

### 5. 画像生成プロンプトを作る

Codexで生成する場合も、Claudeでプロンプト出力する場合も、各スライドのプロンプトには以下を含める。

- 16:9
- 白背景
- 余白多め
- ブランド色をアクセントにする
- 日本語テキストを読みやすく
- 上部に不要なヘッダーを入れない
- カードやロードマップは最小限
- 写真感が不要なら、線画・アイコン・抽象イラストにする
- 文言を勝手に増やさない

チャットにそのまま投げる総合プロンプトが必要な場合は、[references/chat-prompt-template.md](references/chat-prompt-template.md) を読む。

NEXTLEVEL CAREER系の標準UI:

- 白背景
- 黒/濃いグレー本文
- 赤アクセント
- ロゴ: `/Users/kitamuranaohiro/マイドライブ（nextlevel.kitamura@gmail.com）/01_ネクストレベル/素材/ネクレベ　キャリア　ロゴ/ネクストレベル　キャリア　ロゴ　横長.jpg`
- ロゴは表紙では大きめ、2枚目以降は控えめ
- 丸みのある薄いカード
- やわらかい線画アイコン
- 参加者に寄り添う雰囲気

### 6. 生成後の取り扱い

- 生成画像が複数ある場合は、ユーザーが承認した最新版だけを採用する。
- 修正前画像は最終PDFに入れない。
- PDF化前に採用する画像パスを明示的に決める。
- PDF化後はページ数を確認する。
- 完成後に「このUIをテンプレとして保存するか」を確認する。
- 保存する場合は、テンプレ名、ロゴパス、採用画像、共通プロンプト、各ページ構成を記録する。

## 参加者向けキャリア支援資料の文体

優先する言い方:

- 「一緒に整理します」
- 「一緒に探していきます」
- 「相談できます」
- 「ひとりにしません」
- 「少しずつ始められます」
- 「働き始めた後も一緒に進みます」

避ける言い方:

- 「変えましょう」
- 「卒業しなければならない」
- 「あなたに足りない」
- 「正しい働き方」
- 「今のままだと危ない」

## PDF化の実務メモ

Codexでローカル画像をPDF化する場合:

- 採用版画像を順番に並べる
- 可能なら`magick`でPDF化する
- 出力先はユーザーの作業文脈に合う場所にする
- 作成後に`mdls -name kMDItemNumberOfPages`などでページ数を確認する

例:

```bash
magick slide1.png slide2.png slide3.png slide4.png slide5.png output.pdf
mdls -name kMDItemNumberOfPages output.pdf
```

## 今回のスキマバイト卒業プロジェクトの型

再利用できる構成:

1. **スキマバイト卒業プロジェクト**  
   単発・短期の働き方から、少しずつ安定した毎日へ。

2. **毎回仕事を探す生活、少し疲れていませんか？**  
   今の働き方を活かしながら、次の一歩を一緒に考える。

3. **まずは、今のあなたに合う働き方を整理しましょう**  
   現在地チェック: 収入の希望、得意・苦手、続けやすい条件、今後の方向性。

4. **相談からお仕事開始まで、スムーズに進めます**  
   初回面談 → お仕事提案 → 職場見学・条件確認 → 登録・手続き → お仕事開始。

5. **働き始めた後も、毎月相談できます**  
   仕事の悩み、キャリア相談、次のステップ。仕事変更や正社員紹介も相談可能。
