# cockpit-supervisor

orca-cockpitが構築した「実装/レビュー(既定2ペイン)＋任意の計画+監督ペイン」を指揮官（人＋中央Claude）が監督するための判断手順メタスキル。決定的な構築・分割・agent起動・検知は `orca-cockpit`（`scripts/cockpit.sh` / `scripts/watch.sh` / `references/role-prompts.md`）が担う。本Skillは「起きたら何を判断するか」だけを持つ（plans/10 方針2・9: 機構=orca-cockpit／判断と監督=本スキル）。

## 1. 役割

1. 見張り番（`orca-cockpit/scripts/watch.sh`）がWAKEでexitした時、検知理由ごとに次の一手を判断する。
2. フル起動前の構成カード提示（2ペイン既定か3ペイン例外か）を判断する。
3. レビューPASS/FAILの扱い、差し戻し上限、マージ前の人間ゲートを判断する。
4. 対象外: `cockpit.sh`の引数・`watch.sh`の検知ロジック・役割プロンプトの定型文そのもの。正本は `orca-cockpit`。本Skillはそれを本文複製しない。

## 2. 見張り番 起床時の判断手順（WAKE理由ごと）

見張り番は判断をしない（grepと時計だけ）。WAKE理由1行を読んで、指揮官（人＋中央Claude）が以下の手順で判断する。**一次対応者は管轄GM/指揮官**: 権限確認・質問・停滞（6項waiting／9項ダイアログ・利用上限）は目安2分以内に解除しに行き、並列運用で「指示しないと止まる」状態を作らない。人間へ上げるのは人間ゲート項目（マージ・push・削除・外部送信等）だけで、人間がペインを直接突いて解除するのは例外運用（`GLOBAL_AGENTS.md` §7・2026-07-03裁定・背景=send流出でレーンを120分放置した事例）。

1. **error/failed/crashed（即WAKE）**: 対象ペインへ `orca terminal read` で直近ログを確認→原因が明確なら是正send、不明なら人間へ即エスカレーション。
2. **人間確認待ちマーカー（`段階: 人間確認待ち`・即WAKE）**: 変更一覧・承認セットを読み、人間の判断を仰ぐ。AIだけで先に進めない。
3. **実装完了マーカー（`<子番号>_DONE`）→レビュー配布**: 判断不要の機械的な遷移。レビューペインへ「実装完了。差分をレビューして`REVIEW_RESULT`で返答」を送るだけでよい。判断コストが低い遷移なので、将来的に見張り番からの自動送信に置き換えても構わない（4節4項）。**配布先は盤面に映るレビューペイン**（headless codex exec 等での代替はしない＝稼働エージェント数が盤面/Notionに映らないため。`GLOBAL_AGENTS.md` §7・2026-07-03裁定。やむを得ない例外はデイリー行注記1行）。
4. **`REVIEW_RESULT: PASS`→マージ手順と人間ゲート**: レビュー合格をログ。マージ・push・main反映は常に人間ゲート（`GLOBAL_AGENTS.md` §7）。人間へ「レビュー合格。マージしてよいか」を1回確認してから `cockpit.sh down` 等の後片付けへ進む。
5. **`REVIEW_RESULT: FAIL`→差し戻し判断**: FAILの具体的理由を実装ペインへsendして差し戻す。**差し戻し上限＝2回**。3回目のFAILで指揮官（人間）へ即介入を仰ぐ（AIだけでループさせない）。差し戻しsendの型（mokuteki-jissoから吸収 2026-07-03）: FAIL理由の具体行の転記＋「残す成果／直す対象」の区別＋受け入れ条件の再掲＋新しい完了マーカー名。特に2回目の差し戻しでは全面書き直しへ流れないようこの区別を必ず書く。
   - **規模で分岐（レビューゲート・決定ログ#9）**: 上記の探索込み・差し戻し上限2は「フル」の既定。**ライトは1パス・差し戻し上限＝1**（1回のFAIL後もダメなら人間へ）、範囲は差分×完了条件の照合＋変更行のバグ境界のみ（周辺は掘らない）。レビューモデルはフル=codex xhigh／ライト=codex high。規模判定は `GLOBAL_AGENTS.md` §7。実装者が構造3条件を外れたと`_DONE`前に申告したら、指揮官が規模を上げてレビュー方式（範囲・上限・モデル）も引き上げる。
6. **waiting継続（約2分）→権限解除**: 権限確認/質問ダイアログで無音停止している。`orca terminal read` で内容を確認し、承認かキャンセルかを人間/指揮官が判断してsendする。
7. **全ペインidle継続（約4分）→突き**: 完了マーカー無しで停滞。`orca terminal read` で状況確認→BG監視の宣言がターン失効している可能性が高い（7節）。是正sendで再開を促す。
8. **3時間タイムアウト→点検**: 進捗未達。`orca worktree ps` とterminalログで全体を点検し、続行/中断/仕切り直しを人間と判断する。
9. **対話ダイアログ/利用上限の画面シグネチャ（working継続約5分+画面固定文字列・即WAKE）→ペイン解除**: stateはworkingのまま、codex利用上限セレクタ等が画面を塞いでいる（運用学習17=統合program plans/10）。手順: (a) `orca terminal read` で実画面を確認。 (b) モデル切替セレクタなら「3」（現行モデル維持・今後表示しない）をsend——レビューをmini等へは下げない（決定ログ#3/#9。ライトはhigh・フルはxhighが下限）＝miniへの切替提案は常に拒否。 (c) 元ターンは死んでいるので同内容を再投入し送達確認（学び13）。 (d) 上限が未解除（画面の「try again at HH:MM」が未来時刻）なら、その時刻以降の再投入予約とし当該ペインのみ待機・他レーンは継続。 (e) 待てない場合の代替エンジン切替（例: レビューをclaude高位モデルで一時代替）は人間ゲート。予防: xhighレビューを同日に多数回す時は大型レビューを時間分散し、上限到達の発生時刻はplans/10学び17へ追記して残量パターンを学習する（codexの残量はCLIから機械取得不可のため、1分検知＋履歴学習で代替する）。

## 3. 構成カード（起動前の判断）

1. フル起動前は必ず `cockpit.sh plan …` の構成カードを人間へ提示し、OK後に `up` する。モデル構成をAIが黙って決めない。
2. **ライト起動は事後報告でよい**（`GLOBAL_AGENTS.md` §7と同基準）。軽微な修正・小規模タスクで構成が自明な場合は構成カード提示を省略し、起動後に一言報告する。
3. 構成カードの様式（`cockpit.sh plan` が出力・様式自体の正本はorca-cockpit）: レーン名(title) / repo・branch / ペイン構成(位置・役割・kind・モデル・effort) / 権限モード / 規模(scope)。
4. **モデル適用基準（2026-07-02 ユーザー裁定）**: 役割方針と既定モデルの正本は `../AIモデル一覧.md`。構成カードはその一覧を参照して決め、ここでは実行時の例外だけを扱う。
   - **クリティカルパス例外**: そのレーンの完了を人間・指揮官・他レーンが待ってブロックされる場合（直列ボトルネック）は、一覧の通常実装モデルを claude opus 4.8 の fast モードへ格上げしてよい。判断の問いは「**このレーンが遅いと誰が待つか**」— 誰も待たない（並列で吸収される）なら一覧の既定を維持する。
   - fastモードの立ち上げ: `cockpit.sh agent --kind claude --model claude-opus-4-8` で起動後、`/fast` を1通sendしてトグルする（Opus 4.8/4.7のみ対応。初回は効いているか目視確認）。
   - レビューeffortは規模連動（フル=xhigh維持／ライト=high・決定ログ#9で#3を一部上書き）。一覧のレビュー系統を変えず、mini等への降格は常に拒否。
   - 格上げは構成カード（フル）または事後報告（ライト）で必ず宣言する。黙って変えない。
5. **レーン分割前の衝突確認（mokuteki-jissoから吸収 2026-07-03）**: 非自明な計画を複数レーンへ分ける前に、readonly調査（軽ければ指揮官自身のgrepで可）で「触る面・同一ファイルを触るレーンの有無・先に固定すべき共通契約」を確認する。同一ファイルに触るなら直列化するか共通契約を先に固定し、迷ったら直列。worktree分離でもマージ時衝突の予防として必ず行う。

1. **既定＝2ペイン（実装/レビュー）**: 計画が子計画文書として完成している通常ケース。監督の実務（検知・差し戻し判断・マージ）は指揮官＋見張り番に一本化する（2026-07-02裁定）。
2. **3ペイン例外（計画+監督を追加）**: 以下のいずれかに該当する場合のみ、構成カードで3ペイン目を明示宣言する。
   - 計画がまだ子計画文書として固まっていない（企画〜計画段階から始める）。
   - 現場判断（要件の揺れ・仕様の即興決定）が多く見込まれる。
   - 実装ペインへの配布そのものを1ペインに任せたい規模のタスク。
3. 3ペイン目を使わない理由: 「指揮官の配布文と二重の計画圧縮」「ターン失効する監視」「権限停止点」にしかならないと実証済み（2026-07-02裁定）。
4. 「実装完了→レビュー開始キック」のような判断不要な遷移は、指揮官が都度深く迷わず機械的に送信してよい。見張り番自体に判断ロジックを足すのではなく、固定文面の送信のみを足す形での自動化拡張も許容する（現状は未実装・指揮官が手動で行う）。

## 5. 見張り番の運用標準

1. **起動**: 必ず**通知つき背景タスク**として起動する（シェルの`&`では終了通知が来ず自動起床しない）。
2. **生存確認と再起動**: 指揮官は起床のたび（WAKEで戻ってくるたび）見張り番が生きているか確認し、死んでいれば必ず再起動する。見張り番は1回のWAKEでexitして終わる一発ものなので、対応が終わったら次のexitに備えて再度起動し直す。
3. **監視マーカーの再装填**: 段階（企画/計画/実装/レビュー/…）が進んだら、監視対象・マーカーの前提が変わるので見張り番を再起動して条件を再装填する（古いレーン構成のまま回し続けない）。処理済みマーカー（agentのlastAssistantMessage最終行は静的に残り続ける）は `WATCH_SEEN="<最終行>,..."` で必ず除外してから張り直す（渡し忘れると同じマーカーで再ウェイクし続ける）。
4. **常設の最終防衛線はない**: 旧常設キーパーは2026-07-11に廃止。監督中は本節1〜3の `watch.sh` を指揮官が再起動して維持し、指揮官不在中の自動通知を前提にしない。

## 6. 完了判定・マーカー標準・報告様式（参照のみ・本文複製しない）

1. **`state=done` は完了ではない**: `orca worktree ps` の `state=done` は"ターンが終わった"であって"作業完了"ではない。完了判定は明示マーカー（`<子番号>_DONE` / `REVIEW_RESULT: PASS` / `段階: 人間確認待ち`）のみで行う（done誤認の正体）。
2. マーカー標準（段階語彙・`REVIEW_RESULT`・`<識別子>_DONE`・単独最終行ルール）と各ペインの報告様式は `orca-cockpit/references/role-prompts.md` が正本。

## 7. 監督（計画+監督ペイン）の限界

1. BG監視はターン内ブロッキング待ちでのみ有効。ターンを跨いだ自己再開はできない（宣言した「バックグラウンド監視」はターン終了で失効する）。
2. 失効した場合は見張り番の停滞検知（全ペインidle継続・約4分）→指揮官の突きで回復する（実績: 子03実装完了の停滞を4分で検知・回復）。

## 8. 安全方針

1. secret / `.env` / 認証値をプロンプト・ログ・run表示に書かない。
2. 破壊的操作（削除・push・main反映・hook/launchd登録）は常に人間ゲート（`GLOBAL_AGENTS.md` §7）。
3. 本Skillは判断手順のみを持つ。決定的処理（worktree作成・分割・agent起動・検知）は行わない。正本は `orca-cockpit`。
4. **新しい玉は kickoff で起票してから采配**: 監督中に采配外の新しい玉（別作業）に気づいたら、勝手に着手せず kickoff（起票ゲート）で1行起票してから采配へ回す（作業の入口＝起票ゲート・出所の保証。子05）。
