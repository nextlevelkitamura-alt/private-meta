# 設計結論: 仕事デイリーのDB参照化（2026-07-19壁打ち）

2026-07-19の壁打ちで到達した設計結論。子計画はこの文書を骨子として立案する。

## 結論サマリ

1. **デイリーmdの毎回生成をやめ、DBを「今日やること」の正本にするのはすぐ可能。** 取得の仕組み（HTTP API・MCP）は既製で、新規テーブルも不要。
2. **体感速度はほぼ変わらない。** 1クエリ100〜300ms。セッション冒頭に1回まとめて取得＋完了時だけ書き戻す設計なら、LLMの1ターン（数秒〜）に対して誤差。デイリーmd生成の手間が消える分むしろ速くなる。
3. **取得手段はMCPではなくCLI/API。** MCPは全ツールスキーマがセッション常駐となり、使わないセッションでも数千トークンを固定消費する。取得スクリプト1本（curl/tsx）をBash経由で叩けばコンテキスト消費は結果分だけ。
4. **正本は Turso `todos` に合わせる（推奨）。** 理由は下記。

## 正本の選択: Supabase tasks vs Turso todos

| | 案X: Focusmap Supabase `tasks` | 案Y: Turso `todos`（inbox DB）★推奨 |
|---|---|---|
| 位置づけ | アプリ本体のタスク・Todayパネルの実体 | 当日ボードSQL化が新設した「やること箱」。`do_date`/`assignee`(自分\|AI)/`ai_status` を持つデイリー専用設計 |
| 既存方針との関係 | アプリのメイン画面と直結 | **personal-osが既に選んで走っている方向**（md→Turso正本反転の途中） |
| 画面反映 | Supabase Realtimeで即時 | Realtimeなし。focusmap画面は20秒ポーリング |
| エージェント取得 | HTTP API（`sk_focusmap_` キー） | libSQL HTTP pipeline APIへcurl（トークンはKeychain） |
| カレンダー・習慣連携 | 同一DBで自然 | Supabase正本と二層併存になる |

推奨理由: 今からSupabase `tasks` を正本に立てると、activeで走っている「当日ボードSQL化」programと正本が競合し「正本2つ問題」を自作することになる。Supabase側はカレンダー・習慣・Realtime表示の担当として併存させる。

## 構成: 契約md方式

注入するのは「データそのもの」ではなく「**取り方の契約書**」。3点セットで完結させ、MCPに依存しない。

1. **契約md**（取得先・当日判定・ステータス語彙・書き込みルール・失敗時フォールバックを書いた数十行）
2. **取得スクリプト1本**（curl または tsx。実行すると当日タスクがJSONで返る）
3. **`.env` のトークン参照**（契約mdには変数名だけ書き、値は直書きしない）

## 読み込ませ方と置き場所

- **読み込ませ方はリファレンス方式（遅延読み込み）**: 契約mdを /morning・/task の SKILL.md 必読リストに1行足す。既存のスキル構造がそのまま遅延読み込み機構として機能し、追加の仕組みゼロ・戻すのも1行。
- **hooksは見送り**（SessionStart hookでの実データ注入はPhase 2以降の最適化に取っておく）。理由: 常時発火の無駄、トークン管理の増加、`.codex/hooks.json` がMac絶対パスで壊れている前例。
- **契約mdの正本は仕事repo `方針/Focusmapデイリー参照.md`**（名称は立案時に確定。取得先がTursoになるなら名称も合わせる）。理由: ①消費者（/morning・/task）の最寄り、②外部サービス接続は `方針/api-reference.md` に集約する既存規約の並び、③「書き込みは完了報告のみ」等の仕事側ルールを含むため。
- スキーマの真実はDB所有側（focusmap `db/turso/migrations/`）に残し、「テーブルを変えたら同じ作業内で契約mdも更新」を両側に一文入れる。
- personal-os側には二重管理せずリンク参照のみ。

## 段階移行

- **Phase 1（読み取りのみ・リスクほぼゼロ）**: 契約md＋取得スクリプトを作り、/morning・/task が「今日やること」をDBから取得。schedule.mdは残す。
- **Phase 2（書き戻し・正本切替）**: 完了報告を status 更新で書き戻し。ai-todo-sync（md→DB）を停止。**この切替タイミングだけは明示的に決めて実行**（正本が2つある期間を作らない・不可逆）。
- **Phase 3（md格下げ）**: schedule.md は日次スナップショット（アーカイブ）出力先に。/morning のスプシF列参照も統合検討。ここで三重化が解消。

## 前提条件・注意

- **移管前にTursoスキーマ正本を migrations/ に集約する**（sessions / session_logs / session_events のCREATE文が本番手設置のまま。turso現状調査参照）。
- 認証情報は契約mdに直書きしない。
- スキーマ変更時は契約mdを同じ作業内で更新する。
- 取得失敗時のフォールバック（前日スナップショットを読む等）を契約mdに1行入れる。
