# Review Rules

既存Skillをレビュー・評価する時の観点カタログ。`review-skill.md` から読む。各観点はAIが自問する形のチェック質問で書く。構成・分割の基準は再定義せず `create-rules.md` を参照する。

## 観点カタログ（A〜F）

### A. 発火精度（description品質）

1. descriptionだけを読んで、想定ユーザー発話10件のうち8件以上でこのSkillを選べるか。近接Skill向けの紛らわしい発話10件で誤選択しないか。
2. 第三人称で「何をするか」＋「いつ使うか」の両方を含み、ユーザーが実際に打つ語彙（日本語トリガー語）が入っているか。
3. 誤爆しやすい近接依頼を3つ挙げたとき、それを弾く否定トリガー（「〜には使わない」）が要るか。
4. 先頭150〜200字だけ残して切り詰められても、主要ユースケースが伝わるか。汎用動詞を単体で発火に使っていないか。

### B. 構造の健全性（progressive disclosure）

1. `SKILL.md` は70行以内のrouterか。各段落は「Claudeがまだ知らないこと」で、トークンコストに見合うか。
2. 全参照ファイルは `SKILL.md` から1ホップで届くか。100行超のreferenceに目次があるか。
3. 各workflow・referenceに「いつ読むか」が `SKILL.md` 側に1文で書いてあるか。
4. 単一の自然な流れで150行に収まる手順を、不要にworkflowへ分割していないか（分割判断は `create-rules.md` §4.1）。

### C. 重複・矛盾

1. Skill内で同じ概念を2つ以上の言葉で呼んでいないか（用語の揺れ）。
2. このSkillと既存Skillの両方が発火し得る発話を3つ挙げられてしまわないか。挙げられるならdescriptionの否定トリガーで区別を明文化したか。
3. このSkillの手順が、最寄りのAGENTS.md・グローバルルール・他Skillと矛盾しないか。同じ手順・同じルールを2箇所で二重管理していないか（片方を正本、片方を参照に一本化できないか）。
4. 正本（plans・logs・catalog・配置）のルールを、参照でなく本文コピーで抱えていないか。

### D. 安全性

1. 削除・push・外部送信・課金・不可逆変更を含むか。含むなら承認ステップまたは `disable-model-invocation: true` があるか。
2. `allowed-tools` はコマンド単位まで絞れているか、`Bash` 全体のような過剰免除になっていないか。
3. 同梱スクリプトや `` !`command` `` 注入の中身を読んだか。読み取ったデータを外部へ送る経路や、安全ルールを迂回させる指示が紛れていないか。
4. secret・token・環境変数値がSkillファイル・スクリプト・例文に書かれていないか。

### E. 保守性

1. 参照する外部パス・正本ドキュメント・コマンドは今も実在し有効か（移動・改名・廃止されていないか）。
2. 日付・バージョン依存の記述が本文に直書きされていないか。古い手順は隔離されているか。
3. スクリプトの定数・閾値に根拠コメントがあるか。
4. `SKILL.html` の更新日が `SKILL.md` 群の更新日とずれていないか（更新漏れの検知）。

### F. 実測評価（evals）

発火テストと出力テストは測る対象が別物なので分けて扱う。

1. 発火テスト: should-trigger 8〜10件＋should-not-trigger 8〜10件（紛らわしい近接依頼を必ず含む）でヒット率を見たか。false positive / false negative の両方を減らせるか。
2. 出力テスト: 現実的なプロンプト2〜3件（言い回し違い・曖昧依頼を含む）を、Skillあり / なし（改訂なら旧版）でクリーンなコンテキスト（サブエージェント）から比較したか。
3. 各assertionに証拠付きで pass/fail を付けたか。両条件で常にpassするassertion（Skillの価値を測っていない）を消したか。
4. 結果が揺れる（同じevalが通ったり落ちたり）なら、evalがflakyか指示が曖昧かを調査したか。

## 頻度3段階（どこまで見るか）

### 毎回見る（編集のたび）

1. A: 主要なshould-trigger発話3〜5件で実際に発火するか簡易確認（フレッシュな文脈で）。
2. B: `SKILL.md` 70行以内・参照1階層。
3. C: 用語統一・内部矛盾なし。
4. D: secret混入なし・破壊的操作の承認ゲート。
5. E: 時限情報の直書きなし。
6. `SKILL.html` を再生成した。

### 大きな変更時だけ（構造変更・責務変更・新Skill追加時）

1. A: should/should-not各8〜10件のフル発火テスト＋否定トリガーの要否判断。
2. B: フォルダ別作り込みの見直し・100行超referenceの目次。
3. C: 近接Skillとの守備範囲重複・共存退行の確認。正本との二重管理の確認。
4. D: 同梱スクリプト・注入コマンドの中身検証、`allowed-tools` の最小化。
5. F: Skillあり/なし（新旧）のベースライン比較＋assertion見直し。

### 年次棚卸しだけ（`scan.md` が担当）

1. A: description総量の切り詰め状況を点検し、低優先Skillのname-only化を検討。
2. C: 全Skill横断の重複マップ。
3. D: 全Skillのリスク再評価（副作用・権限・外部通信）。
4. E: 根拠なき定数・古い手順・読まれないファイルの掃除。`SKILL.html` 更新日の突合。

## 改善提案の出し方

1. 結論を出す: 現状維持 / 軽量化 / 統合 / 一部削除 / 削除候補。
2. 残す核 → 削る候補 → 寄せ先（workflow / reference / asset / script / AGENTS・registry参照）を分けて示す。
3. description変更がある場合は、拾う発話 / 拾わない発話 / 近接Skillの誤爆リスクを言う。
4. workflowを分割・統合する場合は、理由と親workflowに残す完了条件を示す（`create-rules.md` §4.1）。
5. 中〜大規模改善は plans更新先・対象Skill名・種別、または更新不要の理由を示す（`create-rules.md` §10）。
6. 移行・改名・削除を伴う場合は、他Skillが対象Skill名や行番号（`SKILL.md L123` 等）を参照していないか確認し、行番号参照はworkflow名+Step名へ置き換える案を出す。

## 禁止事項

1. レビュー依頼だけでファイル変更しない（修正の実行は `review-skill.md` の承認Step以降）。
2. 削除・移動・改名・symlink変更・logs/catalog更新を暗黙に実行しない。
3. repo-local Skillを配置理由なしにGlobal化しない。
4. 正本ルールをSkill側へ長くコピーしない。
