# morning-routine（朝会と夜会）

1日の始まりと締めの定型。判断の道具は既存の正本に置き、本skillは「読む順と手順」だけを持つ（scriptなし）。

正本ポインタ（本文に複製しない）:

- 管轄共有・編成・サブスク構成: `~/Private/personal-os/説明書/指揮官ロースター.md`
- 段階語彙・規模・人間ゲート: `~/Private/personal-os/説明書/運用契約.md` §2
- 体制と裁定: `~/Private/personal-os/my-brain/areas/ai運用/plans/active/2026-07-03-マルチ指揮官体制/program.md`（本skillの計画=plans/01）
- 入口判断（規模/経路/起動形/モデル）: skill `plan-triage` ／ 実行中の監督: skill `cockpit-supervisor`

## 朝会モード（発火: おはよう・朝会・朝のルーチン・モーニングルーティン）

- ⓪ usage残を確認する（Claude Code=/usage・Codex CLI=/status。人間に実行してもらい結果を聞いてよい）。ロースターのサブスク構成節と照らし、今日の指揮官モデルを決める。
- ① 前日デイリーの「今日のダイジェスト」「今日終わったこと」「明日へ」と、当日デイリー（朝転記区画があれば）を読む。
- ② 計画ボード（active計画＋次:）とrepo-registry/repo概要.md＋projects/active実体を突き合わせ、repoごとに「繰越・今日やること・動かさない宣言」を人間と壁打ちする。深掘りは「昨日動いたrepo」と「着手可能な作業のあるrepo」だけ。動きのないrepoの大計画照合は週1（月曜朝会）。
- ③ 今日の指揮官数を決める（既定3=メイン2＋サブ1。サブ含め最大4=人間の管理上限。作業量が2レーン分以下なら1〜2本へ縮退。レビュー同時2本制約に当たる日は増やさない）。
- ④ 役割分担をロースターの「今日の編成」へ記入する（見出し日付を当日に書き換え・現在形のみ・履歴はgit）。
- ⑤ 指揮官の起動プロンプトを `references/commander-prompt.md` の定型から生成する（チャット出力のみ・ファイル保存しない）。
- ⑥ 今日のTODOを更新する（前日繰越と朝会結論を追記・行末に（朝会）タグ。既存行の削除・書換は人間のみ。auto:*区画には触らない=契約§3）。インボックスに未処理行が残っていれば振り分け経路（program子03）へ回す。

## 夜会モード（発火: 夜会・今日の締め・おつかれ。18:30と23:30、ダイジェスト3本に同期）

- 最新の「今日のダイジェスト」スナップとレーン実況を読み、全体状況を人間へ短く報告する。
- 今日のTODOの消し込み・繰越を人間と確認する（更新はAI追記・[x]化に注記を添える・削除は人間承認）。
- ロースターの各指揮官「今」を実態に合わせて照合・更新する。
- 23:30のみ: worktree状況まとめを出す: `skills/orca-cockpit/scripts/worktree-sweep.sh` を1回実行し、クローズ候補/注意のリストを夜会レポートに載せる（検知のみ。候補のdown/削除の実行は人間承認後=常に人間ゲート・マルチ指揮官体制program子07）。
- 23:30のみ: 「明日へ」節の記入を促す（朝転記計画と連動）。あわせて逆算Alignmentチェックを行う: ①今日やったことが年間/3年目標に繋がっていたか（逸れは明日へ補正） ②対応の無い放置目標はないか ③「待ち」区画に人間確認待ちが滞留していないか ④次の一手が何日も動かない古いactive計画はないか。検出したら依頼インボックスへ1行残す。

## 指揮官プロンプト生成

- 定型は `references/commander-prompt.md`（固定部=読む順・進め方・人間ゲート／可変部=役割名・担当・管轄パス・並行・特記）。可変部だけ朝会で埋める。
- 配布経路: ハイブリッド期=人間がデスクトップ新チャットへ貼る。司令部期=全体管理者が `orca terminal create --worktree "path:~/Private" --title <役割名> --command "claude --model claude-opus-4-8 --permission-mode acceptEdits"` でペインを立て、`cockpit.sh send` で注入する（program子04）。

## 制約

- 朝会の結論・サマリは全てデイリーへ集約する（専用ファイルを作らない・契約§5）。
- 朝会・夜会の中で発生した新しい玉は、skill `kickoff`（起票ゲート）で1行起票してから動かす（出所の保証）。
- 状態共有はhookではなくskill手順＋ファイル: 各指揮官は節目（レーン起動/完了/退勤）にロースターの自分の「今」を1行更新する。
- Fable系の高コストモデルは判断業務専用（ロースターのサブスク構成節が正本）。実行系レーン・中間指揮官ペインへは使わない。
- usage確認の追加タイミング: フル規模レーンの一斉起動前・レビュー一斉配布前・夜会2回。
