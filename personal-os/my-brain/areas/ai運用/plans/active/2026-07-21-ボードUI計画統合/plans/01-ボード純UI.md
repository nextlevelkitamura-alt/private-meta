# ボード純UI（テーマ軸一本化＋ステップ縦タイムライン）

## 目的

focusmapボードをモックv2の形へ刷新する。DB変更ゼロで「ユーザーが体感できる最小の縦切り」を最初に出す。

## 非対象

- migration（列追加は子02/03）・board.py/plan-ops/hooks の変更
- 計画チップ・計画詳細画面（子02）・サブ個体の展開詳細（子03）
- 「経過◯分」のステップ単位表示（started_at列は前提ゲートで供給済みだが打刻ロジックが子02所掌のため。セッション単位の稼働分数のみ＝sessions由来）

## 現状

- 現行ボードは「目標」「動いているエージェント」「終わったこと」の独立3区画。セッションは未紐付けが多い。
- モックv2（2026-07-21人間承認方向）: テーマカードが唯一の親。中に「やること→セッション→終わったこと(折りたたみ)」。
  人間の仕事は最上部「きみの番」レーンのみ。テーマ外は「未分類」枠。実行中の緑は2.4秒周期のゆっくり明滅（人間調整済み）。

## 実行契約

- 対象repo: ~/Private/projects/active/focusmap
- 実行形: delegated-parallel（30分スプリント・下の「実行体制」節。Orcaは使わない=2026-07-17人間指示。サブエージェント3体+指揮官直営）
- 最初に読む順番: focusmap AGENTS.md → ../program.md → ../実装/共通.md → この計画 → モックv2（../references/）
- 変更可能範囲: focusmap/src/（today系コンポーネント・boardページ・lib/turso読みクエリ）
- 変更禁止範囲: db/turso/migrations/・server actionの書き込み系・session-board・plan-ops
- 維持する契約: 既存の2層チェック（見出し完了=人間/flow-done）の意味・質問回答UIの機能・375pxモバイルファースト・白背景ライト単色
- 検証: ローカルdev + 375px実機相当で全状態（朝: テーマのみ／昼: 稼働あり／夜: 済み多数）のスクリーンショット
- 停止・エスカレーション条件: 既存server actionの変更が必要になった時・テーマ外データの表示先が決めきれない時

## 方針

- テーマカード: 帯=タイトル＋進捗%＋ライブ帯（稼働体数・SQL導出）。中=やること行→セッション行（ぶら下げ）→終わったこと（折りたたみ・件数とセッション数）。
- やること行の展開: todo_steps を seq昇順の縦タイムラインで表示。done=✔＋取り消し線グレー、doing=明滅ドット＋「今ここ」、kind=review/fix はバッジ。steps未登録は「ステップ未定義」バッジで可視化（沈黙させない）。
- ~~「きみの番」レーン: 質問・提案・確認待ちを最上部に集約。0件なら非表示。~~ → **修正02で廃止**（2026-07-21実機確認の人間指示。質問待ちはセッション行の待機表示＋行内回答導線に置き換え）。
- 通常状態＝デフォルト折りたたみ（修正02）: テーマカードは1行サマリ（テーマ名・進捗%・稼働/待機・完了数）のみ。やること・ステップ・セッションは展開時だけ表示。完了条件・目的・計画本文はボードに出さず、計画チップ→plans詳細ページの2段導線とする。
- 「未分類」枠: theme_id無所属のtodo/セッションを最下部に。所属提案の採否は既存質問UIを流用。
- 進捗%: done/(total−skipped) をSQL導出。保存しない。
- PC幅対応: 同一ページのレスポンシブでPC幅（目安1024px以上）も最適化する（テーマカードの多カラム化等・別ページは作らない）。データは同一DBなのでPC/スマホの内容同期は構造上担保済み（2026-07-21人間要望）。
- PCサイドバー統一: PCのカレンダー/Todo画面の右サイドバー（現「選んだ予定/この日の予定」）に、スマホと同一のテーマ軸ボードを**同一部品・同一クエリ**で表示する（部品の二重実装禁止）。カレンダー予定リストとの同居方法（ボード上段+予定下段の縦積みを既定・タブ切替は代替案）は実装時に画面幅で判断し、決めきれなければ人間確認（2026-07-21人間要望・スクショ指示）。

## 実行体制（30分スプリント・2026-07-21人間要望）

30分は「実装スプリント」の壁時計目標。前提ゲート消化・合意ゲート・評価/修正・push/本番反映（人間ゲート）は30分に含めない。

- 0〜8分（指揮官直営・レーンD）: 部品契約の先行確定。today系部品のprops型・`lib/turso` 読みクエリ骨格・ファイル配置を1コミットで固定する。**ここが並列衝突防止の要**（A/B/Cはこの契約に従う）。
- 8〜25分（impl-opus×3・worktree分離で並列）:
  - レーンA: テーマカード＋ステップ縦タイムライン部品（新規ファイルのみ: theme-card / step-timeline）
  - レーンB: きみの番レーン＋未分類枠（新規ファイルのみ: ask-lane / stray。既存質問UIは流用参照）
  - レーンC: boardページ組替え＋PCサイドバー統合＋レスポンシブCSS（既存ページを触るのはCだけ）
- 25〜30分（指揮官）: 3レーンをマージ→ローカルdev実走→375px/1024pxスクリーンショット取得。
- 衝突防止契約: 同一ファイルを2レーンで触らない（A/Bは新規ファイルのみ・既存改変はC専任）。Dの部品契約変更が必要になったレーンは自走せず指揮官へ差し戻す。
- 事後: impl-evaluatorで完了条件を採点（評価md）→ 修正ラウンド → push・本番反映は人間承認後。
