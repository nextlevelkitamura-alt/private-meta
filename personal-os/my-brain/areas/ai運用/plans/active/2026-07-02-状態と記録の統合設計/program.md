# 状態と記録の統合設計（Personal OS運用 v1）

## 目的

「企画 → 計画 → 計画レビュー → 実装 → 実装レビュー →（修正⇄再レビュー）→ 人間確認 → 完了」の作業パイプラインを、人間の関与が全体の2割（入口の企画・要所確認・出口の確認）で回せるようにする。そのために今バラバラな3つを1つの設計に統合する。

1. リアルタイムの状態可視化（どの計画が今どの段階か。AIも人間もそこを見て次の作業を選ぶ）。
2. 漏れない記録（Claude Code / Codex / Orca / 人間の手作業が、hookの成否に依存せず必ずデイリーに載る）。
3. 横断できる計画構造（program乱立と恒久ルールの埋没を解消し、仕様変更の波及先が分かる形にする）。

## 方針

### 1. 3層モデル（正本の役割分担）

- 記録層（事実）: 各サーフェスのネイティブ記録を事実の正本とする（Claude transcript / `~/.codex/sessions` と `session_index.jsonl` / 各repoのgit履歴 / ai-jobsフォルダ）。hookは正本ではなく「更新の引き金」。
- 状態層（宣言）: 計画ライフサイクルは現行どおりバケット＋program子マップが正本。実行中の「段階」はgit文書に第2の正本を作らず、表示層にのみレンダリングする（契約§5の二重管理禁止を守る）。
- 表示層（ビュー）: 人間が見る・書くページは当日デイリー1枚に統合し、1本のレンダラが `auto:*` 区画を冪等に再生成する（契約§3「デイリー=レンダラ」を全ビューへ拡張）。`ダッシュボード.md` は独立ページとしては廃止する（§4）。

### 2. 段階語彙（作業パイプライン）

- フル: `企画 → 計画 → 計画レビュー → 実装 → 実装レビュー →（修正⇄再レビュー）→ 人間確認 → 完了`。
- ライト: `計画 → 実装 → 実装レビュー → 完了`（plan.mdは必須だが最小行数でよい。人間確認は事後報告のみ）。フル/ライトは計画冒頭に `規模:` で宣言する。
- cockpit 3ペインと段階の対応: 計画+監督ペイン＝計画/計画レビュー、実装ペイン＝実装/修正、レビューペイン＝実装レビュー。ペインへの配布＝段階遷移として記録できる。
- 人間ゲートは「入口（企画・計画レビュー）と出口（人間確認）」の2点＋任意の途中確認。ゲート待ちはボードで最上部に表示する。

### 3. 取り込み（漏れない記録）

- 原則: push（hook）は速報、pull（レンダラが記録層から吸い上げ）が保証。hookが落ちても次のレンダで必ず追いつく（冪等・全置換マーカー方式は現行契約を踏襲）。
- Claude Code: 現行Stop hookを継続し、同hookからレンダラを起動（debounce 60秒、async）。
- Codex: hookは増やさない。レンダラが `~/.codex/session_index.jsonl` から当日分を取り込む（transcript読解の要約は将来）。
- Orca: `cockpit.sh` の up/send/down が段階イベントを直接追記する（自前の決定的スクリプトなので確実）。ペイン内のclaudeは既存hookで記録済み、codexペインはCodex pullで拾う。
- 人間の手作業: レンダラが active repo 群の当日コミットを走査して拾う＋任意の1行メモコマンド。計画に紐付かない作業は「未紐付け」としてボードに表示する（漏れをゼロにするのではなく、漏れを見えるようにして直す）。
- hookにスキル（LLM判断）を入れて記録させる方式は不採用。記録の保証は決定的スクリプト＋pull再構築で持ち、スキルの仕事は要約・レビュー・closeoutなど判断部分に限定する。

### 4. ビュー（デイリーとダッシュボードの統合・リアルタイム化）

- 人間が見る・書くページを当日デイリー1枚に統合する。セクション構成（上から）:
  1. 長期軸の逆算（auto:goal。年間計画から転記）
  2. 今日のTODO（人間が書く。未処理分は翌日へ自動シード）
  3. 依頼インボックス（人間がいつでも1行書く。§8）
  4. 今やっていること（auto: セッション / ペイン / 段階 / 最終活動）
  5. 待ち（auto: 人間確認待ち・着手可能・未紐付け作業）
  6. 計画ボード（auto: 全領域のactive一覧＋次の一手。旧ダッシュボードの内容）
  7. 今日終わったこと（人間の実績行＋auto:done）
  8. ログ（auto:log 生ポインタ）
- 人間が手で書くのは 2・3・7の実績行のみ。それ以外はレンダラが再生成する（マーカー外は保護、の現行契約を踏襲）。
- 状態の正本は変えない: 計画=バケット＋programマップ、実行中=イベント。デイリーはビュー＋人間の書き込み欄であり、第2の状態保管庫にしない。過去日のファイルは「その日の終了時点のスナップショット」として凍結され履歴になる。
- `ダッシュボード.md` は役目を終えるため廃止し、1行のポインタ（ボードは当日デイリーへ統合）だけ残す。参照している規約（`areas/AGENTS.md` §2・卒業手順・`ゴール/AGENTS.md` ほか）は移行手順1で更新する。優先マーク（◎/○）は plan.md / program.md 冒頭の `優先:` 行へ移す。
- レンダラが当日ファイルをテンプレから自動生成し（朝の手動コピー廃止）、`auto:*` はイベント毎に再生成、23:30は締めの最終レンダに変更。`auto:carry` / `auto:progress` は実装するまでテンプレから外す（実装の無い空マーカーを残さない）。
- Notion（将来・課題4）: 「1日1ページ」構造はNotionのデイリーページ運用とそのまま対応する。マーカー区画と見出し名を安定契約として凍結し、レンダラに「MD→Notion API」出力先を1つ足すだけで移行できる形にしておく。正本は常にMD。今は着手しない。

### 5. 計画構造の統合（program乱立の解消）

- レター呼称（A/B/C）を廃止し、ai運用の4本を1本の「Personal OS運用」programに統合する。本計画が承認されたら本計画をprogram化し、A/C/B/オーケストレーションの未完了子だけを子計画マップに移記して各programはdoneへ送る（実行済みの価値はすでに規約・基盤に反映済み）。
- 恒久ルールを計画フォルダから出す: `共有コントラクト.md` を `説明書/運用契約.md` へ昇格（説明書=運用モデルの正本、契約はその一部）。計画は契約を参照する側に回り、`説明書/README.md` のポインタも更新する。段階語彙（上記2）と モードA活性化条件（下記7）は契約に追記する。
- 知識の再利用: `areas/ai運用/決定ログ.md` を新設（追記のみ。日付・決定・理由・関連パスの4行程度/件。program内の決定ログを移記して以後一本化）。計画立案時の読み順を固定する: `identity.md` → `説明書/`（運用モデル＋契約）→ `決定ログ.md` → 関連するarchive計画。
- 語彙・規約の変更手順を契約に1行足す: 正本を変えたら同一作業単位で `AGENTS.md` / README / テンプレ を grep して追従（矛盾1の再発防止。将来は整合チェックをloop化）。

### 8. 依頼インボックス（人間の思いつき → 計画の自動起案）

- 入口は1つ: 当日デイリーの「依頼インボックス」に人間が1行書く（例: 「◯◯を調査してほしい」）。書く場所を覚えるのはここだけ。
- 巡回loop（launchd・30〜60分間隔）が未処理の行を検知し、トリアージSkillをheadlessで起動する。Skillの仕事: (a) 領域判定（ai運用/work/money/health）、(b) 規模判定（ライト/フル）、(c) 該当areaの `plans/active/<日付-企画名>/plan.md` を起案、(d) インボックス行に「→計画作成済み（パス）」を刻む。
- loopがやるのは起案まで。実行はしない（実行GOは人間の計画レビュー＝入口ゲートのまま）。「配るのは指揮官」のモードB原則と矛盾しない。
- 完了時のボード反映は§1の仕組みで自動: バケット/マップが変わればレンダラが計画ボードを再生成する。手動のボード更新という作業自体が消える。
- 導入順序はレンダラv1の安定後（移行手順6）。それまでインボックス行は指揮官チャットが手動で拾う。
