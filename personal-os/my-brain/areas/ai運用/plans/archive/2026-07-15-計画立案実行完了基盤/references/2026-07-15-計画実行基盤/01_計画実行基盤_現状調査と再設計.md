# 計画実行基盤の現状調査と再設計案

作成日: 2026-07-14  
対象リポジトリ: `nextlevelkitamura-alt/private-meta`  
対象領域: `personal-os/AIエージェント基盤/`、`personal-os/my-brain/areas/`

---

## 0. 結論

現在の `plans/planning|active|paused|done|archive` と、Program配下の `program.md + plans/NN-子計画.md` は、基礎構造として残してよい。

今回の主な変更は、フォルダを増やすことではなく、次の責務を明確にすることにある。

1. `plan.md` / 子計画を、そのまま実装担当へ渡せる実行契約にする。
2. `program.md` を、依存関係・実行Wave・統合責任を持つ親索引にする。
3. worktree・branch・runtime・session IDは計画本文に固定せず、実行時にハーネスが割り当てる。
4. 実装結果と評価結果から、子計画のチェックボックスとProgramマップを決定的に同期する。
5. Hookは計画内容を推測して直接編集せず、「同期が済んでいなければ終了させない」ガードにする。
6. `archive` を「成功済み」だけでなく「閉じた計画」とし、終了区分と理由を必須にする。
7. Orcaを既定経路から外し、Claude→Codex、Codex→Claudeを共通Task Packetで動かせるようにする。
8. カスタムエージェントは役割・権限・出力契約だけを持ち、worktreeや濃い性格を固定しない。
9. `identity.md` の短い内容は `AGENTS.md` に統合し、area直下の `知識/` は必須構成から外す。
10. 計画固有の知識・調査資料は、その計画の `references/` に置く。

## 1. 現在、計画はどこから実行されているか

### 1.1 フォルダを置いただけでは自動実行されない

現在、`plans/active/` を監視して子計画を自動spawnする常駐実行エンジンは存在しない。

`plan.md` / `program.md` は、次の仕組みから参照される「正本」である。

```text
ユーザー依頼
  ↓
session-board UserPromptSubmit
  └─ 計画の要否・置き場・Program判定を短く注入
  ↓
kickoff
  └─ デイリーの依頼インボックスに起票
  ↓
plan-triage
  └─ 規模・対象repo・計画置き場・起動形・モデルを判断
  ↓
plan-ops
  ├─ plan/program/子計画の雛形生成
  ├─ planning→active昇格
  ├─ Programマップの既存ブロック更新
  └─ 静的lint
  ↓
実装委譲
  ├─ 現在の実働: Claudeメイン → /codex-impl → codex exec
  ├─ 任意: handoff-plan-supervisor が別チャット用指示を生成
  └─ 旧既定: Orca cockpit
  ↓
impl-reviewer
  └─ 計画の完了条件とdiffを照合して評価NN.mdを作る
  ↓
修正NN.md → codex exec resume
  ↓
計画・Programマップ・バケット更新
  └─ 現在は大部分が手動
  ↓
session-board session-end
  └─ program-lint、Daily完了記録、git仕上げ
```

### 1.2 現在の主要コンポーネント

| 役割 | 現在の正本 | 実際にしていること |
|---|---|---|
| 全体ルール | `AIエージェント基盤/GLOBAL_AGENTS.md` | 規模、レビュー、人間ゲート、計画ルーティング |
| 最初の注入 | `hooks-registry/hooks/session-board/common.py` | 計画の3判定と置き場をAIへ通知 |
| 起票 | `skills/kickoff/SKILL.md` | デイリーへ依頼を1行追加 |
| 入口判断 | `skills/plan-triage/SKILL.md` | サクッと/ライト/フル、repo、計画場所、起動形 |
| 計画の機械操作 | `skills/plan-ops/` | scaffold、Program map更新、lint、planning→active |
| 外部実装への委譲 | `agents-registry/claude/commands/codex-impl.md` | ClaudeメインがCodex CLIを直接駆動 |
| 実装評価 | `agents-registry/claude/agents/impl-reviewer.md` | read-onlyで完了条件を採点 |
| 汎用引継ぎ | `skills/handoff-plan-supervisor/SKILL.md` | 実装プロンプト、参照順、禁止範囲、戻り形式を作る |
| セッション終了 | `hooks-registry/hooks/session-board/session-end.md` | Daily完了記録とgit仕上げを促す |
| 実行可視化 | `session-board` | セッション状態とDailyログのみ。計画状態は所有しない |

### 1.3 重要な現状認識

- `session-board` はセッション状態とDailyログを所有する。
- `session-board` は `plan.md`、バケット、Program子計画マップを所有しない。
- `plan-ops` も「ジョブ完了後の親子集約」を現在は手動としている。
- したがって、「実装が終わったのに子計画・チェックボックス・Programマップが古い」という穴が残る。
- Claudeには節目を促すPrompt型Stop Hookがあるが、現行の共有Stop commandはボードを⏸へ変えるだけ。
- Codex側も現行repo実装ではStop時にボードを⏸へ変えるだけで、計画同期を検査しない。
- 現行の `plan-triage` はOrcaの2/3ペインを既定出力にしており、現在の利用実態と合っていない。
- 実際の非Orca経路は `/codex-impl` である。
- Codex→Claudeの逆方向委譲は、同じ粒度で正本化されていない。

## 2. 現在の問題

### 2.1 計画テンプレートが薄い

現在のplan/子計画テンプレートは、主に次だけを持つ。

- 目的
- 現状
- 方針
- 完了条件

実装担当へ渡すには、次が不足している。

- 非対象
- 対象repo
- 最初に読む順番
- 変更可能path
- 変更禁止path
- 依存成果
- 検証コマンド
- 停止・エスカレーション条件
- 完了時に返す結果
- 実装結果
- 終了理由

### 2.2 計画更新が手動

現行 `progctl` は、既存Program子ブロックの次だけを変更できる。

- 状態
- 次の一手
- 参照commit

評価結果から次を同期する機能はない。

- 子計画の完了条件 `[ ] → [x]`
- 子計画の実装結果
- Programマップの状態とチェックボックス
- Program全体の完了可否
- 終了区分
- 後継計画

### 2.3 archiveの意味が実運用と合っていない

現行規約はおおむね次である。

```text
done    = 完了・未評価、またはAIレビュー済み
archive = 評価済みOK
```

一方、実運用では次の理由でもarchiveへ入れる。

- 並行計画と矛盾した
- 別計画に統合した
- 後継計画に置き換えた
- 継続しないと判断した

したがって `archive=成功` と固定すると、実態を正しく記録できない。

### 2.4 Hookへ意味判断を持たせると危険

「Stop時にHookが自動でProgramを編集する」は一見便利だが、現在の設計原則と衝突する。

- Hookは軽量・決定的・非ブロッキングを基本としている。
- Stopは毎ターン発火しうる。
- Hookは、計画が本当に完了したかという意味判断に必要な文脈を持たない。
- worker側と親側では更新責任が異なる。
- 別repoのworktreeから、中央Programを直接編集すると競合しやすい。

したがって、次の分担がよい。

```text
ハーネス / planctl:
  実装結果・評価結果を使って計画を更新する

Stop Hook:
  明示されたrun manifestを検査し、
  「レビューPASSなのに計画同期が未実施」なら終了を継続させない
```

### 2.5 カスタムエージェントとworktreeが混同されている

カスタムエージェントは再利用可能な役割である。

worktreeは一回のTask実行に割り当てる作業場所である。

```text
implementerという役割
  + task_id=loop-turso-02
  + base_commit=f0361b0d
  + task専用worktree
  = 今回起動するworker
```

agent定義に固定worktreeを入れるべきではない。

### 2.6 area構成が少し重複している

現在は各areaに次がある。

```text
AGENTS.md
CLAUDE.md -> AGENTS.md
identity.md
知識/
plans/
```

`identity.md` は、目的・判断基準・置くもの・置かないものを短く記した内容であり、`AGENTS.md`と責務が重なる。

また、計画中に必要な知識をarea直下の `知識/` に置くと、どの計画のための資料かが切れやすい。

## 3. 採用する原則

### 3.1 4種類の情報を分ける

| 種類 | 置き場 | 内容 |
|---|---|---|
| Area規約 | `AGENTS.md` | 目的、判断基準、置くもの、ルーティング |
| 計画正本 | `plan.md` / `program.md` / 子計画 | 何を、なぜ、どの条件で実行するか |
| 実行時情報 | run manifest | runtime、worktree、branch、base SHA、run ID |
| 実行結果 | result packet + 評価MD | commit、変更path、検証、リスク、採点 |

### 3.2 計画本文へ入れない情報

次は実行ごとに変わるため、計画本文へ固定しない。

- 実際のworktree絶対パス
- 実際のbranch名
- session/thread ID
- 一時ログpath
- 今回選んだモデルID
- worker個体名

### 3.3 Hookは検査、意味判断はAIと計画

Hookがすること:

- 明示的なmanifestを読む
- 必須ファイルの存在を確認する
- phaseと同期済みマーカーの整合を確認する
- 未同期なら継続理由を返す

Hookがしないこと:

- 会話から完了を推測する
- 完了条件を勝手にPASSにする
- どの計画をarchiveするか選ぶ
- 計画本文を直接生成する
- 別計画との矛盾を自動判定する

## 4. 推奨area構成

### 4.1 標準

```text
areas/<area>/
├── AGENTS.md
├── CLAUDE.md -> AGENTS.md
└── plans/
    ├── planning/
    ├── active/
    ├── paused/
    ├── done/
    ├── archive/
    └── 移行済み/    # 必要なareaのみ
```

### 4.2 `identity.md`

`identity.md`の内容は `AGENTS.md` の次へ統合する。

```markdown
## 目的
## 判断基準
## 置くもの
## 置かないもの
## 計画ルーティング
```

移行後、重複する `identity.md` は人間承認後に削除する。

全areaを一括削除しない。まず `ai運用` をpilotにし、内容が完全に統合できたareaだけ追従する。

### 4.3 `知識/`

area直下の `知識/` は必須構成から外す。

既定:

```text
plans/<bucket>/<計画>/references/
```

へ置く。

Programの場合:

```text
<program>/
├── references/             # 全子が共有する資料
└── plans/
    └── NN-子計画.md
```

子だけに必要な資料は、`references/NN-資料名.md` のように識別する。

例外:

- 2つ以上の計画で長期に再利用する
- 計画が終了してもAreaの判断基準として残る
- AGENTS.mdへ入れるには詳細すぎる

この3条件を満たすものだけ、area共通referenceとして残してよい。

既存 `知識/` は一括移動せず、1ファイルずつ所有先を判定する。

## 6. PlanとProgramの分け方

### 6.1 サクッと

次の全条件を満たす場合だけ、計画書なし。

1. 変更1〜2ファイル
2. 容易に戻せる
3. 人間ゲートなし

### 6.2 単発plan

次を満たすなら、手順が多くても単発planでよい。

- 主成果が1つ
- 責任を持つ実装担当が1人
- merge/rollback単位が1つ
- 依存が基本的に直列
- レビュー結果を1つの評価で閉じられる

### 6.3 Program

次のいずれかがあればProgram候補。

- 独立して実行・commit・mergeできる成果が2つ以上
- 2つ以上のwrite laneを並列に動かす
- 複数repoに別々の所有者・handoffがある
- 共有契約を先に固定し、その後に複数Waveへ分かれる
- 人間ゲートやrollback単位が別々
- 一人のworkerへ渡すには変更可能範囲が広すぎる

### 6.4 柔軟性

条件に当てはまっても、分割による調整コストが実装の明確さを上回る場合は、AIが単発planを選べる。

その場合は計画冒頭に短い判定メモを残す。

```text
形態判定: 単発
理由: 手順は5段あるが、同じrepo・同じ所有範囲・同じrollback単位で直列に閉じるため
```

ファイル数や行数だけでProgram化しない。

## 9. archiveの新しい意味

### 9.1 バケットの意味

```text
planning = 未着手・方針検討中
active   = 実装・修正・AIレビュー中
paused   = 再開条件がある一時停止
done     = 実装済み・最終評価PASS・人間のクローズ判断待ち
archive  = 人間が閉じると確認した参照専用計画
```

`archive` は成功だけを意味しない。

### 9.2 終了区分

```text
completed   = 成功完了
superseded  = 後継計画へ置換
merged      = 別計画へ統合
conflict    = 並行計画との矛盾・重複により終了
cancelled   = 意図的に中止
```

### 9.3 終了記録

```markdown
# 終了記録

- 終了区分: <completed/superseded/merged/conflict/cancelled>
- 終了日時:
- 人間確認:
- 理由:
- 後継・統合先:
- 実装済み範囲:
- 未完了事項:
- レビュー・判断根拠:
- 関連commit/評価:
```

### 9.4 遷移

```text
planning → active
active   → paused
paused   → planning / active
active   → done                 # 最終評価全PASS
done     → active               # 問題発見
done     → archive              # disposition=completed

planning/active/paused → archive
  # disposition=superseded/merged/conflict/cancelled
  # 終了記録＋人間確認が必須
```

`conflict`や`merged`を、未完了なのに`completed`として偽装しない。

### 9.5 archive lint

archive配下では `終了記録` を必須にする。

- `completed`
  - 全完了条件が `[x]`
  - Programの全子が完了
  - 最終評価が全PASS
  - 人間確認あり

- その他
  - 理由あり
  - 人間確認あり
  - `superseded/merged/conflict` は後継・統合先あり
  - 未完了事項を明記
  - 全チェックを無理に `[x]` にしない

## 10. 実装結果から計画を同期する仕組み

新しい `planctl` façadeを追加する。

```text
skills/plan-ops/scripts/planctl.py
```

既存の `progctl` と `bucketctl` は互換維持し、`planctl`が上位操作をまとめる。

### 10.1 推奨サブコマンド

```text
planctl prepare
  Task Packetとrun manifestを生成

planctl progress
  子状態・次・参照を更新

planctl apply-evaluation
  評価MDを解析
  完了条件の文言と完全一致したPASSだけを[x]にする
  実装結果を追記
  Program子マップを完了へ同期
  lintを実行

planctl close
  終了区分・理由・人間確認を記録
  bucketctl経由でarchiveへ移動

planctl sync-check
  result/evaluation/子計画/Programマップ/バケットの整合を検査
```

### 10.2 `apply-evaluation`の安全条件

- 評価MDの対象計画が一致する
- 完了条件の文言が計画と完全一致する
- 全項目がPASS
- `対象外`や未採点がない
- result commitが存在する
- 禁止範囲違反がない
- Program子の場合、親backlinkと子番号が一致する

一つでも満たさなければ、自動で完了にしない。

## 11. 実行ハーネス

### 11.1 実行時manifest

計画本文とは別に、実行時だけJSONを作る。

```json
{
  "version": 1,
  "task_id": "loop-turso-02",
  "role": "implementer",
  "runtime": "codex",
  "repo_root": "/path/to/repo",
  "plan_path": "/path/to/plan.md",
  "program_path": "/path/to/program.md",
  "child_id": "02",
  "base_commit": "f0361b0d",
  "worktree_path": "/path/to/worktree",
  "branch": "agent/loop-turso-02",
  "result_path": "/path/to/result.json",
  "evaluation_path": null,
  "phase": "running"
}
```

これはgit管理しない。

### 11.2 phase

```text
running
implemented
review_passed
synced
closed
blocked
```

### 11.3 task-scoped worktree

worktreeはagentではなくTaskに所属する。

```text
agent/role = implementer
task       = loop-turso-02
worktree   = task専用
```

read-only explorer/reviewerは、必ずしもworktreeを作らない。

write workerが並列、dirty checkout、別repo handoffの場合はtask専用worktreeを作る。

### 11.4 明示base

Claudeのagent定義に `isolation: worktree` を固定しない。

理由:

- 正確なbase SHAを使いたい
- 親HEADとdefault branchが違う場合がある
- branch/worktree命名を親が一元管理したい
- runtimeをClaude/Codexで入れ替えたい

ハーネスがworktreeを作り、そのcwdでClaudeまたはCodexを起動する。

## 12. カスタムエージェント

定義する役割は最小3つでよい。

### 12.1 explorer

```text
目的: 実行経路・所有ファイル・依存関係を確認
権限: read-only
性格: まず地図を作る。編集しない
出力: 根拠path、依存、リスク、実装候補
```

### 12.2 implementer

```text
目的: 一つのTask Packetを実装
権限: workspace-write
性格: 最小で安全な変更。隣の仕事へ広げない
出力: result packet
```

### 12.3 reviewer

```text
目的: 完了条件とdiffを照合
権限: read-only
性格: 自信より証拠
出力: 評価MD
```

### 12.4 定義へ入れないもの

- 固定worktree
- 固定branch
- タスク固有ファイルpath
- 特定Programの背景
- 長い人物設定
- 毎回変わるモデルID
- 会話履歴

モデル選定は `AIモデル一覧.md` とハーネスが行う。

## 19. 受け入れ条件

1. フォルダを置いただけで自動実行されると誤解しない導線説明がある。
2. 新規ライト/フル計画は、新テンプレートで作成される。
3. Program判定は硬いファイル数規則だけでなく、所有・rollback・並列単位を使う。
4. workerは親会話を知らなくても、計画とrun manifestだけで作業できる。
5. 実装agent定義に固定worktree・branch・Program固有背景がない。
6. write laneごとに明示baseからtask-scoped worktreeを作れる。
7. result packetと評価MDから、完了条件とProgramマップを同期できる。
8. 未評価・対象外・文言不一致を自動PASSにしない。
9. `archive`に終了区分・理由・人間確認が必ず残る。
10. conflict/merged計画をcompletedとして偽装しない。
11. Stop Hookは計画を直接編集せず、review_passed未同期だけを継続させる。
12. SubagentStopはwrite workerのresult packet欠落を検出できる。
13. session-boardはセッション状態の所有者のままである。
14. OrcaなしでClaude→Codexを実行できる。
15. Codex→Claudeは実機CLI仕様を確認してから有効化される。
16. 既存 `/codex-impl` は互換性を維持する。
17. Codex custom agentの現行仕様と内部referenceの矛盾が解消される。
18. Hook登録・trust・symlink変更・計画移動は人間承認前に行われない。
19. 既存の未コミット変更を巻き込まない。
20. 変更対象テスト、plan lint、program lint、hook E2Eが緑になる。
