# impl-explain（実装説明の型）

実装・変更の説明で、「コーディングしないがGO / NGを判断する人」が判断できる状態を作る。ページ全体の並びは `html-structure.md` §2のサンドイッチ構造が正本で、ここには③説明部の必須要素と合意の回し方だけを置く。

## 1. 3点セット（説明部の必須要素）

1. **どこを触る**: フォルダ構成図（R1）で全体の地図を見せ、今回触る場所だけ着色する。触らない場所も見せる（影響範囲が一目で分かる）。
2. **こう変わる**: before / after対比図（R5）。同じ図法で並べ、差分だけgo（緑）で着色する。
3. **中身の意味**: コード・設定・コマンドを見せたら、直後に「つまり: 」で始まる日常語の1行訳を必ず添える。全文は `details` に折りたたみ、主役は訳文にする。1行で訳せないなら、見せる単位が大きすぎるサイン。

## 2. 理解ループ（合意ゲート）

1. 人間の合意まで、説明対象の実装・削除・移動・改名・symlink変更をしない。
2. 指摘ごとに同じHTMLを更新し、更新履歴にvN（指摘 → どう直したか）を積む。新しいURLを増やさない。
3. ④の質問への回答が全部そろい、人間が「これでいい」と明示した時だけ合意とする。
4. 同じ箇所に2回指摘が来たら、内容ではなく見せ方を変える（図 → ウォークスルー → 具体例の順）。
5. 合意後は、合意内容の計画正本への反映を提案し、実装・削除・移動は実装レーンへ渡す。

## 3. 根拠と選択肢の書き方

1. 「今こう動いている」はpathと実ファイル本文を根拠にし、確認できなかった箇所は「推測」と明記する。
2. 選択肢を出すときは推奨案を先頭に置き、理由を1行付ける。「どれでもいい」にしない。
3. 新規・変更ファイルの本文ドラフトは省略しない。大きい場合は差分と要点を主表示にし、全文はファイル単位で折りたたむ。
