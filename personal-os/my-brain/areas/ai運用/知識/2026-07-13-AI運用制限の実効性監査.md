# AI運用制限の実効性監査

調査日: 2026-07-13 JST

## 結論

現在の仕組みは、計画を置く場所・active への昇格・作業開始時の注意喚起を一貫して導いており、**area ごとの active 最大3件**は実運用で機能している。実際の active は ai運用=2、work=2、health=0、money=0で、各areaは上限内である。

ただし「全体で active を3件まで」にしたい意図なら、現状は満たしていない（合計4件）。また、計画ファイル数には数値上限がなく、`bucketctl` を使わない直接の `git mv` は技術的には防げない。したがって、現状の評価は「強い誘導＋昇格時の機械ガード」であり、「絶対に逸脱不能な強制」ではない。

## 確認した導線

1. 正本: `my-brain/areas/AGENTS.md` が、状態をバケットで持ち、`active` を「今週着手・進行中のみ、1 area あたり最大3件」と定義する。
2. 入口: `plan-triage` と `kickoff` が、ライト以上の計画を `planning/` に起案し、指揮官が昇格を決めるまで active に入れないよう案内する。
3. 機械ガード: `plan-ops/scripts/bucketctl.sh` が `planning → active` のとき active 数を数え、3件以上なら拒否する。既定は dry-run で、追い出し・削除は扱わない。
4. 毎プロンプトの注意: session-board の `UserPromptSubmit` hook が、最初の依頼で計画の置き場判定と3条件を注入し、以後も現在の計画参照・ズレ修正を短く注入する。
5. runtime露出: `GLOBAL_AGENTS.md`、Codex/Claudeのhook受け口、`~/.codex/hooks.json` は正本へのsymlinkで露出している。本文コピーはない。

## 実測

- active 件数: ai運用 2 / health 0 / money 0 / work 2（各areaの上限内、全体は4）
- `bucketctl` テスト: 20 PASS / 0 FAIL。dry-run、昇格、上限拒否、dirty tree拒否、入力経路制限、commitを確認。
- session-board テスト: 172 PASS / 0 FAIL。計画列、初回注入、復帰、subagent、互換、永続化境界、runtime shimを確認。
- 合計: 192 PASS / 0 FAIL。
- この監査セッションでも、Codexの UserPromptSubmit から実際に計画判断ガイドが注入された。

## 監査判断

### 効いている点

- activeへの通常入口が一本化され、上限超過時の昇格は停止する。
- planning / active / paused / done / archive の役割が分かれ、状態を plan.md 本文に重複しない。
- 規約だけでなく、毎プロンプトの注意とテスト済みスクリプトがある。
- runtime側はsymlink露出で、基盤側を正本に保っている。

### 残る穴

- 上限は global ではなく per-area。全体WIPを3に抑える保証はない。
- `bucketctl` は通過時には強制するが、直接の `git mv` までOSレベルで遮断しない。
- `planning`、`paused`、`references/`、programの子計画数には数値上限がない。現在の「追加ファイルは読みやすい時だけ」は判断規約であり、機械チェックできない。
- hookの主目的は注意喚起と記録。hook単独で計画の作成・移動を拒否する仕組みではない。

## 推奨する次の改善

1. 「3件」のスコープを裁定する。areaごと3件を維持するのか、全area合計3件にするのかを決める。後者は現状4件なので、先に何を止めるかを人間が選ぶ必要がある。
2. active監査を読み取り専用で定期化する。全areaのactive数、直接移動の疑い、壊れたsymlink、計画直下の想定外ファイルを1枚に出し、違反は自動修正せず人間へ示す。
3. 計画フォルダの上限を明文化する。推奨は「直下は plan.md または program.md と派生HTMLのみ」「programの独立した子は最大3件」「資料は references/ へ」「それ以上は別計画へ分割」。数値を決めない場所は上限と呼ばない。
4. 昇格ルールの文言を統一する。`planning → active` は `bucketctl` のみ、他バケット遷移は人間判断の `git mv` と明記し、手順書・skill・AGENTSで同じ語に揃える。
