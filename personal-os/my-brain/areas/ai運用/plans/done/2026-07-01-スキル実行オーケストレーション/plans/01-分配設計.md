### 2モードの定義

1. **モードB ＝ 既定・主線**: **テキスト状態・単一指揮官（orca-cockpit）**。
   - 状態は ai-jobs のフォルダ位置ではなく、**コックピットに座る単一指揮官が保持するテキスト**が正本。
   - 分配は「**指揮官（コックピット）が各ペインへ直接配る**」（`cockpit.sh send --terminal <handle> --prompt`）。
   - 実行機構は `orca-cockpit` skill（1タブ3ペイン: 左=計画+監督 / 右上=実装 / 右下=レビュー）。
   - `orca-cockpit` 正本: `~/Private/personal-os/AIエージェント基盤/skills/orca-cockpit/SKILL.md`。
2. **モードA ＝ 従・将来の無人並行用 seam**: **ai-jobs/ready への run-card 投下 ＋ `mv` claim**。
   - 状態はフォルダ位置が正本。初版で設計した run-card 機構はこのモードとして**残す（seam）**が、既定では走らせない。

### §1 の読み替え（状態の単一正本）

1. モードB: 状態＝**単一指揮官のテキスト**（コックピット）。writer が1人なので同時書き込み衝突が無く、フォルダ状態機械は不要。
2. モードA: 状態＝**ai-jobs の run-card フォルダ位置**（初版どおり）。無人・並行で複数 writer になる時に初めて必要。
3. どちらも「状態を2箇所に複製しない」は不変。B→A 卒業時にテキスト状態を run-card へ**移す**（両方に持たない）。

### 分配ルータの主線/従（初版「方針 §2」の上書き）

1. 主線 = **指揮官(コックピット)が直接配る**。resolver の出力（`{skill, prompt, engine/surface}`）を、
   run-card ではなく**ペインへの注入プロンプト**として渡す。engine選択＝どのペイン/agent
   （左=claude 計画+監督 / 右上=codex 実装 / 右下=codex レビュー）へ配るか。
2. 従 = **ai-jobs フォルダ mv**（モードA）。無人並行が要るジョブだけ run-card 化して ready へ出す。

### 増分1 の書き換え（初版「方針 §6」の増分1を上書き）

増分1（手動パイプ）は **モードB前提**で走らせる:

1. 安全・idempotent・read-only な skill を1つ選ぶ。
2. `orca-cockpit` でコックピットを立て、指揮官がその skill を**ペインへ直接注入**
   （`cockpit.sh send --prompt`、skill名指定でよい）→ engine実行 → 結果を指揮官が**テキストで受けて着地判断**（done相当）。
3. ＝resolver（skill照合・prompt合成・engine=ペイン選択）＋ **テキスト状態での着地**を実証。run-card / ai-jobs は**使わない**。

**モードA seam の保持**: 「run-card を ai-jobs/ready に投下 → `mv` claim」は**増分2以降（無人並行が要るとき）の seam**として残す。
増分2 の Orca automations は、まず**モードB（コックピット監督を時間起動）**を主にし、無人化が実証できた skill だけ
モードA（run-card）へ卒業させる。

## 目的

スキルを最小実行単位として、**(A) 手動（通常チャット/CLI）** と **(B) 自動（時間トリガ）** の両方を
**同じ分配ロジック** で回す汎用オーケストレーション層を設計する。
分配（distribution/routing）＝「どのスキルを／どの内容(prompt・入力)で／どのengine・surfaceで／いつ」実行するかの決定。
手動と自動は**可搬コア（resolver）を共有**し、入口（trigger）と着地（landing）だけをアダプタで差し替える。
状態の単一正本は ai-jobs の run-card（フォルダ位置）、スキル本文は SKILL.md、スケジュールは別レイヤに置き、二重管理を作らない。

   run-card 1枚＝1実行依頼（`担当(engine)`/`出所`/`対象repo`/`依頼(prompt)`/`許可`/`前提`/`完了条件`/`戻し方`）。repo-aware。

   launchd 系の実体は「1 plist(60秒) → dispatcher → 各loopの `shouldRun()` → runner(`ai`/`script`)」。
   リファレンス実装は `仕事/scripts/nextlevel-dispatcher/`（本番稼働）。
3. **Orca automations（実在確認済み）**: `orca automations create --trigger <preset|cron|rrule> --provider <codex|claude|gemini>
   --prompt <text> [--precheck <cmd>] [--repo|--workspace] --workspace-mode <existing|new-per-run>`。

   `orca automations` が第一候補、判断は監督terminalが持つ、配送/待受/gateは orchestration。最初に作るのは
   **監督＝振り分けスキル**（ai-jobs/ready の run-card を読み、engine・skill・worktree/branch を決め `dispatch --inject` で注入）。
5. **coding-task-orchestrator（重複禁止対象）**: コーディング特化。確認優先→規模判定→surface選択→branch/worktree/port→

6. **runtime露出**: Global Skill正本 `AIエージェント基盤/skills/<skill>/SKILL.md` を各runtime(`~/.claude/skills` 等)へ

## 方針

### 1. モデル図（skill → 分配 → engine → 着地）

```text
  intent の源
  ┌── (A) 手動: 人間 / チャット / CLI ────────────┐
  │                                              │
  └── (B) 自動: 時間トリガ ───────────────────────┤
        Orca automations(主) / launchd(従)        │
        + schedule(=loop entry) + precheck(=shouldRun / due判定)
                          │
                          ▼
        ┌──────────────────────────────────────┐
        │   分配ルータ（distribution resolver） │  ← 共有コア・可搬
        │   1. どの skill  … カタログ照合         │
        │   2. どの内容    … prompt / 入力 合成    │
        │   3. どの engine/surface … 選択ヒューリスティクス│
        │   4. (自動のみ) いつ … schedule 参照     │
        └──────────────────────────────────────┘
                          │  出力＝ run-card を ai-jobs/ready へ
                          ▼   ★ここが状態の単一正本（§1）。他所へ状態を複製しない
        ┌──────────────────────────────────────┐
        │  監督 / dispatcher: claim(mv) → dispatch │
        │   engine : claude / codex / gemini       │
        │   surface: chat / codex exec / orca / headless│
        │   skill注入 = 名前参照（runtime symlink → 正本 SKILL.md）│
        │   coding 系タスクは coding-task-orchestrator へ委譲│
        └──────────────────────────────────────┘
                          │  実行 → 完了通知(worker_done)
                          ▼
        着地: ai-jobs done  →  daily(= レンダラ・Stream2 領域)
              ※ daily に状態を複製しない（§3/§5）
```

要点: **(A)/(B) は入口が違うだけで、分配コア以降（resolver → run-card → 監督 → 着地）は完全共有**。
run-card が手動と自動の共通表現であり、同時に状態の単一正本（§1）でもある。

### 2. 分配ロジック（推奨案＋対抗案＋選定理由）

**推奨案 — run-card を共有表現にした「分配ルータ＝監督スキル」**

- intent（手動でも自動でも）→ resolver が `{skill, prompt, engine/surface, (when)}` を決定 → **ai-jobs/ready に run-card を出す**。
- 監督/dispatcher が `mv ready→running` で claim（アトミック＝奪い合い防止）→ `担当` engine で実行 → done へ着地。
- 手動と自動は「入口アダプタ」だけ差し替え、**resolver と run-card は共有**。
- engine/surface 選択は skill の要件から引く（`execution-surfaces.md` を共通参照）:
  ローカル `.env`/DB/UI が要る → Orca worktree / ローカル。repoだけで閉じる → codex cloud / `codex exec`。
  会話的確認が要る → chat。決定的な非AI処理 → runner=`script`。

**対抗案（最低1）**

- 対抗A: **engine内蔵のルーティングに委ねる**。汎用プロンプト1本を渡し、engine自身が skill を自己選択・自己実行する。
- 対抗B: **skillごとに専用 automation を直接生やす**（分配コアを持たず「1 skill＝1 automation」）。

**選定理由**

1. 対抗A/Bは**手動と自動でロジックが分岐**し、KICKOFFの「同じ分配ロジックで回す」を満たせない。
2. 対抗Bは状態が run-card 以外（automation の run history / chat）に散る → **共有コントラクト §1（状態の単一正本＝run-card）違反**。
3. 対抗Aは skill選択が engine実装依存で、**監査・差し戻し・human gate が効かない**（誰が何を選んだかが run-card に残らない）。
4. 推奨案は run-card を単一正本にでき、手動/自動で**同一 resolver**を通せる。参考プランの「監督＝振り分けスキル」とも一致する。

### 3. 手動(A) / 自動(B) パスと共有部

**共有部（差し替えない）**

- 分配 resolver（skill照合・prompt合成・engine選択）。
- run-card スキーマと ai-jobs のフォルダ状態機械。
- 監督/dispatcher の claim→dispatch→着地。
- 安全ゲート（前回run未完なら新runを増やさない、human gate、secret非記載）。

**手動(A) の差分**

- 入口: 人間/チャットが intent を渡す（自然言語）。
- 小タスクは run-card を省き**直実行**してよい（coding-task-orchestrator の Small と同じ軽量方針）。
- 監督＝そのチャットの AI。会話で方向修正できる（loop-types ① の「見えて・止めて・引き継げる」）。

**自動(B) の差分**

- 入口: 時間トリガ（Orca automations / launchd）＋ `precheck`(=shouldRun / due判定)。
- intent の源は **schedule レイヤ**（どの skill がいつ due か）。
- 監督＝automation が起動する監督terminal（無人だが Orca orchestration 経由で可視・介入可）。
- 着地とレポートは worker_done → 監督が受けて次へ（参考プランの orchestration 標準フロー）。

### 4. 自動実行の接続（4点）

**(a) schedule の保持先**

- SKILL.md frontmatter には**置かない**。理由: schedule は deploy 関心で、SKILL.md は複数文脈で共有する skill本文の正本（§4）。
  1 skill を手動でも複数スケジュールでも回せるよう、**本文とスケジュールを分離**する。
- 保持先の推奨: **scheduled skill-run ＝ loops-registry の loop entry として持つ**（新規 schedule registry を作らない＝§6 重複回避）。
  scheduled skill-run は実質「runner=ai・command＝『分配ルータで skill X を回す』」の loop。
  schedule(trigger/shouldRun) は loop spec が単一正本、Orca automation はその**実体(instance)**（`loop.md`-spec ＋ dispatcher登録 と同型）。

**(b) launchd vs Orca automations（主/従・根拠つき）**

- **主: Orca automations（runner=ai の定期 skill 実行）**。根拠:
  1. trigger + precheck(=shouldRun) + provider(=engine) + workspace-mode を**nativeに提供**し、自作 dispatcher の8割を再実装せずに済む。
  2. Orca orchestration の可視監督に直結し、**loop-types ① の「見えて・止めて・引き継げる」を保持**したまま時間トリガを足せる。
  3. 参考プランが既に第一候補として最終判断済み。
  4. 位置づけ＝「**①キューをスケジュール起動する**もの」。②headless への卒業前でも ① のまま時間トリガを付けられる（契約 §2 と整合）。
- **従: launchd + dispatcher** を次に使う:
  1. runner=`script` の非AI定期処理（例: GWS の求人更新）。agent/worktree不要なので Orca automations は過剰。
  2. Orca runtime 停止時の fallback（常時ローカルで確実に起きる）。
  3. 安定・無人でよいと実証できた skill-run の **②headless への卒業先**。
- **二重管理でない根拠**: 両者は**runner種別(ai / script)と安定度**で住み分ける。同じ処理を2箇所に登録しない。

**(c) (prompt + skill) の合成＝skill注入の方法**

- **推奨: 名前参照**。run-card / automation prompt は **skill名（＝索引・ポインタ）だけ**運び、engine は runtime露出 symlink 経由で**正本 SKILL.md をロード**する。
  §4/§5 と整合（本文を複製せず、索引/ビューだけ運ぶ）。
- **fallback: パスpointer**。engine が名前ロードを headless文脈で拾わない場合、SKILL.md への**絶対パスを渡し「最初に読め」**と指示する（まだ本文コピーではない）。
- **最終手段: inline展開**。skill機構もFSアクセスも無い engine のみ。§4/§5 に反するので**原則回避**し、使う場合は理由を run-card に記録する。

**(d) 結果の着地**

- 着地は **ai-jobs done → daily(= レンダラ)**。daily に状態を複製しない（§3/§5）。
- 自動実行の状態は run-card のフォルダ位置だけが持つ。automation の run history は補助（取りこぼし確認）で、正本にしない。
- **状態の単一正本を守る接続の推奨(B1)**: automation の prompt は「**ai-jobs/ready の due card を拾って dispatch する監督**」にする。
  ＝スケジュールは heartbeat、状態は ai-jobs。特定の安定 skill だけ automation が skill を直接起動(B2)することを許可する。

### 5. 既存資産との関係と重複回避

- **coding-task-orchestrator … 新規（拡張しない）＋共通reference化**。
  - 拡張しない: coding特化の trigger/責務を汎用分配で汚さない（skill-creator §7「無理に汎用化しない」）。
  - 新規: skillカタログ全体を対象にした**汎用「分配ルータ skill」**を新設（＝参考プランの「監督＝振り分けスキル」）。
  - 共通reference化: surface選択・worktree統治・human gate・supervisor mode の汎用機構（`execution-surfaces.md` /
    `orca-patterns.md`）を中立な共通置き場へ lift し、両skillが**同じ参照を読む**（重複本文を作らない）。
  - 委譲: 分配ルータは、resolvedタスクが**コーディング**なら coding-task-orchestrator に委譲する（再実装しない）。
- **ai-jobs / loop-runbook … その上に乗せる（重複させない）**。
  - run-card＝状態の単一正本、loop entry＝schedule の単一正本、dispatcher/automation＝起動、と役割を分ける。
  - 分配ルータは新しい状態ストアや schedule ストアを**作らない**。既存レーンに run-card を出すだけ。

### 6. 段階ビルド計画（最小第一増分＝1skillを手動・自動の両方で流す薄いパイプ）

- **増分0（このフェーズ）**: 設計 doc のみ（本書）。実装・launchd・push なし。
- **増分1（手動パイプ）**: 安全で idempotent な read-only skill を1つ選ぶ →
  分配ルータが手動 intent から **run-card を ai-jobs/ready に生成** → 監督が `mv` で claim → engine実行 → **done 着地**。
  ＝resolver + run-card + 着地 を手動で実証。
- **増分2（時間アダプタ・未有効）**: 同 skill に対し
  `orca automations create --disabled --trigger daily --provider <engine> --precheck <due-check> --prompt <ready-queue監督prompt>` を1本用意。
  fire で **同じ resolver → 同じ run-card → 同じ着地**を通す設計を確認（有効化はしない）。
- **増分3（有効化・human gate）**: precheck 付きで automation を enable（人間ゲート）。
  「前回 run が未完なら新 run を増やさない」規律（参考プラン安全条件）を組み込む。
- **後続**: skillカタログ照合の高度化、engine選択ヒューリスティクスの精緻化、実証済み skill-run の ②headless(launchd) 卒業。

## 未決事項・人間に確認すべき点

共有コントラクトと衝突しうる点は、勝手に逸脱せずここに置く。★は conductor / 人間の確認が要る。

1. ★ **Orca runtime 依存**: Orca automations は fire 時に Orca runtime 稼働が前提。PCスリープ/Orca未起動でも回すべき自動処理があるか。
   あるなら、その処理は launchd(headless) を主にする（主/従の切り分けを user 確認）。
2. ★ **engine の名前ロード**: claude / codex / gemini が headless・automation 文脈で skill を「名前」で確実にロードするか未確認
   （特に `codex exec`）。実装前に小スパイクで確認し、非対応なら (c) の pointer-fallback を既定にする。
3. ★ **schedule の置き場**: 「scheduled skill-run ＝ loop entry を再利用（新規 schedule registry を作らない）」でよいか。
4. ★ **宣言と実体の分離**: schedule を「宣言(git＝loop spec) + 実体(Orca automation)」に分けると §5 に触れないか。
   `loop.md`-spec + dispatcher登録 と同型と主張。drift 防止として loop entry に automation id を記録し、**宣言を単一source**扱いにする。
5. ★ **上位構図**: 汎用「分配ルータ skill」が coding-task-orchestrator と 参考プラン(Orca運用skill) の**上位に座り委譲する**構図でよいか。
6. **最小パイプの skill 選定**: 増分1で使う「安全・idempotent・read-only な skill」を user と選ぶ。
7. **着地×Stream2 の seam**: 着地の daily 連携は Stream2「daily=レンダラ」分析に依存。seam だけ切り、Stream2 の結論を先取りしない。
