## 方針

1. **plansync新設（plan-ops配下）**: active配下をスキャンし、1文書1行で抽出（path・program_slug・kind program|child|role|eval|single・nn・title・bucket・body生md・git_commit・synced_at）。チェック集計（子N/M・完了条件x/y・parse_ok）は `_planops_map.py` で事前計算し、focusmap側ではパースしない。送信は store.py を import 再利用。content_hash一致はスキップ（冪等）。secret疑いパターン（token・鍵・password等の正規表現）にヒットした文書は同期拒否し人間へ通知
2. **migrationと初回投入**: inbox DBに `plan_docs`・`plan_progress` を新設（DDLは設計正本の草案に従う）。適用は人間が turso db shell で実施。初回は `--all` のdry-run提示→人間GO→一括投入
3. **post-commit自動同期**: ~/Private の git post-commit hook（登録=人間ゲート）が計画パスの差分だけ plansync を起動。多重起動はロックで防ぎ、送信失敗は spool 再送に乗せる（spoolの許可リストをplan文まで広げる）
4. **保険と鮮度**: SessionStart時に未同期HEAD差分があれば掃除（既存reconcileへの相乗り・注入文の範囲）。plan_docs.git_commit とローカルHEADの突合で「古い可能性」を判定できる形にする
5. **追従**: activeから出た計画（git mv・削除）はミラーからDELETE（表示キャッシュなので非破壊扱い・履歴はgitが持つ）。孤児行はフルスキャンreconcileで残さない
