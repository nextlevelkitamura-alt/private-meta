# 上限超過の警告とAIガード

## 目的

計画バケットの上限を、AIへのお願いで終わらせず、機械的な移動窓口・PreToolガード・Prompt Submit・AGENTS/Skillの4層で守る。満杯時は事情と次の人間判断を見せ、AIが勝手に計画を退避しない。

## 現状

`bucketctl` は planning→active の上限3だけを実装している。UserPromptSubmit は計画3判定を注入するが、paused/doneの容量や超過時の動作を知らない。Codex/Claude はいずれも PreToolUse でBashを拒否できるが、現在のruntime登録はsession-boardの開始・Prompt Submit・終了・サブエージェントだけであり、計画バケットへの生の `git mv` / `mv` を止めるガードはない。

## 方針

1. 01の `bucketctl check --json` を容量の唯一の機械判定にする。各 `plans/` root について planning/archiveは無制限、active=3、paused=3、done=8を返し、超過なら件数・上限・フォルダ一覧・「人間が選ぶまで自動移動しない」を含む警告にする。
2. Codex/Claude 共通の shared hook を `hooks-registry/hooks/` に置き、runtime別の薄いPreToolUseシムを作る。Bash内の計画バケットを対象にする生の `git mv` / `mv` を deny し、`bucketctl` の許可遷移へ誘導する。`bucketctl` 自身は01の検証に委ねる。AIの通常作業を止めないため、対象パスを含まないコマンドは無処理で通す。
3. Prompt Submit は強制ではなく常時の認知補助に留める。初回・計画ミラーに上限と `bucketctl check` を案内し、PreTool拒否または `check` の超過警告を受けた時は、対象一覧と必要な人間判断を報告する。自動で paused/done/archiveを選ばない。
4. 根拠と導線を同期する。`~/Private/AGENTS.md`、`areas/AGENTS.md`、ai運用/AGENTS、plan-ops SKILL、session-boardの手順MD/README、hooks-registry AGENTSのうち実際に計画移動を案内する箇所をgrepで特定し、正本へのポインタに揃える。規則本文のコピーは増やさない。
5. PreToolはAI runtimeのガードであり、人間が直接ターミナルから任意のファイル操作をすることまでは止められない。この場合も `bucketctl check` と03の監査で超過を検出する。新しいPreTool hookのruntime登録とCodex再trustは、実装・テスト後に人間へ具体的な差分を示して承認を得てから行う。

- [ ] Codex/Claude両方で、計画バケットへ直接移動する `git mv` / `mv` は拒否され、通常のBashと `bucketctl` は拒否されないことをstdin fixtureで検証する。
- [ ] 初回・ミラー・拒否メッセージが、active=3／paused=3／done=8、`planning/archive=無制限`、自動退避禁止、人間判断の必要性を同じ意味で伝える。
- [ ] AGENTS/Skill/手順MDの更新対象をgrepで特定し、計画移動の唯一の窓口と上限の正本へポインタを揃えている。
- [ ] 新hookの登録・Codex再trustは、未承認のまま実行されず、承認後にruntime別のE2E確認を行える。
