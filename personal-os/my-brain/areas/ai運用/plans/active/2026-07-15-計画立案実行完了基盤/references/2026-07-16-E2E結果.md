## 発見した機構欠陥

E2Eを「本物の`new-plan.sh`/`program-lint.sh`が生成・受理する正本テンプレ形式」で組み立てたところ、既存の単体テストが使う簡略化されたfixture形式では見えない、2件の機構欠陥が見つかった。いずれも**修正はせず、FAILのまま記録する**（修正判断は指揮官）。

### 欠陥A: `program_run.py::_section()` が正本テンプレの見出しを検出できない

- **対象ファイル**: `personal-os/AIエージェント基盤/agents-registry/harness/program_run.py`
- **該当箇所**: モジュール内に独自定義された `_section(lines, heading)` 関数（`_child_blocks()`経由で`子計画マップ`を、`_contract()`経由で`実行契約`を探すのに使われる）。
  ```python
  def _section(lines, heading):
      start = next((i + 1 for i, line in enumerate(lines) if line.rstrip() == f"## {heading}"), None)
  ```
  `line.rstrip() == f"## {heading}"` という**完全一致**でしか見出しを判定しない。
- **一方、`plan_lint_core.py`・`progctl_core.py`が共有する `_planops_map.find_section()` は前方一致**（`text.startswith(heading_prefix)`）であり、同モジュールの docstring 自身が「想定フォーマット」として
  ```
  ## 子計画マップ   ※ 子の状態変更と同じコミットでここを更新
  ```
  という、見出しの直後に注記が続く形を明記している。
- **実害の確認方法（最小repro）**: `new-plan.sh --program` で実際に生成した`program.md`（見出しは上記の注記付き形式そのもの）を`program_run.read_program()`に渡すと、`ProgramRunBlocked: 子計画マップに実行対象がありません`が発生する。この見出し文字列は、**本program自身（`2026-07-15-計画立案実行完了基盤/program.md`）を含め、この repository に存在する全ての`program.md`と完全に同一**である。

  # 実行例（agents-registry/harness で python3 -c 相当）
  import program_run
  program_run.read_program(Path("<new-plan.sh --program の出力>"))
  # → ProgramRunBlocked: 子計画マップに実行対象がありません

- **影響範囲**: `program-run`（ゴールコマンド）は、現状のテンプレ・既存program.mdに対して**子を1件も検出できず、実質的に起動できない**。既存の単体テスト（`test_program_run.py`）がこの欠陥を検出しないのは、そのテストの`write_program()`ヘルパーが独自に**注記なしの`## 子計画マップ`（bare見出し）**を書いており、実際のテンプレ形式と異なるため。

### 欠陥B: `planctl.py` の変更可能範囲パーサがbacktick付き・複数path・注記混在の実書式を解釈できない

- **対象ファイル**: `personal-os/AIエージェント基盤/skills/plan-ops/scripts/planctl.py`
- **該当箇所**: `range_items()` と `matches_range()`。
  ```python
  def range_items(value):
      return [item.strip().rstrip("/") for item in re.split(r"[,、]", value) if item.strip() and item.strip() not in {"なし", "該当なし"}]
  def matches_range(path, item):
      item = item.rstrip("/")
      return path == item or path.startswith(item + "/")
  ```
  backtickを剥がさず、区切り文字も半角/全角カンマ（`,`／`、`）だけを見る。
- **実際の`変更可能範囲:`書式**は、本program自身の既存子計画（例: `plans/02-遷移統制とplanctl同期.md`）で
  ```
  変更可能範囲: `skills/plan-ops/scripts/`（bucketctl拡張・planctl.py新設・共通core）、`skills/plan-ops/__tests__/`、`SKILL.md`・`references/script-map.md` の該当節、...
  ```
  のように、**backtick付きpath・`・`区切り・日本語注記混在**が標準形。
- **実害の確認方法（最小repro）**: 上記の実文をそのまま`planctl.range_items()`に渡すと
  ```python
  range_items("`skills/plan-ops/scripts/`（bucketctl拡張・planctl.py新設・共通core）、`skills/plan-ops/__tests__/`、...")
  # → ['`skills/plan-ops/scripts/`（bucketctl拡張・planctl.py新設・共通core）', '`skills/plan-ops/__tests__/`', ...]
  ```
  のように、backtickと日本語注記が付いたままの文字列がitemになり、`matches_range("skills/plan-ops/scripts/bucketctl.sh", item)`は**常にFalse**になる。E2E(1)の初回試行でも、子計画の`変更可能範囲: \`src/e2e1.txt\``（backtick付き・単一path）だけで`planctl apply-evaluation`が「変更可能範囲外の差分」として実装済みの正当な差分を拒否した。
- **一方 `program_run.py::_contract()` は自前でbacktickを正しく処理**しており（`re.findall(r"`([^`]+)`", scope)`で抽出してから、無ければbareパスへフォールバック）、同じ`変更可能範囲:`フィールドに対して**モジュールごとに解釈結果が食い違う**状態になっている。
- **既存テストが検出しない理由**: `test_program_run.py`の`RealGitProgramRunTest.write_program()`（実際に`planctl.py`を呼ぶテストクラス）は`変更可能範囲: src/{nn}.py`と**backticksなしで書かれており**、この欠陥を踏まない。同ファイル内のもう一方のテストクラス（`ProgramRunTest`）はbacktick付きで書いているが、Operationsを丸ごとfakeにしているため実際の`planctl.py`を経由せず、欠陥が表面化しない。
- **本E2Eでの回避**: E2E(1)以降、全ての合成fixtureで`変更可能範囲:`はbareなpath（backtickなし）で統一し、以降のE2E(2)(6)はこの回避のもとで実施した（本文冒頭「方法論」に明記）。
- **推奨対応（実施はしていない）**: `planctl.py`の`range_items()`にも`program_run._contract()`と同じbacktick抽出ロジックを持たせる（できれば両モジュールで共有関数化する）。

## 運用上の注意点（欠陥ではないが記録に残す）

1. **`終了記録.md`は`close --apply`が失敗しても書き出される**（E2E(3)で確認）。既存ファイルは上書きされないため、rejectされた試行の後に正しいdispositionで再試行しても、stale recordがそのまま使われて再度拒否される。訂正時は別ディレクトリで試すか、`終了記録.md`を確認・削除してからにする必要がある。
2. **`completed`でのarchiveは`done`バケット経由が必須**（`active`から直接`completed`disposition ではarchiveできない。E2E(4)で確認）。これは`bucketctl_core.move()`の意図的な設計であり、`my-brain/areas/AGENTS.md`§3の2段階ライフサイクルと一致する。欠陥ではないが、単発planの「completed→archive」を試す際は`active→done→archive`の順で操作する必要がある。
